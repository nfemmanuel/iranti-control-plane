                                        GET /api/control-plane/events/stream

```typescript

```

### 1.2 Supporting Modules

```
src/lib/staff-events-adapter/
  index.ts         ← Adapter start/stop lifecycle: startAdapter(), stopAdapter()

  emitter.ts       ← Insert synthetic events into staff_events table
```

The adapter starts when the control plane server starts and runs on a `setInterval` loop. It does not run as a separate process or worker thread in Phase 1.

## 2. Database Migration

### 2.1 Migration SQL

The CP-T001 spec defines the `staff_events` table schema. The exact DDL for this migration, reconciling the spec's schema sketch with the acceptance criteria:

```sql
-- Migration: create_staff_events_table
-- Spec ref: CP-T001 §3 (Option 1 schema)

CREATE TABLE IF NOT EXISTS staff_events (
  -- Primary key: UUID generated by DB
  event_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Timestamp: when the event occurred
  -- NOT NULL — every event must have a timestamp
  timestamp       TIMESTAMPTZ NOT NULL DEFAULT now(),

  -- Staff component: which subsystem emitted this event
  -- Constrained to the 4 known components for data integrity
  -- VARCHAR(32) is sufficient for all current component names
  staff_component VARCHAR(32) NOT NULL
    CHECK (staff_component IN ('Librarian', 'Attendant', 'Archivist', 'Resolutionist')),

  -- Action type: what happened within the component
  -- VARCHAR(64) covers all current action type strings with room for future additions
  action_type     VARCHAR(64) NOT NULL,

  -- The agent whose operation triggered this event
  agent_id        TEXT,           -- Nullable: may be unknown for adapter-inferred events

  -- The source surface (mcp, api, cli, etc.)
  source          TEXT,

  -- Entity targeting fields (all nullable — not all events target an entity)
  entity_type     TEXT,
  entity_id       TEXT,
  key             TEXT,

  -- Human-readable note from the triggering operation
  reason          TEXT,

  -- Event level: audit (always on) or debug (configurable)
  level           VARCHAR(16) NOT NULL DEFAULT 'audit'
    CHECK (level IN ('audit', 'debug')),

  -- Component-specific supplementary data
  metadata        JSONB
);

-- Index 1: Primary query pattern — time-ordered event stream
-- Supports: SELECT * FROM staff_events ORDER BY timestamp DESC
-- Supports: SELECT * FROM staff_events WHERE timestamp > :cursor ORDER BY timestamp ASC
CREATE INDEX idx_staff_events_timestamp
  ON staff_events (timestamp DESC);

-- Index 2: Filter by Staff component
-- Supports: WHERE staff_component = 'Librarian'
CREATE INDEX idx_staff_events_component
  ON staff_events (staff_component);

-- Index 3: Filter by agent
-- Supports: WHERE agent_id = 'product_manager'
CREATE INDEX idx_staff_events_agent
  ON staff_events (agent_id);

-- Index 4: Filter by entity (compound — covers both type+id and type-only queries)
-- Supports: WHERE entity_type = 'ticket' AND entity_id = 'cp_t001'
-- Supports: WHERE entity_type = 'ticket'
CREATE INDEX idx_staff_events_entity
  ON staff_events (entity_type, entity_id);

-- Index 5: Filter by level (most queries filter to level='audit')
CREATE INDEX idx_staff_events_level
  ON staff_events (level);
```

**DDL note — `event_id` vs `id`**: The CP-T001 spec uses `event_id` as the column name. The CP-T012 acceptance criteria states the column should be named `event_id UUID`. The API response field is `eventId`. The column name `event_id` maps to Prisma model field `eventId` by default convention. Use `event_id` in SQL and `eventId` in Prisma and API response.

**`IF NOT EXISTS`**: The `CREATE TABLE IF NOT EXISTS` guard makes the migration idempotent — safe to re-run if the migration runner fails mid-flight and retries.

### 2.2 Iranti Migration System

Iranti uses **Prisma v7** for migrations. The migration is applied using the standard Prisma migration workflow:

1. Add a Prisma model for `staff_events` to `prisma/schema.prisma`:

  agentId        String?                                                  @map("agent_id")
  source         String?
  entityType     String?                                                  @map("entity_type")
  entityId       String?                                                  @map("entity_id")
  key            String?
  reason         String?
  level          String    @default("audit") @db.VarChar(16)
  metadata       Json?

  @@index([timestamp(sort: Desc)], name: "idx_staff_events_timestamp")
  @@index([staffComponent],        name: "idx_staff_events_component")
  @@index([agentId],               name: "idx_staff_events_agent")
  @@index([entityType, entityId],  name: "idx_staff_events_entity")

2. Run `prisma migrate dev --name create_staff_events_table` — generates the SQL migration file.
3. Review the generated SQL against §2.1 above to confirm it matches.
4. Commit both the `prisma/schema.prisma` change and the generated migration file.

**Migration in production**: Run `prisma migrate deploy` (not `migrate dev`) in any non-development environment.

**`CHECK` constraints**: Prisma does not generate `CHECK` constraints from the schema model — these are added manually to the generated SQL migration file after `prisma migrate dev` generates the base migration. Verify this step explicitly.

## 3. Adapter Layer

### 3.1 Design Principles

- Runs in the same process as the API server via `setInterval` — no separate worker thread in Phase 1.
- Never blocks the main event loop: all DB operations are async, awaited inside the interval callback.
- Detects new rows by comparing a stored timestamp cursor against `created_at` on `knowledge_base` and `archive`.
- Does not produce duplicate events on restart: the cursor is persisted in the `staff_events` table itself (the timestamp of the last-inserted synthetic event), not in memory.

### 3.2 Cursor Strategy

The adapter does not store a separate cursor file. Instead, it reads the most recent `timestamp` from `staff_events` at startup to know where to resume from:

```typescript
// src/lib/staff-events-adapter/cursor.ts

async function loadCursor(): Promise<{ kbCursor: Date; archiveCursor: Date }> {
  // Find the most recent synthetic event inserted by the adapter
  // We use the metadata.sourceCreatedAt field to track what KB/archive row timestamps
  // have already been processed — but for simplicity in Phase 1, use the adapter's
  // own last-run timestamp as the cursor.

  // Read the timestamp of the most recently inserted staff_event whose
  // actionType is one of the adapter-generated types
  const adapterActionTypes = ['write_created', 'write_replaced', 'entry_archived'];

  const row = await prisma.$queryRaw<{ max_ts: Date | null }[]>`
    SELECT MAX(timestamp) AS max_ts
    FROM staff_events
    WHERE action_type = ANY(${adapterActionTypes})
  `;

  const lastAdapterEventTs = row[0]?.max_ts;

  if (lastAdapterEventTs) {
    // Resume: look for KB/archive rows created after the last event we processed
    // Use the metadata.sourceCreatedAt if available; fall back to the event timestamp
    return {
      kbCursor:      lastAdapterEventTs,
      archiveCursor: lastAdapterEventTs,
    };
  }

  // First run: start from 60 seconds ago to backfill recent activity
  // (avoid processing the entire table history on first startup)
  const INITIAL_LOOKBACK_MS = 60_000;
  const initialCursor = new Date(Date.now() - INITIAL_LOOKBACK_MS);
  return {
    kbCursor:      initialCursor,
    archiveCursor: initialCursor,
  };
}
```

**Cursor precision limitation**: The cursor above uses the adapter event's `timestamp` (insertion time) to determine what source rows to look at next. This works correctly when the adapter runs continuously. On restart, if the adapter was down while KB/archive rows were being written, it resumes from the last adapter event's timestamp — which may miss rows written after the last adapter event but before the crash. To fix this precisely in Phase 2, store `metadata.sourceCreatedAt` (the `created_at` of the source KB/archive row) in each synthetic event and use that as the cursor.

For Phase 1: this is an acceptable limitation. Document it clearly.

### 3.3 Polling Loop (Pseudocode)

```
ADAPTER STARTUP:
  cursor ← loadCursor()   // Read from staff_events table

EVERY 2 SECONDS (setInterval):
  now ← current timestamp

  // Batch 1: New KB rows
  newKbRows ← SELECT * FROM knowledge_base
               WHERE created_at > cursor.kbCursor
               ORDER BY created_at ASC
               LIMIT 100

  FOR each row IN newKbRows:
    // Check if a KB fact with this entity+key existed before this row's created_at
    // (i.e., was this a write_created or write_replaced?)
    priorExists ← any archive row with same entity_type + entity_id + key
                  AND archived_at <= row.created_at
                  AND archived_reason = 'superseded'

    IF priorExists:
      actionType ← 'write_replaced'
    ELSE:
      actionType ← 'write_created'

    INSERT INTO staff_events:
      staff_component = 'Librarian'
      action_type     = actionType
      agent_id        = row.agent_id
      source          = row.source
      entity_type     = row.entity_type
      entity_id       = row.entity_id
      key             = row.key
      reason          = 'Detected by control plane adapter'
      level           = 'audit'
      metadata        = {
        confidence:      row.confidence,
        valuePreview:    TRUNCATE(JSON.stringify(row.value_raw), 200),
        sourceCreatedAt: row.created_at.toISOString(),
        sourceRowId:     row.id
      }

  // Update KB cursor to newest processed row
  IF newKbRows.length > 0:
    cursor.kbCursor ← MAX(newKbRows.map(r => r.created_at))

  // Batch 2: New archive rows
  newArchiveRows ← SELECT * FROM archive
                   WHERE created_at > cursor.archiveCursor
                   ORDER BY created_at ASC
                   LIMIT 100

  FOR each row IN newArchiveRows:
    INSERT INTO staff_events:
      staff_component = 'Archivist'
      action_type     = 'entry_archived'
      agent_id        = row.agent_id
      source          = row.source
      entity_type     = row.entity_type
      entity_id       = row.entity_id
      key             = row.key
      reason          = row.archived_reason ?? 'Detected by control plane adapter'
      level           = 'audit'
      metadata        = {
        archivedReason:  row.archived_reason,
        archivedFactId:  String(row.id),
        sourceCreatedAt: row.created_at.toISOString()
      }

  // Update archive cursor
  IF newArchiveRows.length > 0:
    cursor.archiveCursor ← MAX(newArchiveRows.map(r => r.created_at))
```

### 3.4 Adapter Implementation

```typescript
// src/lib/staff-events-adapter/index.ts
import { prisma } from '../db';
import { loadCursor } from './cursor';

let adapterInterval: NodeJS.Timeout | null = null;
let cursor: { kbCursor: Date; archiveCursor: Date } | null = null;
let isPolling = false;  // Guard against overlapping polls

const POLL_INTERVAL_MS = 2000;

export async function startAdapter(): Promise<void> {
  cursor = await loadCursor();
  console.log('[staff-events-adapter] Starting. KB cursor:', cursor.kbCursor.toISOString());

  adapterInterval = setInterval(async () => {
    if (isPolling) return;  // Skip if previous poll is still running
    isPolling = true;
    try {
      await pollOnce(cursor!);
    } catch (err) {
      console.error('[staff-events-adapter] Poll error:', err);
      // Do not crash — log and continue on next interval
    } finally {
      isPolling = false;
    }
  }, POLL_INTERVAL_MS);
}

export function stopAdapter(): void {
  if (adapterInterval !== null) {
    clearInterval(adapterInterval);
    adapterInterval = null;
    console.log('[staff-events-adapter] Stopped.');
  }
}
```

```typescript
// src/lib/staff-events-adapter/poller.ts
async function pollOnce(cursor: { kbCursor: Date; archiveCursor: Date }): Promise<void> {
  await Promise.all([
    pollKnowledgeBase(cursor),
    pollArchive(cursor),
  ]);
}

async function pollKnowledgeBase(cursor: { kbCursor: Date }): Promise<void> {
  const newRows = await prisma.$queryRaw<KnowledgeBaseRow[]>`
    SELECT id, entity_type, entity_id, key, agent_id, source, confidence, value_raw, created_at
    FROM knowledge_base
    WHERE created_at > ${cursor.kbCursor}
    ORDER BY created_at ASC
    LIMIT 100
  `;

  if (newRows.length === 0) return;

  const events = await Promise.all(newRows.map(async (row) => {
    // Determine write_created vs write_replaced:
    // A row is a "replace" if there is an archive row for the same entity+key
    // with archived_reason = 'superseded' and archived_at ≈ row.created_at
    const priorArchived = await prisma.$queryRaw<{ count: bigint }[]>`
      SELECT COUNT(*) as count
      FROM archive
      WHERE entity_type   = ${row.entity_type}
        AND entity_id     = ${row.entity_id}
        AND key           = ${row.key}
        AND archived_at   > ${new Date(row.created_at.getTime() - 5000)}  -- within 5s before
        AND archived_at   <= ${new Date(row.created_at.getTime() + 5000)} -- within 5s after
    `;

    const isReplacement   = Number(priorArchived[0].count) > 0;
    const actionType      = isReplacement ? 'write_replaced' : 'write_created';
    const valuePreview    = truncateJson(row.value_raw, 200);

    return {
      staffComponent: 'Librarian' as const,
      actionType,
      agentId:    row.agent_id ?? null,
      source:     row.source   ?? null,
      entityType: row.entity_type,
      entityId:   row.entity_id,
      key:        row.key,
      reason:     'Detected by control plane adapter (Phase 1)',
      level:      'audit' as const,
      metadata: {
        confidence:      row.confidence,
        valuePreview,
        sourceCreatedAt: row.created_at.toISOString(),
        sourceRowId:     String(row.id),
      },
    };
  }));

  // Batch insert all new events
  if (events.length > 0) {
    await prisma.$executeRaw`
      INSERT INTO staff_events (staff_component, action_type, agent_id, source, entity_type, entity_id, key, reason, level, metadata)
      SELECT * FROM jsonb_to_recordset(${JSON.stringify(events)}::jsonb)
        AS t(staff_component TEXT, action_type TEXT, agent_id TEXT, source TEXT,
             entity_type TEXT, entity_id TEXT, key TEXT, reason TEXT, level TEXT, metadata JSONB)
    `;
  }

  // Advance cursor
  cursor.kbCursor = newRows[newRows.length - 1].created_at;
}
```

**Batch insert**: Rather than inserting one row at a time, the adapter batches all events from a single poll into one `INSERT ... SELECT * FROM jsonb_to_recordset(...)`. This reduces DB round trips. The `jsonb_to_recordset` pattern is safe because the values are passed as a parameterized JSONB literal — not string-interpolated SQL.

**Alternative simpler batch insert**: If `jsonb_to_recordset` is unfamiliar or causes type mapping issues, fall back to individual `INSERT` statements in a `for...of` loop inside a transaction:

```typescript
await prisma.$transaction(
  events.map(e => prisma.staffEvent.create({ data: e }))
);
```

This is less efficient but easier to debug. Use whichever approach the team is comfortable with; optimize later.

**`write_replaced` detection**: The heuristic used is: an archive row for the same entity+key exists within 5 seconds of the KB row's `created_at`. This is approximate — the exact timing depends on how quickly the Archivist moves superseded facts. If this heuristic produces false positives or negatives, fall back to always emitting `write_created` for new KB rows and document the limitation.

```typescript
async function pollArchive(cursor: { archiveCursor: Date }): Promise<void> {
  const newRows = await prisma.$queryRaw<ArchiveRow[]>`
    SELECT id, entity_type, entity_id, key, agent_id, source, archived_reason, created_at
    FROM archive
    WHERE created_at > ${cursor.archiveCursor}
    ORDER BY created_at ASC
    LIMIT 100
  `;

  if (newRows.length === 0) return;

  for (const row of newRows) {
    await prisma.staffEvent.create({
      data: {
        staffComponent: 'Archivist',
        actionType:     'entry_archived',
        agentId:        row.agent_id  ?? null,
        source:         row.source    ?? null,
        entityType:     row.entity_type,
        entityId:       row.entity_id,
        key:            row.key,
        reason:         row.archived_reason ?? 'Detected by control plane adapter (Phase 1)',
        level:          'audit',
        metadata: {
          archivedReason:  row.archived_reason ?? null,
          archivedFactId:  String(row.id),
          sourceCreatedAt: row.created_at.toISOString(),
        },
      },
    });
  }

  cursor.archiveCursor = newRows[newRows.length - 1].created_at;
}
```

### 3.5 Adapter Startup Integration

The adapter must be started after the Express server is listening and the Prisma client is connected:

```typescript

import { startAdapter, stopAdapter } from './lib/staff-events-adapter';

const server = app.listen(port, async () => {
  console.log(`Control plane listening on port ${port}`);
  await startAdapter();
});

// Graceful shutdown
process.on('SIGTERM', () => {
  stopAdapter();
  server.close();
});
process.on('SIGINT', () => {
  stopAdapter();
  server.close();
});
```

**Phase 1 limitation — Attendant and Resolutionist events**: The adapter cannot reconstruct Attendant session lifecycle events or Resolutionist decision events from DB state alone. These events require native emitter injection into the upstream Staff components (Phase 2). This limitation is documented here and must be surfaced in:
1. The `/events` endpoint's OpenAPI description (comment in code)
2. The UI empty state when no Attendant/Resolutionist events are present

## 5. `GET /api/control-plane/events/stream` (SSE)

### 5.1 SSE Response Setup

```typescript
// src/routes/control-plane/events.ts — SSE handler

const sseHandler = async (req: Request, res: Response, next: NextFunction) => {
  // Step 1: Validate params before opening the stream
  // (Invalid params → send error JSON before switching to SSE mode — simpler for client)
  let params: ValidatedStreamParams;
  try {
    params = validateStreamParams(req.query);
  } catch (err: unknown) {
    if ((err as ApiError).code === 'INVALID_PARAM') {
      return res.status(400).json({ error: (err as ApiError).message, code: 'INVALID_PARAM' });
    }
    return next(err);
  }

  // Step 2: Check staff_events table exists
  try {
    await assertStaffEventsTableExists();
  } catch (err: unknown) {
    if ((err as ApiError).code === 'EVENTS_TABLE_MISSING') {
      return res.status(503).json({ error: (err as ApiError).message, code: 'EVENTS_TABLE_MISSING' });
    }
    return next(err);
  }

  // Step 3: Set SSE headers — from this point, the response is committed
  res.setHeader('Content-Type',  'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection',    'keep-alive');
  res.setHeader('X-Accel-Buffering', 'no');  // Disable nginx buffering if behind a proxy
  res.flushHeaders();  // Send headers immediately — opens the SSE stream

  // ...
};
```

**`res.flushHeaders()` is critical**: Without this call, Express buffers the response headers until the first `res.write()`. This means the client does not receive the SSE content-type or the heartbeat until the first event arrives. `flushHeaders()` sends headers immediately, establishing the stream.

### 5.2 Cursor Initialization

```typescript
  // Step 4: Determine initial cursor
  // Priority: Last-Event-ID header → 'since' param → now (live only)
  let pollCursor: Date;
  const lastEventId = req.headers['last-event-id'] as string | undefined;

  if (lastEventId) {
    // Resume: find the timestamp of the event with this eventId
    // to resume polling from that point
    try {
      const resumeRow = await prisma.staffEvent.findUnique({
        where:  { eventId: lastEventId },
        select: { timestamp: true },
      });
      pollCursor = resumeRow?.timestamp ?? new Date();
    } catch {
      pollCursor = new Date();  // Fallback: resume from now if lookup fails
    }
  } else if (params.since) {
    pollCursor = new Date(params.since);
  } else {
    pollCursor = new Date();  // Default: live events only
  }
```

**`Last-Event-ID` resume semantics**: The cursor is set to the timestamp of the event with that `eventId`. The next poll runs `WHERE timestamp > cursor` — note `>` (strictly after), not `>=`. This means the event at the cursor itself is not re-emitted on reconnect. This is correct: the client already received the event with `Last-Event-ID`; it wants events that came after it.

### 5.3 Polling Loop

```typescript
  // Step 5: Set up polling interval and heartbeat
  let lastHeartbeatTime = Date.now();
  const POLL_INTERVAL_MS      = 1000;
  const HEARTBEAT_INTERVAL_MS = 15_000;

  const poll = async () => {
    try {
      const newRows = await queryNewEvents(pollCursor, params);

      for (const row of newRows) {
        const event = serializeStaffEvent(row);
        // SSE wire format per spec: data line then id line, blank line separator
        res.write(`data: ${JSON.stringify(event)}\n`);
        res.write(`id: ${event.eventId}\n`);
        res.write('\n');
        lastHeartbeatTime = Date.now();  // Reset heartbeat timer on any event
      }

      if (newRows.length > 0) {
        pollCursor = newRows[newRows.length - 1].timestamp;
      }

      // Heartbeat: send keep-alive comment if no events in the last 15 seconds
      if (Date.now() - lastHeartbeatTime >= HEARTBEAT_INTERVAL_MS) {
        res.write(': keep-alive\n\n');
        lastHeartbeatTime = Date.now();
      }
    } catch (err) {
      // DB connection lost or other fatal error
      console.error('[events/stream] Poll error:', err);
      // Emit error SSE event before closing
      res.write(`event: error\n`);
      res.write(`data: ${JSON.stringify({ error: 'Database connection lost', code: 'DB_UNAVAILABLE' })}\n`);
      res.write('\n');
      // Clean up and close — browser SSE protocol will auto-reconnect
      cleanup();
      res.end();
    }
  };

  const intervalId = setInterval(poll, POLL_INTERVAL_MS);

  // Step 6: Cleanup on client disconnect
  const cleanup = () => {
    clearInterval(intervalId);
    console.log(`[events/stream] Client disconnected. Active streams: (decremented)`);
  };

  req.on('close', cleanup);

  // Run first poll immediately (don't wait for first interval tick)
  await poll();
```

### 5.4 Query for New Events

```typescript
async function queryNewEvents(
  cursor: Date,
  params: ValidatedStreamParams
): Promise<StaffEventRow[]> {
  const where: Prisma.StaffEventWhereInput = {
    timestamp: { gt: cursor },
    level:     params.level ?? 'audit',
  };

  if (params.staffComponent) where.staffComponent = params.staffComponent;
  if (params.actionType)     where.actionType     = params.actionType;
  if (params.agentId)        where.agentId        = params.agentId;
  if (params.entityType)     where.entityType     = params.entityType;
  if (params.entityId)       where.entityId       = params.entityId;

  return prisma.staffEvent.findMany({
    where,
    orderBy: { timestamp: 'asc' },  // ASC for streaming — emit oldest-first
    take:    50,                     // Per spec: LIMIT 50 per poll
  });
}
```

**`ORDER BY timestamp ASC` for streaming**: The SSE stream emits events in chronological order (oldest first within each poll batch), which is the natural expectation for a live log view. This is the opposite of the list endpoint which returns newest-first.

**`LIMIT 50` per poll**: If more than 50 events arrive in a single 1-second interval (e.g., a bulk agent session), the cursor advances by 50 rows per poll. The backlog is caught up over multiple poll cycles. This prevents flooding the client with thousands of events at once if the stream is reconnected after a gap.

### 5.5 Resource Leak Prevention Summary

| Risk | Mitigation |
|---|---|
| `setInterval` not cleared on disconnect | `clearInterval(intervalId)` called in `cleanup()`, triggered by `req.on('close')` |
| Multiple `cleanup()` calls | `setInterval` handle is idempotent if cleared multiple times (no-op after first clear) |
| `res.write()` after `res.end()` | After calling `cleanup()` and `res.end()` in the error handler, no further poll will run (interval is cleared) |
| Overlapping polls | Poll is `async` and awaits DB; if a poll takes longer than 1 second, the next interval tick calls `poll()` again concurrently. Guard with `isPolling` flag similar to the adapter: |

```typescript
let isPolling = false;

const poll = async () => {
  if (isPolling) return;
  isPolling = true;
  try {
    // ... poll logic
  } finally {
    isPolling = false;
  }
};
```

| Risk | Mitigation |
|---|---|
| Large result set flooding client | `LIMIT 50` per poll — backlog caught up over multiple ticks |
| Memory leak from unclosed connections | Express's `req.on('close')` fires on both explicit client disconnect and TCP RST — cleanup is guaranteed |

## 7. Known Phase 1 Limitations (Document in Code and Handoff)

1. **Attendant events absent**: Session lifecycle events (`handshake_completed`, `session_expired`, etc.) cannot be reconstructed from DB state. Requires native emitter injection into the Attendant component (Phase 2 per PM decision OQ-1).

2. **Resolutionist events absent**: Resolution decision events cannot be reconstructed from DB state. Same reason.

3. **`write_replaced` detection is approximate**: The heuristic (archive row within 5s window) may misclassify some writes. If an implementation simplification is needed, always emit `write_created` and document.

4. **Adapter cursor precision on restart**: Events written between the last adapter run and a crash may be missed. The cursor improvement (using `sourceCreatedAt` in metadata) is a Phase 2 fix.

5. **No event deduplication beyond cursor**: If two adapter processes run simultaneously (e.g., during a deployment rollover), duplicate events may be inserted. A unique constraint on `(action_type, metadata->>'sourceRowId')` could prevent this but is not implemented in Phase 1.

## 8. Open Questions (Flagged for Implementer)

1. **`created_at` column on `knowledge_base`**: Does `knowledge_base` have a `created_at` column suitable for adapter polling? If the column is named differently (e.g., `inserted_at`, `timestamp`), update the adapter queries. This is a hard dependency — the adapter cannot function without a polling cursor column.
2. **`created_at` on `archive`**: Same question for the `archive` table.
3. **LISTEN/NOTIFY**: PostgreSQL LISTEN/NOTIFY could reduce the adapter's polling latency from ~2 seconds to <100ms. This requires adding `NOTIFY` calls to the Librarian and Archivist — an upstream Iranti change. Evaluate for Phase 2 if operators find 2-second latency unacceptable.
4. **Prisma model field name for `eventId`**: With the `@map("event_id")` annotation in the Prisma schema, the Prisma client exposes `prisma.staffEvent.findUnique({ where: { eventId: ... } })`. Confirm this accessor works after applying the schema change.
5. **`res.flush()` vs `res.flushHeaders()`**: Some Express/Node.js versions require `res.flush()` (from `compression` middleware) in addition to `res.flushHeaders()` to ensure SSE data is not buffered by the response compression layer. If SSE events are delayed, check whether `compression` middleware is active and either disable it for the `/events/stream` route or call `res.flush()` after each `res.write()`.
