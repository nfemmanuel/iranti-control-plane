## Problem Statement

No `staff_events` table or event stream endpoint exists. The Staff Activity Stream UI (CP-T014) requires both a queryable history endpoint and a live SSE stream. Phase 1 cannot wait for upstream Staff component emitter changes (OQ-1 decision deferred to Phase 2), so this ticket implements an adapter layer that reconstructs Librarian write events and Archivist archive events by detecting new rows in `knowledge_base` and `archive`.

## User Value

Operators can watch Staff activity in real time and query past events ŌĆö without raw DB access. Making Librarian writes and Archivist archival events visible is the most immediate value: operators can confirm that their agent writes are reaching Iranti's memory and that no silent conflicts are occurring.

## Scope

**In:**
- DB migration: create `staff_events` table per CP-T001 ┬¦3 schema (SQL DDL with all indexes)
- `GET /api/control-plane/events` ŌĆö paginated, filterable Staff event list per CP-T002 Group 7
- `GET /api/control-plane/events/stream` ŌĆö SSE endpoint with DB polling loop per CP-T001 ┬¦5 / CP-T002 Group 7
- Adapter layer: populate `staff_events` by detecting new `knowledge_base` and `archive` rows
  - `write_created` event when a new KB row is inserted
  - `write_replaced` event when an existing KB row is updated/superseded
  - `entry_archived` event when a new `archive` row is inserted
  - Adapter polls the DB (or uses Postgres LISTEN/NOTIFY if available) to detect new rows since last check
- SSE heartbeat (`: keep-alive` every 15 seconds)
- SSE cursor resume via `Last-Event-ID` header
- Error SSE event emitted before stream close on fatal errors

**Out:**
- Native Staff component emitter injection into Librarian/Attendant/Archivist/Resolutionist (Phase 2 per OQ-1 decision)
- Attendant session lifecycle events (cannot be reconstructed from DB state ŌĆö document limitation clearly)
- Resolutionist decision events (cannot be reconstructed from DB state in Phase 1 ŌĆö document limitation)
- Event replay, export

**Per PM OQ-1 decision:** Phase 1 adapter layer covers Librarian write events and Archivist archive events only. Attendant and Resolutionist events require native emitter injection (Phase 2). This limitation must be clearly documented in the endpoint's OpenAPI description and in the UI empty state guidance.

- [ ] SSE heartbeat `: keep-alive` emitted every 15 seconds when no new events.
- [ ] SSE `Last-Event-ID` resume: client reconnects with header ŌåÆ server resumes from that `eventId` cursor.
- [ ] Adapter layer populates `write_created` events for new `knowledge_base` rows (includes entityType, entityId, key, agentId, source, confidence in metadata).
- [ ] Adapter layer populates `write_replaced` events when an existing KB row is superseded.
- [ ] Adapter layer populates `entry_archived` events for new `archive` rows.

- [ ] All Librarian/Archivist adapter events have correct `level: "audit"`.

- [ ] Stream handles DB connection loss gracefully: emits error SSE event, closes connection (browser will auto-reconnect via SSE protocol).

## Risks

- Adapter layer polling adds a background task to the process ŌĆö must not block the main request loop. Use `setInterval` or a worker-thread approach.
- Duplicate event risk on restart: cursor tracking must use an `eventId` or `timestamp` to avoid re-emitting events that were already inserted before the restart.
- Attendant and Resolutionist events absent from Phase 1 stream ŌĆö this is expected per PM decision, but the UI must communicate this clearly (not look like a bug).
- If Iranti writes happen at high frequency (e.g., during a bulk agent session), the adapter polling interval may produce bursts of events. The SSE endpoint should batch these with a small delay rather than flooding the browser.
- Prisma schema may not expose a `created_at` timestamp on knowledge_base rows suitable for adapter polling ŌĆö confirm against actual schema.

## Open Questions

- Does the Prisma schema for `knowledge_base` include a `createdAt` or `updatedAt` timestamp that the adapter can use as a polling cursor?
- Is PostgreSQL LISTEN/NOTIFY available and appropriate as a lower-latency alternative to polling? If so, evaluate vs the 1-second poll approach.
- Should the adapter run in the same process as the API server, or as a separate background worker? (Recommendation: same process, separate setInterval loop in Phase 1.)
