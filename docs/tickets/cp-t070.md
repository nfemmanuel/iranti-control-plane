# CP-T070 — Global Keyboard Shortcuts: View Navigation Hotkeys

**Type:** Feature
**Priority:** P3
**Assigned:** frontend_developer
**Phase:** 4 (Wave 10)
**Status:** PM-ACCEPTED — 2026-03-21
**Epic:** CP-E015 (Iranti Desktop)

## Summary

Phase 2 shipped the command palette (Cmd+K) and an inline shortcuts reference. But the only global keyboard shortcut is Cmd+K itself. Desktop-class tools — Terminal, VS Code, Docker Desktop — let power users navigate without touching the mouse.

This ticket adds a set of view-navigation hotkeys that work from any view, displayed in the command palette's shortcut reference. The scope is deliberately narrow: this is not a full keyboard interface overhaul. It is the specific addition of `G + <letter>` navigation shortcuts (the Gmail / GitHub pattern) that feel natural to technical operators.

The Cmd+K palette (CP-T024/CP-T042) already exists and already shows a shortcuts panel — this ticket populates it with real navigation shortcuts and wires the actual handlers.

## What to Build

### Keyboard Shortcut Handlers

Add a `useViewNavigationShortcuts` hook at `src/client/src/hooks/useViewNavigationShortcuts.ts`.

The hook listens for a `G` prefix pattern: press `G`, then within 1500ms press a letter to navigate to a view. This is the standard "go to" shortcut pattern used in GitHub, Linear, Notion, and similar tools. It does not conflict with typing in inputs because the handler must check that no input, textarea, or contenteditable element has focus.

```typescript
export function useViewNavigationShortcuts(): void
```

The hook uses `window.addEventListener('keydown', ...)` and must:
1. Ignore events when focus is inside an input, textarea, select, or `contenteditable` element.
2. Ignore events when the Cmd/Ctrl/Alt/Meta modifier is held (so it doesn't interfere with browser shortcuts).
3. Detect the `G` key press and set a "go mode" ref (`goModeActive: boolean`) with a 1500ms timeout that clears it.
4. When in go mode, detect the second keypress and navigate accordingly.

**Navigation shortkeys (G + key):**

| Keys | Destination | Label |
|------|-------------|-------|
| `G H` | `/overview` | Home |
| `G M` | `/memory` | Memory Explorer |
| `G A` | `/archive` | Archive |
| `G T` | `/activity` | Activity (Staff) |
| `G L` | `/logs` | Logs |
| `G I` | `/instances` | Instances |
| `G D` | `/health` | Health & Diagnostics |
| `G X` | `/metrics` | Metrics |
| `G C` | `/conflicts` | Conflicts |
| `G P` | `/providers` | Providers |
| `G G` | `/agents` | Agent Registry |
| `G S` | `/getting-started` | Getting Started |

Rationale for key choices: `H` = Home, `M` = Memory, `A` = Archive, `T` = sTaff/acTivity, `L` = Logs, `I` = Instances, `D` = Diagnostics, `X` = metrX (X is unusual but memorable), `C` = Conflicts, `P` = Providers, `G` double = aGents, `S` = Setup.

### Visual Feedback for Go Mode

When go mode is active (the operator has pressed `G` but not yet the second key), show a small status chip in the bottom-right corner of the screen — above the toast container (CP-T069) — that reads:

```
⌨  go mode — press a key
```

The chip auto-hides when go mode resolves (navigation occurs or timeout expires). Style: dark pill with emerald border, monospace text, low opacity transition in/out.

If CP-T069 is not yet shipped, the go mode indicator is still implemented — position it at bottom-right without worrying about z-index conflicts.

### Command Palette Integration

Update `src/client/src/components/shell/CommandPalette.tsx` to include the navigation shortcuts in the shortcuts panel (the panel shown when the user types `?` in the palette).

Add a "Navigation shortcuts" section to the shortcuts list with each `G + key` combination listed as `G → <key>` alongside its destination label.

### AppShell Integration

Call `useViewNavigationShortcuts()` inside `AppShell.tsx` — adjacent to the existing `useEffect` that handles Cmd+K.

Also render the go mode indicator chip. The chip needs access to the go mode state, which should be returned from the hook:

```typescript
export function useViewNavigationShortcuts(): { goModeActive: boolean }
```

## Acceptance Criteria

### AC-1 — G + H navigates to /overview

From any view, press `G` then `H` within 1500ms. App navigates to `/overview`. No modifiers held.

### AC-2 — All 12 shortcuts navigate correctly

Each `G + key` pair listed in the table navigates to the correct route. Test at minimum: G+H (Home), G+M (Memory), G+D (Health), G+L (Logs), G+G (Agents).

### AC-3 — Shortcuts do not fire when focus is in an input

When a text input, textarea, select, or contenteditable element has focus, pressing `G` does not activate go mode. This is critical — operators frequently type in filter fields.

### AC-4 — Go mode indicator appears

When `G` is pressed with no input focused, the go mode chip ("go mode — press a key") appears in the bottom-right corner. It disappears when the second key is pressed or after 1500ms.

### AC-5 — Modifier keys are excluded

Pressing Ctrl+G, Cmd+G, Alt+G, or any combination with a modifier key does NOT activate go mode.

### AC-6 — Shortcuts appear in command palette shortcuts panel

When the user types `?` in the command palette, a "Navigation shortcuts" section shows the `G → <key>` pairs with destination labels.

### AC-7 — tsc clean

`tsc --noEmit` passes in `src/client`. No `any`.

### AC-8 — No regressions

Cmd+K still opens the command palette. Existing shortcuts (defined in CP-T042) still work. No conflicts with browser default shortcuts.

## Implementation Notes

- The "go mode" timeout should be 1500ms — long enough for deliberate two-key sequences, short enough not to feel sticky.
- Use `useRef` for `goModeActive` inside the hook to avoid stale closure issues in the event listener. Return `goModeActive` as a stateful value (`useState`) for the visual indicator.
- The focus check: `document.activeElement?.tagName` being `INPUT`, `TEXTAREA`, `SELECT`, or the element having `contenteditable="true"` attribute.
- Key comparison: use `e.key` (not `e.keyCode`). Case-insensitive: `e.key.toLowerCase() === 'g'`.
- The go mode chip should be rendered as a portal or a fixed-position element in AppShell to avoid layout interference.
- Do not use a library for this — the implementation is straightforward with native browser events.
