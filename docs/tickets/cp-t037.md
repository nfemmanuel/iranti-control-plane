## Problem Statement

The Phase 1 Staff Activity Stream is implemented as a scrollable event log. It is accurate and filterable, but it does not convey the feel of *watching something happen*. When an operator runs `iranti_write` or observes an agent handshake and wants to watch Iranti process it in real time, the current UI does not differentiate between:

- "no events have happened in 10 minutes" (quiet system)
- "20 events just fired in the last 3 seconds" (active system under load)

The event list looks the same in both cases unless the operator is actively watching the scroll position. There is no ambient signal of system activity — no pulse, no velocity indicator, no auto-scroll that tracks the live head.

Additionally, the Phase 1 `useEventStream` hook implements a live vs tail mode toggle (`isLive` / `isPaused`), but the visual affordance for this is minimal. Users do not clearly understand whether the stream is tracking the live head or is paused showing historical events.

**User signal (2026-03-20):** User explicitly requested the ability to "see Iranti Staff logs as they work" — meaning the operational feel of watching agents run, not just browsing after the fact. This is distinct from browsability and search (which the current stream handles well). It is about immediacy, presence, and ambient activity awareness.

This ticket is the UX complement to CP-T025 (native emitter injection). CP-T025 delivers the data (all 4 Staff components, <200ms latency). This ticket delivers the presentation that makes that data feel live.

## Scope

**In:**

### 1. Activity Pulse Indicator

A small animated indicator in the stream header that conveys whether events are actively firing.

- **Active state**: A pulsing dot (using the existing CSS animation vocabulary — conic-gradient or a simple opacity pulse in `var(--color-accent-primary)`) that animates faster when events arrive in rapid succession.
- **Quiet state**: The dot dims to a neutral color when no events have fired in the last 5 seconds.
- **Disconnected state**: The dot turns red/error color when the SSE connection is down (already detected by `useEventStream` `connectionStatus`).

Implementation: track `lastEventAt` timestamp in the hook. Compute time-since-last-event every 500ms. Drive the CSS animation speed based on event recency:
- Event in last 1 second: "hot" — fast pulse
- Event in last 5 seconds: "active" — slow pulse
- No event in 5+ seconds: "idle" — static dim dot

### 2. Event Velocity Counter

A small counter showing events-per-minute (or events in the last 60 seconds) in the stream header alongside the pulse indicator.

- Format: "4 events/min" or "12 events/min". Zero-state: "No activity".
- Computed over a rolling 60-second window using timestamps of events in the ring buffer.
- Resets visually to zero after 60 seconds of no events.
- Tooltip on hover: "Events from Staff in the last 60 seconds"

This gives operators an at-a-glance sense of system load without counting events themselves.

### 3. Auto-Scroll with Pause-on-Hover

The stream should auto-scroll to the newest event while in live mode, and pause scrolling when the user hovers over the event list or manually scrolls up.

- **Live mode (default)**: New events are prepended (newest-first, matching current Phase 1 behavior) and the view tracks the top of the list (no scroll needed since newest is always first).
  - If the stream is currently in oldest-first order, preserve the order and auto-scroll to bottom instead. Check Phase 1 implementation and match.
- **Pause on hover**: When the user's mouse is over the event list, do not inject new events into the DOM (buffer them). Show a "N new events — click to resume" banner at the top of the list.
  - This is distinct from the existing pause/resume toggle — this is transient hover-based buffering, not a deliberate mode switch.
  - The banner dismisses automatically when the mouse leaves the event list.
- **Resume on mouse leave**: When the mouse leaves the event list, flush buffered events into the DOM (up to MAX_PAUSE_BUFFER, already defined as 1000 in the Phase 1 hook).

Note: The Phase 1 `useEventStream` hook already implements a `isPaused` / `pauseBuffer` model. This ticket extends it with hover-triggered pausing rather than replacing the existing manual pause toggle.

### 4. Live Mode Visual Clarity

Make it unambiguous whether the stream is in live mode (tracking the head) or tail mode (browsing history):

- **Live mode badge**: A small "LIVE" badge with the pulse dot in the stream header — green-tinted, uses `var(--color-accent-primary)`.
- **Tail mode indicator**: When manually paused or scrolled into history, the badge changes to "PAUSED" with a neutral color and a "Resume live" button.
- The existing manual pause/resume toggle button from Phase 1 is preserved — this ticket only adds visual clarity around it, not a different toggle mechanism.

### 5. "Watching [Agent]" Mode (stretch goal — if time permits)

When the `agentId` filter is active, add a contextual header: "Watching [agent_id] — [N events] in this session". This reinforces the "following an agent's activity" use case.

- Event persistence or history search (Phase 3)

## Risks

- **Hover detection conflicts with click-to-expand event rows**: The hover-pause interaction must not conflict with users clicking to expand individual event rows. Use `onMouseEnter`/`onMouseLeave` on the container, not on individual rows.
- **Velocity counter accuracy with Phase 1 polling**: The rolling 60-second velocity counter will show artificially bursty readings when the Phase 1 polling adapter delivers batches of 5-10 events at once every 2 seconds. Accept this in Phase 2 — it will smooth out once CP-T025 delivers native emission. Add a tooltip note: "Event timing may appear bursty — will improve with CP-T025."
- **Animation performance**: The pulse dot uses CSS animations. Avoid JavaScript-driven animations for the pulse itself — use CSS `@keyframes` with a `animation-duration` property driven by a CSS custom property updated from React state. This keeps rendering on the GPU compositor thread.
