## Problem Statement

When the Iranti Resolutionist creates an escalation — a conflict between an existing fact and a new challenger that it cannot resolve automatically — the operator currently has no UI surface to review it. Pending escalations live in filesystem files inside the escalation directory or as pending entries in the `archive` table. Reviewing them requires direct filesystem access or SQL. Resolving them requires understanding the Resolutionist's expected resolution pathway and invoking it correctly from the CLI.

This creates a high-friction, high-stakes workflow: the most important operational decision an Iranti operator makes (resolving a conflicting fact) is also the one most buried behind infrastructure. PRD FR10 explicitly requires that pending escalations be accessible without file editing, and FR11 requires all resolutions go through the Resolutionist pathway, not direct DB writes.

This ticket implements the Conflict and Escalation Review UI: a dedicated view in the control plane where operators can see all pending escalations, compare the conflicting facts side-by-side, and resolve them through the correct Resolutionist pathway.

## User Value

An operator who sees unexpected behavior from an agent can immediately open the Escalation Review, find the pending conflict for the relevant entity, read both the existing fact and the challenger in a clear side-by-side layout, understand why the conflict was escalated (the escalation reason), and choose a resolution — all without leaving the browser. The resolved escalation goes through the Resolutionist, preserving Iranti's consistency model and auditability. This transforms a multi-minute filesystem excavation into a sub-60-second operator action.

## Scope

**In:**

**Pending Escalation List (main view):**
- A filterable, sortable list of all pending escalations. Columns: entity (type/id), key, escalation reason, conflict type, age (time since escalation created), source, challenging agent.
- Filter by: entity type, escalation reason, conflict type (value_conflict / confidence_conflict / source_conflict), age (older than N hours).
- Sort by: age (default, oldest first), entity, source.
- Each row is clickable and opens the side-by-side comparison view (either as a drawer from the right, or navigation to a dedicated escalation detail page — implementation choice, see Open Questions).
- Count badge in the navigation sidebar showing the number of pending escalations (updates in real time via polling or SSE — same event source as the Staff Activity Stream).

**Side-by-Side Comparison View:**
- Two-column layout: left column = existing fact, right column = challenger fact.
- Each column shows all relevant fields: entity (type/id), key, valueRaw (formatted JSON), summary, confidence, source, createdBy, createdAt, validFrom, reason/note.
- Confidence difference highlighted: if challenger confidence is materially higher or lower than existing, show the delta with a visual indicator (e.g., "↑ 15 points" in green or "↓ 8 points" in amber).
- Escalation reason shown prominently above the comparison (e.g., "Conflict: both facts claim to be current for the same entity/key with conflicting values").
- Affected entity links: the entity (type/id) shown in both columns links to the entity detail page in Memory Explorer (opens in main content area if the comparison is in a drawer, or in a new section if it is a full page).

**Resolution Actions:**
- Three resolution buttons at the bottom of the comparison view:
  1. "Keep Existing" — resolves by retaining the existing fact, archiving the challenger with `resolutionState = resolved_keep_existing`.
  2. "Accept Challenger" — resolves by promoting the challenger fact to active, archiving the existing with `resolutionState = resolved_accept_challenger`.
  3. "Use Custom Value" — opens an inline edit panel where the operator can type a custom value. The custom value becomes the resolved fact; both existing and challenger are archived.
- All three resolution actions call the Resolutionist API pathway — no direct DB writes.
- Before executing any resolution: a confirmation step inline (a brief "Are you sure?" with the action described in plain language — not a modal). Confirm with `Enter` or the "Confirm" button, cancel with `Escape` or "Cancel".
- After resolution: the escalation immediately moves from the Pending list to the Resolved list. The comparison view closes and returns to the Pending list with focus on the next escalation.

**Resolved Escalation List (secondary tab in the same view):**
- Paginated list of resolved escalations. Columns: entity, key, resolution type (kept existing / accepted challenger / custom value), resolved at, resolved by (agent/user).
- Read-only — no actions. Supports filtering by resolution type and date range.
- Accessible via a tab or toggle at the top of the Escalation view.

**Resolutionist Action Log:**
- Below the resolved escalation list (or as a third tab): a chronological log of Resolutionist processing events sourced from the Staff Activity Stream filter. Shows: timestamp, entity, action, outcome.

**Archive Consumption Queue Visibility:**
- After a resolution is confirmed: a brief informational banner appears below the confirmation: "Resolution queued for Archivist processing. The Archivist will consume this resolution on next cycle." This sets correct expectations — the archived entries will not disappear immediately.
- A "Pending Archivist consumption" count shown in the Health view (implementation detail for CP-T015/Health view to pick up).

**Out:**
- Bulk resolution of multiple escalations at once (Phase 3)
- Custom resolution templates or macros (Phase 3)
- Escalation assignment to other team members (multi-user, Phase 3)
- Direct editing of the `archive` table (this is always out of scope — FR11)
- Creating new escalations from the UI (out of scope — escalations are created by the Librarian/Resolutionist automatically)

## Dependencies

- Phase 1 fully accepted: the backend API surface and database schema must be stable. The escalation review backend endpoints depend on the same connection and authentication patterns established in Phase 1.
- **Resolutionist API surface specification:** This is a hard dependency. Before backend_developer begins implementation, a spec must exist for:
  - `GET /api/control-plane/escalations?status=pending` — returns list of pending escalations with all comparison fields
  - `GET /api/control-plane/escalations/:id` — returns full detail for a single escalation (both sides, all fields)
  - `POST /api/control-plane/escalations/:id/resolve` — body: `{ resolution: "keep_existing" | "accept_challenger" | "custom", customValue?: any }`. Must go through Resolutionist pathway, not direct DB write.
  - If the Resolutionist does not yet expose a programmatic resolution pathway (it may only be CLI-invoked today), this must be flagged to the PM and system_architect before implementation begins.
- CP-T017 (shell) must be complete — the escalation view uses the Phase 1 shell layout.

- [ ] Pending escalation count is shown as a badge in the sidebar navigation next to the Escalations nav item. Badge updates without a full page reload (polling interval acceptable, target < 30 seconds refresh).
- [ ] The side-by-side comparison view shows both existing and challenger facts with all relevant fields as specified (entity, key, valueRaw, summary, confidence, source, createdBy, createdAt, validFrom, reason/note). Confidence delta is displayed.
- [ ] Resolution action (Keep Existing, Accept Challenger, or Use Custom Value) calls the Resolutionist API pathway — confirmed by the backend_developer with a test showing no direct DB writes occurred.
- [ ] After resolution, the escalation moves immediately from the Pending list to the Resolved list. No manual page refresh required.
- [ ] Resolved escalation list is accessible and shows at least: entity, key, resolution type, resolved at. Pagination works for large resolved lists.
- [ ] "View in Memory Explorer" links from the comparison view navigate to the correct entity detail page.
- [ ] Custom value resolution: the operator can type a custom JSON value. The UI validates that the input is valid JSON before allowing confirmation. Invalid JSON shows an inline error.
- [ ] Confirmation step exists before any resolution is executed. The operator must explicitly confirm the action.
- [ ] TypeScript compiles without errors. No `any` types in the escalation component tree.

## Risks

- **Resolutionist resolution pathway may not exist programmatically.** If the Resolutionist only supports resolution via CLI commands or file-system operations, the backend_developer must either: (a) wrap the CLI in a subprocess call from the API, or (b) escalate to system_architect for a proper API pathway. Direct DB writes are not acceptable regardless of the path. This must be resolved in the first 2 days of task pickup.
- **Escalation file format may change.** If escalation data is stored in filesystem markdown files (as described in PRD), the API must translate from file format to a structured response. This is fragile if the file format is not formally specified. The backend_developer must confirm the escalation data source (DB table vs filesystem) before building the list endpoint.
- **Race condition on resolution.** If two operators have the same escalation open simultaneously (possible in a future multi-user scenario), the second resolution will fail. For Phase 2 (single local user), this is not a concern — but the error handling for a "resolution already applied" response must be implemented cleanly: show an inline message "This escalation was already resolved" and refresh the list.

## Open Questions

Recommendation: right-side drawer for quick review, since operators often need to process multiple escalations in sequence. The drawer allows seeing the list context while reviewing. PM to confirm.
- Should the Escalation Review be accessible from a dedicated nav item in the sidebar (adding a 7th nav item, requiring CP-T017 amendment), or should it be a sub-section within the existing Instances or Health views? Recommendation: dedicated nav item `Escalations` (replacing the Settings placeholder for Phase 2). PM to confirm nav structure.
- What is the correct API surface for the Resolutionist resolution pathway? Needs backend_developer investigation and system_architect input before implementation.
