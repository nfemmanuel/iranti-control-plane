## Problem Statement

The Staff Activity Stream in Phase 1 shows only Librarian events (and possibly Archivist events, depending on Phase 0 adapter completeness). The Attendant and Resolutionist adapters are not yet instrumented in Phase 1. This is a known, planned scope limitation — not a defect.

However, without explicit labeling, all three user personas will interpret the absence of Attendant and Resolutionist events as a broken stream or a broken Iranti installation:

- **Dev (Power User)** will specifically look for Archivist events after running a retention scenario. When none appear, he will cross-check in Adminer, confirm the adapter is not instrumented, and correctly flag the stream as incomplete. His trust in the product's accuracy is on the line.
- **Priya (Technical Founder)** is evaluating product completeness. An unlabeled gap reads as immaturity or deception — exactly the category of issue that ends an evaluation.
- **Marcus (Solo Dev)** wants passive reassurance that something is happening. A stream that shows some events but not others, with no explanation, plants anxiety rather than confidence.

The fix is not to instrument the missing adapters before v0.1.0 (that is Phase 2 work). The fix is to be honest: label exactly which components are streaming and which are not, frame it as Phase 2 scope, and make this label persistent so it cannot be missed.

This is the single highest-severity usability risk identified in Phase 1 user research. PM agrees without reservation.

---

## Scope

**In:**

Add a persistent, non-dismissible indicator to the Staff Activity Stream view that:

1. Clearly states which Staff components are currently emitting events and appearing in the stream.
2. Clearly states which Staff components are not yet instrumented.
3. References Phase 2 as when full Staff observability will be available.
4. Is visible on initial load without any user interaction (no expand, no hover, no scroll).
5. Persists in the UI until Phase 2 event emitters for Attendant and Resolutionist are live and integrated. It is not dismissible by the user.

**Recommended implementation:**

A persistent header bar directly above the event list — visually subordinate to the stream itself but always in the viewport. Example content:

> **Phase 1 event coverage:** Librarian ✓ | Archivist ✓ (if instrumented) | Attendant — Phase 2 | Resolutionist — Phase 2

The indicator should use an informational visual style (not a warning — this is expected, not a problem). Use the Iranti token system's informational color (blue/grey, not amber).

The specific label for Archivist should reflect actual Phase 1 adapter state at ship time:
- If Archivist events are instrumented by Phase 1 ship: show Archivist ✓
- If Archivist events are not instrumented: show Archivist — Phase 2

The frontend developer should confirm Archivist instrumentation state with the backend developer at implementation time.

**Out:**

- Instrumenting the Attendant or Resolutionist adapters (Phase 2 work).
- Making the indicator dismissible.
- Building a settings page for coverage configuration.
- Adding per-event source labels to individual events (that is a separate stream improvement, not this ticket).

## Acceptance Criteria

- [ ] The Staff Activity Stream view has a visible coverage indicator on initial load, without any user interaction.
- [ ] The indicator explicitly names Attendant and Resolutionist as not yet instrumented in Phase 1.
- [ ] The indicator references Phase 2 as when full Staff observability ships (either as a text label or a link to docs/roadmap — link preferred if a target page exists).
- [ ] The indicator is not dismissible — it persists across page reloads and navigation until Phase 2 emitters are live.
- [ ] The indicator does not obstruct or crowd the event stream — it is a header element or persistent sidebar note, not a blocking modal, overlay, or toast.
- [ ] The indicator uses an informational visual style (not amber warning, not red error) — this is expected scope, not a failure state.
- [ ] Librarian is labeled as actively streaming (✓ or equivalent). Archivist label reflects actual Phase 1 adapter state confirmed with the backend developer.
- [ ] The indicator is visible in both light and dark mode with sufficient contrast.
- [ ] TypeScript compiles without errors for the new component.

## Dependencies

- Phase 0 event bus implementation — the coverage label must reflect actual adapter completeness at Phase 1 ship. Confirm with backend_developer which adapters are live before finalizing label content.
- Iranti token system — informational color token must be defined. Confirm with frontend design system.

---

## Risks

- **Label goes stale after Phase 2 ships:** The indicator must be driven by a runtime capability flag, not hardcoded text. If it is hardcoded, it will incorrectly persist after Phase 2 emitters are live. Implementation should check a backend capability endpoint or feature flag for instrumented components, and hide or update the indicator dynamically. If a capability endpoint does not exist, use a hardcoded label for Phase 1 with a clear code comment marking it for removal when Phase 2 adapters ship.
- **Archivist state unknown at implementation time:** Coordinate with backend_developer to confirm whether Archivist events are included in Phase 1. Do not assume — label must be accurate.
