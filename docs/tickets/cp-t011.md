## Problem Statement

No API exists for instance metadata or health diagnostics. The Instance Manager UI (CP-T015) and Health & Diagnostics UI (CP-T016) cannot be built without stable backend data. The instance aggregation logic — reading `.env.iranti`, filesystem stats, and health endpoint status — is non-trivial cross-platform work that requires its own implementation ticket.

## User Value

Operators can see the state of all local Iranti instances (config, running status, env health, integration status) and a consolidated diagnostics summary without running CLI commands or inspecting raw files. This is FR4 (Instance Awareness) and FR4/ER2 (Fast Time to Root Cause) from the PRD.

## Scope

**In:**
- `GET /api/control-plane/instances` — aggregate instance metadata per CP-T002 Group 5 / CP-T003 spec
- `GET /api/control-plane/instances/:instanceId/projects` — per-instance project bindings (stubbed per PM OQ-2 decision if binding registry not confirmed)
- `GET /api/control-plane/health` — structured health summary per CP-T002 Group 6
- All `InstanceMetadata`, `InstanceListResponse`, `HealthSummaryResponse` schemas per CP-T002/CP-T003 specs
- Instance discovery: registry file (`~/.iranti/instances.json`) + fallback candidate path scan
- Cross-platform path handling: macOS, Windows (using `os.homedir()`, `path.join()`)
- All edge cases from CP-T003 §4: missing env file, offline instance, missing project path, malformed integration config, multiple instances
- All 10 health checks from CP-T002 Group 6: `db_reachability`, `db_schema_version`, `vector_backend`, `anthropic_key`, `openai_key`, `default_provider_configured`, `mcp_integration`, `claude_md_integration`, `runtime_version`, `staff_events_table`

**Out:**
- Write operations on instances or env files
- Project binding creation/removal
- Auth

**Scoped per PM OQ-2 decision:** Project bindings (`ProjectBinding[]` in InstanceMetadata) will be an empty array or stub in Phase 1 unless system_architect confirms a binding registry source exists during CP-T006 spike. The endpoint returns `projects: []` with a metadata flag `"projectBindingsUnavailable": true` so the UI can show an informative empty state rather than appearing broken.

## Acceptance Criteria

- [ ] `GET /api/control-plane/instances` returns `InstanceListResponse` per CP-T003 §5.1–5.5. `discoveredAt` and `discoverySource` fields present.
- [ ] Instance discovery checks `~/.iranti/instances.json` (or `%USERPROFILE%\.iranti\instances.json` on Windows) first, falls back to candidate path scan if registry absent/empty.
- [ ] All `InstanceMetadata` fields populated correctly: `instanceId` (derived from hash of runtimeRoot), `runtimeRoot`, `database` (credentials redacted), `configuredPort`, `runningStatus`, `irantVersion`, `envFile`, `integration`.
- [ ] Edge case: missing `.env.iranti` — `envFile.present: false`, database/integration fields null, `keyCompleteness: null`. Instance health check still attempted.
- [ ] Edge case: instance not running — `runningStatus: "stopped"` (TCP refused) or `"unreachable"` (timeout). All other config fields still populated from env file.
- [ ] Edge case: project path missing from filesystem — `projectExists: false`, all filesystem-derived project fields null.
- [ ] Edge case: malformed `.mcp.json` — `mcpConfigPresent: true`, `mcpConfigHasIranti: false`, `mcpConfigError` field populated.
- [ ] Health check endpoint returns `HealthSummaryResponse` with `overall` status and all 10 required checks.
- [ ] `overall` status logic: `healthy` = all ok, `degraded` = any warn, `error` = any error.
- [ ] Health check `staff_events_table` returns `warn` if `staff_events` table does not exist (CP-T001 migration not applied yet).
- [ ] Database credentials never returned in raw form — `urlRedacted` shows `postgresql://***@host:port/dbname` only.
- [ ] API key values never returned — only boolean presence.
- [ ] Both endpoints work correctly on macOS and Windows file paths (use `path.join()`, `os.homedir()`, no hardcoded separators).
- [ ] `/instances` always returns 200 even if instances array is empty or all stopped. Per-instance errors surfaced within each object, not as HTTP errors.
- [ ] `/health` always returns 200 even if overall status is `error`.
- [ ] Health check TCP timeout for instance health probe: 500ms maximum.

## Risks

- `IRANTI_DEFAULT_PROVIDER` env var name is assumed — must confirm against actual `.env.iranti` conventions in running Iranti before implementing.
- `lastActiveTimestamp` DB query (KB entries per project) may not be feasible without confirmed agent-to-project mapping in schema — stub this field as `null` in Phase 1 if not confirmable.
- Windows `%USERPROFILE%` path resolution must be tested on actual Windows (developer environment is Windows 11).
- pgvector health check may require connecting to DB and querying `pg_available_extensions` — ensure this does not add significant latency to the health endpoint.

## Open Questions

- What is the exact env var name for default provider in Iranti? (`IRANTI_DEFAULT_PROVIDER`? `DEFAULT_PROVIDER`?) — confirm against `.env.iranti.example` in Iranti repo.
- Does `~/.iranti/instances.json` exist in any current Iranti installations? If not, the fallback scan path will be the only discovery mechanism in Phase 1.
- Should the health endpoint be accessible to the frontend before any instance is discovered (e.g., before `.env.iranti` exists)? Yes — it should always return something useful.
