## Problem Statement

Instance metadata (runtime root, DB connection target, port, env file state) and project binding metadata (bound projects, Claude/Codex integration file presence, integration status) are currently scattered across the filesystem, env files, and runtime config. The control plane needs a unified queryable surface for this to power the Instance & Project Manager view (FR4). Without a specified aggregation design, the backend_developer cannot implement the instance endpoint and frontend_developer cannot design the Instance Manager view, because neither knows where the data lives or what it will look like when aggregated.

## User Value

Operators can see all their instances and bound projects from one place, understand what is configured vs what is missing, and identify integration health without manually inspecting individual config files or running CLI commands. This is a foundational transparency feature: it makes Iranti's own configuration legible without infrastructure expertise.

## Scope

In:
- Map every metadata field listed in PRD Section 4 (Instance and Project Manager) to its current source: which file, DB table, env var, or runtime state provides it
- Define the aggregation strategy: push (Iranti writes metadata to a queryable store at runtime), pull (control plane reads sources directly at request time), or hybrid — with explicit tradeoffs for each approach
- Document edge cases and how they should be handled: missing env file, unbound project, offline instance, partially configured integration, multiple instances on the same host
- Produce an output schema: what the aggregated metadata object looks like in JSON, with all fields named and typed
- Document OS-level considerations: filesystem paths that differ between macOS, Linux, and Windows
- Produce spec at `docs/specs/instance-metadata-aggregation.md`

Out:
- Implementation of the aggregation layer
- UI for displaying instance data (that is CP-E004 in Phase 1)
- Changes to upstream Iranti instance management (note as proposals if needed, flag clearly)

## Dependencies

- None — this ticket can run in parallel with CP-T001 and CP-T002
- Access to upstream Iranti filesystem and config conventions is needed to accurately map field sources; architect should consult the running codebase rather than speculating

## Acceptance Criteria

- [ ] Every metadata field listed in PRD Section 4 (Instance and Project Manager) is explicitly mapped to its current source — identified by file path, DB table name, env variable name, or runtime state description; no field is listed as "TBD" or "unknown" without a documented investigation note
- [ ] Aggregation strategy is specified: at least push vs pull (and hybrid if applicable) are compared with concrete pros/cons; the recommended strategy is identified with rationale tied to latency, reliability, and implementation complexity
- [ ] The following edge cases are explicitly addressed in the spec with defined behavior: (1) missing or malformed env file, (2) unbound project directory, (3) instance process not running, (4) partially configured integration (e.g., CLAUDE.md present but MCP config missing), (5) multiple instances on the same host
- [ ] Output schema is concrete: a JSON object with all field names, types, and nullability documented — specific enough that frontend_developer can design the Instance Manager view without requiring further backend input
- [ ] Spec addresses OS-level path differences (macOS vs Linux vs Windows) for any filesystem-based field sources
- [ ] PM has reviewed and approved the spec

## Risks

- Some metadata fields may only be accessible via filesystem reads that are slow at request time — spec must explicitly address whether caching is required, at what granularity, and what staleness is acceptable to the operator
- Instance discovery may require filesystem scanning (e.g., finding all Iranti runtime roots on the host), which is brittle across OS environments and user configurations; spec must choose between registry-based discovery and scan-based discovery and document the tradeoff
- If upstream Iranti does not maintain a registry of running instances, the aggregation layer may need to implement one — which could be a larger scope than a pure read spec; any such finding should be flagged to the PM before the spec is finalized

## Open Questions

- Should instance discovery be registry-based (Iranti maintains a canonical list of known instances) or filesystem-scan-based (control plane scans known root directories)? Registry is more reliable; scan is more discoverable for instances that were not set up through the control plane.
- What is the right staleness tolerance for instance metadata? Should it refresh on every request (accurate but potentially slow), or be cached with a periodic refresh (fast but potentially stale)?
- Does upstream Iranti today persist any instance registration metadata that this spec can build on, or does it need to be designed from scratch?
