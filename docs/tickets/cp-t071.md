**Type:** Feature

**Phase:** 5 (Wave 11)

**Depends on:** CP-T068 (shell infrastructure, nav pattern)

## Description

Iranti v0.2.16 added durable interrupted-session recovery with four lifecycle transitions:

- `POST /memory/checkpoint` — an agent saves its current task state
- `POST /memory/resume` — an agent or operator resumes a checkpointed session
- `POST /memory/complete` — marks a session as done
- `POST /memory/abandon` — marks a session as abandoned

Before v0.2.16, sessions had no persistent state. Now they do — but the control plane surfaces none of it. Operators cannot see which sessions are active, which are interrupted mid-task, which have been abandoned, or which completed successfully. When an agent crashes or a session is interrupted, the operator has no visibility without querying the Iranti database directly.

This ticket builds a **Session Recovery** view in the control plane. The primary operator question it answers: *"What sessions exist and which ones need attention?"*

## What to Build

### Backend — Session Proxy Endpoint

Add a new route to the control plane's server:

```
GET /api/control-plane/sessions
```

**Query parameters:**
- `state` (optional): filter by session state — `interrupted`, `checkpointed`, `complete`, `abandoned`, `all` (default: `all`)
- `agentId` (optional): filter to a specific agent
- `limit` (optional, default 50): cap result count

**Implementation approach:**

The Iranti API does not have a list-sessions endpoint. Sessions are stored as KB facts or in a sessions table (determined by Iranti's internal schema). The proxy must query Iranti's data layer. Two options:

1. **If Iranti exposes a sessions endpoint** (check `/agents` or `/kb` for session data): proxy it directly.
2. **If sessions are stored as KB facts under a known entity type** (e.g., `entityType: 'session'`): call `GET /kb/query?entityType=session` and filter by state.
3. **Fallback:** Query the control plane's local Postgres for session facts if the Iranti proxy returns empty — sessions written via `iranti_ingest` or `iranti_write` to a session entity type may be in the local KB.

The backend developer must check the actual Iranti session storage mechanism before implementing. If sessions are not queryable via a single endpoint, document what was found and propose the closest available surface.

**Response shape:**

```typescript
interface SessionRecord {
  sessionId: string;
  agentId: string;
  state: 'interrupted' | 'checkpointed' | 'complete' | 'abandoned' | 'unknown';
  task: string | null;
  startedAt: string | null;
  lastCheckpointAt: string | null;
  completedAt: string | null;
  abandonedAt: string | null;
}

interface SessionsResponse {
  sessions: SessionRecord[];
  total: number;
  fetchedAt: string;
}
```

**Error handling:** If Iranti is unreachable or sessions are not queryable, return `{ sessions: [], total: 0, fetchedAt: "...", error: "..." }` with HTTP 200. Never 500.

### Frontend — Sessions View

Create a new view at route `/sessions`.

**Nav item:** Add "Sessions" to the AppShell nav with a clock/history icon, after Agents and before Health.

**Layout:**

```
┌─────────────────────────────────────────────────────┐
│  SESSIONS                                           │
│  State filter tabs: All | Interrupted | Active | Complete | Abandoned
├─────────────────────────────────────────────────────┤
│  SESSION LIST                                       │
│  [state badge] sessionId    agentId    task preview │
│               started: relative time               │
│               last checkpoint: relative time        │
│  [Resume] [Abandon]  (shown only for resumable states)
└─────────────────────────────────────────────────────┘
```

**State badge colors:**
- `interrupted` / `checkpointed` — amber (needs attention)
- `complete` — emerald (done)
- `abandoned` — muted gray (terminal)
- `unknown` — rose (data gap)

**Session list:**
- Paginated at 20 per page
- State filter tabs update the query param: `/sessions?state=interrupted`
- Each row shows sessionId (monospace, truncated), agentId, task preview (first 60 chars), started relative time, last checkpoint relative time
- Clicking a row expands a detail panel (inline, not a new route)

**Session detail panel (expanded inline):**
- Full sessionId (copyable)
- agentId (links to Agent Registry)
- Full task text (scrollable)
- Timeline: startedAt → lastCheckpointAt → completedAt / abandonedAt (whichever applies)
- Raw JSON of checkpoint data (collapsible)
- **Resume action**: button visible when state is `interrupted` or `checkpointed`. Calls `POST /api/control-plane/sessions/:sessionId/resume`. Requires the agentId as a field (pre-filled from session data). Shows confirmation before executing.
- **Abandon action**: button visible when state is `interrupted` or `checkpointed`. Calls `POST /api/control-plane/sessions/:sessionId/abandon`. Shows confirmation: "This will mark the session as abandoned and cannot be undone."

**Empty state:** "No sessions found. Sessions appear when agents call `iranti_ingest` or use the Iranti SDK checkpoint flow." Link to Getting Started guide.

**Loading state:** Skeleton rows (3), skeleton state badges.

---

### Backend — Session Action Endpoints

```
POST /api/control-plane/sessions/:sessionId/resume
POST /api/control-plane/sessions/:sessionId/abandon
```

Both accept `{ agentId: string }` in the body. Both proxy to the Iranti `/memory/resume` and `/memory/abandon` routes respectively.

**Authorization note:** These are write actions. They should require the control plane's operator API key scope — use the same auth pattern as other write endpoints.

**Response:** Forward the Iranti API response. On error, return `{ error: "..." }` with appropriate HTTP status.

## Acceptance Criteria

### AC-1 — Sessions list endpoint exists

`GET /api/control-plane/sessions` returns HTTP 200 with `SessionsResponse` shape. `sessions` array may be empty if no sessions exist. TypeScript clean, no `any`.

### AC-2 — State filter works

`?state=interrupted` returns only sessions in interrupted/checkpointed states. `?state=all` (or no param) returns all. Query parameter is validated — invalid state values return a clear error.

### AC-3 — Sessions view renders at /sessions

Navigating to `/sessions` renders the Sessions view. The AppShell nav has a Sessions item. The state filter tabs work and update the URL query param.

### AC-4 — Session list shows correct fields

Each session row shows: state badge (correct color), sessionId (monospace), agentId, task preview, started relative timestamp, last checkpoint relative timestamp.

### AC-5 — Session detail panel expands

Clicking a session row expands a detail panel inline. Full task text, full sessionId (copyable), agentId (linked to Agent Registry), timeline, and raw checkpoint JSON (collapsible) are shown.

### AC-6 — Resume action works for resumable sessions

For sessions in `interrupted` or `checkpointed` state, a Resume button is shown. Clicking it shows a confirmation. Confirming calls the resume endpoint and updates the session state in the UI (refetch or optimistic update).

### AC-7 — Abandon action works for resumable sessions

Abandon button is shown for resumable states. Confirmation required. On confirm, calls the abandon endpoint and updates state in the UI.

### AC-8 — Empty state is clear

When no sessions exist (or Iranti returns empty), the empty state message is shown with guidance for operators.

### AC-9 — Iranti unreachable degrades gracefully

If the Iranti proxy fails, the sessions list shows an error notice ("Could not load sessions — Iranti may be unreachable") rather than crashing. The page renders.

### AC-10 — tsc clean

`tsc --noEmit` passes in both `src/server` and `src/client`. No `any`.

### AC-11 — No regressions

All existing routes and views continue to function. Nav item does not displace existing items — it is inserted between Agents and Health.
