## Problem Statement

The Phase 1 Instance Manager shows integration status for project bindings: whether `.mcp.json`, `CLAUDE.md`, and provider key configuration are present and valid. But it is entirely read-only. When a status shows a problem — a missing `.mcp.json`, a stale `CLAUDE.md`, a misconfigured provider key reference — the user must leave the control plane, find the relevant files, edit them manually, and return to verify.

This is the exact failure mode the PRD was designed to prevent. PRD section 4 ("Instance and Project Manager — Desired Actions") explicitly lists:
- "rebind project to instance"
- "run doctor for a specific instance"
- "run upgrade / setup actions from the UI"

And the Phase 1 Health dashboard (CP-T028) classifies a missing `.mcp.json` as a Warning-level health event with the label: "Claude integration file not found for project [name]. Slash command memory access may not work." But Phase 1 offers no repair path from that warning — the user has to figure out the fix themselves.

This ticket adds targeted repair actions to the Instance Manager and Health dashboard for the integration states that are both common and fixable without filesystem expertise. The goal is not a full write surface — it is a narrow set of guided repair flows that eliminate the most common "user drops to filesystem" scenarios.

## Scope

**In:**

### Repair Actions (Phase 2 initial set)

The following repair actions are in scope for this ticket. Each action must be triggered from the relevant integration status row in the Instance Manager or Health dashboard, not from a global settings panel.

**1. Regenerate `.mcp.json`**

Condition: project binding shows `.mcp.json` missing or malformed.

Action: a "Repair" or "Regenerate" button next to the status that triggers:
`POST /api/control-plane/instances/:instanceId/projects/:projectId/repair/mcp-json`

Backend behavior: generates a fresh `.mcp.json` file at the project root using the current instance configuration (database host, port, runtime root). Returns the generated file path and a diff-style summary of what was written.

Frontend behavior: shows a confirmation modal before writing (showing which project directory will be modified and what will be written), a loading state while the write completes, and a success confirmation with the file path. Offers a "View in File Explorer" (OS-level open, not a built-in file browser).

**2. Regenerate `CLAUDE.md` integration block**

Condition: project shows `CLAUDE.md` integration block missing or outdated (version mismatch detected).

Action: "Update" button triggering:
`POST /api/control-plane/instances/:instanceId/projects/:projectId/repair/claude-md`

Backend behavior: appends or replaces the Iranti integration block in `CLAUDE.md` at the project root. Must preserve any user-authored content above or below the Iranti block — do not overwrite the full file. Confirm with backend_developer the exact detection and replacement heuristic (likely a comment-delimited block).

Frontend behavior: confirmation modal showing the diff (what block will be replaced, what will be inserted), loading state, success confirmation with file path.

**3. Re-run Instance Doctor**

Condition: any health probe or manual user intent.

Action: "Run Doctor" button on each instance row in the Instance Manager, triggering:
`POST /api/control-plane/instances/:instanceId/doctor`

Backend behavior: runs the same health checks as `GET /api/control-plane/health` but scoped to one instance, then returns structured results including pass/fail per check, error messages, and suggested remediation steps. This is not a live shell command — it is a structured diagnostic endpoint.

Frontend behavior: opens a Doctor Results panel (drawer from right side) showing each check as a row with pass/fail icon, plain-English description, and (if failed) a one-line suggested fix. If a fix has a repair action (e.g., regenerate `.mcp.json`), the repair button is available inline in the results panel.

### Backend: Repair Endpoints

The three endpoints listed above must:
- Require explicit confirmation query parameters (e.g., `?confirm=true`) to prevent accidental mutations from misdirected GET requests.
- Return structured success/error responses including the file path modified, the action taken, and a `revertable: false` warning since file writes are not transactional.
- Log all repair actions to the `staff_events` table (or equivalent audit log) with `agentId: control_plane_repair`, `source: control_plane`, and the action payload.
- Fail explicitly if the project directory is not writable, returning a structured error with the permission problem and suggested OS-level fix.

**Out:**
- General file browser or editor within the control plane (Phase 3, if at all).
- Repair actions for database schema, migrations, or runtime configuration (these require infrastructure access beyond the control plane's appropriate scope).
- Automated healing without user confirmation — all repair actions must require explicit user confirmation before writing.
- Bulk repair across all projects at once.

## Risks

- **File mutation is irreversible:** The repair actions write to the user's project filesystem. A bug in the `.mcp.json` generator or the `CLAUDE.md` block replacement logic could corrupt a file the user cares about. The confirmation modal and the `revertable: false` warning are essential. Consider writing a backup copy (`.mcp.json.bak`) before overwriting for Phase 2.
- **CLAUDE.md block detection fragility:** Detecting and replacing the Iranti integration block in `CLAUDE.md` requires a reliable delimiter. If the user has edited or removed the comment delimiters, the detection will fail. The endpoint must fail safely (return an error, not write the wrong block).
- **Server permissions vary by deployment:** On some macOS setups, the Iranti server process may not have write access to project directories outside its own runtime root. Test against a standard local install before finalizing the backend implementation.
