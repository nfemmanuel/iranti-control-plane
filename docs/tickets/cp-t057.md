# CP-T057 — Entity Detail: WhoKnows Contributor Panel

**Type:** Feature
**Priority:** P3
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 5)
**Status:** PM-ACCEPTED 2026-03-21 — backend + frontend (minor type gap noted, non-blocking)
**Epic:** CP-E002 (Memory Explorer)

## Summary

Iranti exposes `GET /memory/whoknows/:entityType/:entityId` which returns every agent that has contributed facts to a given entity, along with their write counts and last contribution timestamp. This endpoint is currently invisible in the control plane — the Entity Detail view shows what Iranti believes about an entity but gives no indication of _who_ contributed those beliefs.

This ticket surfaces the WhoKnows data in the Entity Detail view as a "Contributors" panel, answering the operator question: "Which agents have shaped this entity's memory?"

## Background

From the Iranti API route map (cross-repo audit 2026-03-21):
- `GET /memory/whoknows/:entityType/:entityId` — requires `memory:read` scope
- Returns: list of `{ agentId, writeCount, lastContributedAt }` objects for all agents that have written facts for this entity

Use case: An operator is debugging a conflicted or unexpected fact on `project/iranti_control_plane`. The Entity Detail view shows the current value but not who wrote it. The WhoKnows panel answers "which agents touched this entity and how recently?" without requiring a full staff logs search.

This is distinct from `createdBy` on individual facts (which shows the agent for a single fact) — WhoKnows gives an aggregate contributor picture across all facts for the entity.

## Acceptance Criteria

### Backend (backend_developer)

#### AC-1 — `GET /api/control-plane/kb/whoknows/:entityType/:entityId` endpoint

Proxy `GET /memory/whoknows/:entityType/:entityId` from the connected Iranti instance. Auth: forward `X-Iranti-Key` with `memory:read` scope (same pattern as existing `kb.ts` proxies).

Response shape (pass through from Iranti, normalized):
```json
{
  "contributors": [
    {
      "agentId": "backend_developer",
      "writeCount": 14,
      "lastContributedAt": "2026-03-21T09:00:00.000Z"
    }
  ],
  "total": 3
}
```

If Iranti returns 404 (entity not found) or 401 (no memory:read scope), return HTTP 503 with `{ error: "...", code: "WHOKNOWS_UNAVAILABLE" }`. Return an empty `{ contributors: [], total: 0 }` if Iranti returns an empty list (valid state — entity exists but has no attributed contributors).

#### AC-2 — Route registration

Register the new endpoint in `src/server/routes/control-plane/kb.ts` (or a new `whoknows.ts` alongside it). Reference `src/server/routes/control-plane/providers.ts` for the proxy pattern. Add to `src/server/routes/control-plane/index.ts`.

#### AC-3 — TypeScript clean

No `any`, no type assertions without comment. `tsc --noEmit` passes with zero new errors.

### Frontend (frontend_developer)

#### AC-4 — "Contributors" panel in Entity Detail view

In the Entity Detail view (`/memory/:entityType/:entityId`), add a "Contributors" panel below the entity facts section. On mount, call `GET /api/control-plane/kb/whoknows/:entityType/:entityId`.

Render as a compact horizontal card row (or vertical list for narrow viewports):
- Each contributor: agent ID (bold, monospace or badge), write count (integer), last contributed (relative time, absolute on hover)
- Sort by `writeCount` descending
- If an agent from this list matches an agent in the Agent Registry (CP-T051), link the agent ID to `/agents/:agentId` — if the Agent Registry is unavailable or the agent is not registered, show the ID as plain text (no broken link)

#### AC-5 — Empty states

- No contributors returned: "No attributed contributors for this entity." (do not show error)
- Endpoint returns 503 / Iranti unreachable: "Contributor data unavailable. Check that your Iranti API key has `memory:read` scope."
- Loading state: skeleton row (consistent with existing fact row loading)

#### AC-6 — Visual consistency

Panel must use existing Terminals palette CSS tokens — no hardcoded colors. Match the card/section styling of the existing Entity Detail panels (facts, relationships, temporal history).

#### AC-7 — TypeScript clean

No `any` in new component tree. `tsc --noEmit` passes.

## Implementation Notes

- The Entity Detail view is in `src/client/src/components/memory/MemoryExplorer.tsx` or a dedicated `EntityDetail.tsx` — check both
- The WhoKnows endpoint on the control plane backend proxies to `/memory/whoknows/...` on Iranti — this is the `/memory/` path, not `/kb/` — check the existing proxy routing in `index.ts` for how memory routes are handled
- The panel can be lazy-loaded (only fetch when the Entity Detail view mounts) — no eager prefetch needed
- The link to `/agents/:agentId` is a best-effort enhancement — degrade gracefully if Agent Registry is unavailable
