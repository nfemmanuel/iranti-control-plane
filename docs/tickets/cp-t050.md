# CP-T050 — Structured Staff Logs View (All Four Components)

**Type:** Feature
**Priority:** P2
**Phase:** 3
**Epic:** CP-E003 (Staff Activity Stream — Phase 3 evolution)
**Assigned:** Unassigned
**Status:** Open
**Created:** 2026-03-20

---

## Summary

Implement a persistent, queryable Staff Logs view that provides structured log history across all four Iranti Staff components (Librarian, Attendant, Archivist, Resolutionist). This is distinct from the live Activity Stream (CP-T037): the Activity Stream is a live tail with no persistent history. The Staff Logs view is a browseable, filterable, searchable log of all Staff processing events, backed by the `staff_events` database table.

---

## Background and PM Assessment

**What the Activity Stream provides (CP-T037):**
- Live SSE tail of Staff events for the current browser session
- Events for Librarian and Archivist (Phase 2, via polling); Attendant and Resolutionist events require CP-T025 native emitter
- In-session event buffer (up to 500 events per session, lost on page reload)
- Per-component filtering, keyword filter, live/paused mode
- Velocity counter, pulse indicator, hover-pause UX

**What the Activity Stream does NOT provide:**
- Persistent history — closing the browser loses all events
- Query across past sessions — there is no way to ask "what did the Archivist do last Tuesday?"
- Per-component structured log with all event fields visible (the stream shows a compact summary per row)
- Log level filtering (info, warning, error tiers within Staff events)
- Export (e.g., download log as JSONL or CSV for debugging)
- Full-text search across event payloads
- Per-agent event history (all Librarian writes by `agent_id = X` over the past week)

**User signal:** The user explicitly requested a "logs view for all Staff" — interpreted as wanting the above capabilities: a persistent, queryable log history that goes beyond the ephemeral live tail. This is a distinct product surface.

**Data source:** The `staff_events` table (implemented in CP-T039) is the backend. All Staff events written to this table are available for persistent querying. The Phase 2 Activity Stream reads from this table via SSE; this ticket creates a direct query surface over it.

**Phase rationale:** This is Phase 3 because:
1. It requires CP-T025 (native emitter) to provide meaningful data for Attendant and Resolutionist — without CP-T025, the logs view is only Librarian + Archivist events, which is already covered by the Activity Stream.
2. The Phase 2 Activity Stream live tail covers the "watch what's happening right now" use case. The persistent logs view is the "audit what happened" use case — important, but not blocking Phase 2 delivery.
3. The `staff_events` table schema may evolve as CP-T025 injection points are finalized. Building the query surface before the schema is stable risks rework.

## Scope

**In:**

### 1. Staff Logs Route (`/logs`)

A new top-level route at `/logs` in the control plane navigation. This is distinct from the Activity Stream at `/activity`.

Navigation: add "Logs" as a nav item in the AppShell sidebar. Badge: optionally show unread error-level events since last visit (stretch goal).

### 2. Log Table View

A paginated, sortable, filterable table of `staff_events` rows. Default sort: `createdAt DESC` (most recent first).

**Columns:**
| Column | Source field | Notes |
|--------|-------------|-------|
| Timestamp | `createdAt` | Relative time + absolute on hover |
| Component | `componentName` | Librarian / Attendant / Archivist / Resolutionist — color-coded badge |
| Event Type | `eventType` | e.g., `write_created`, `decay_check`, `conflict_resolved` |
| Agent | `agentId` | The agent whose action triggered the event |
| Entity | `entityType/entityId` | The entity affected (if applicable) |
| Key | `key` | The KB key affected (if applicable) |
| Summary | `payload.summary` or derived | Short human-readable description of the event |
| Level | derived | info / warning / error tier — see Level Classification below |

**Level Classification (derived, not stored in DB):**
- `error`: events where payload contains an `error` field, or eventType includes `failed`, `rejected`, `error`
- `warning`: events where eventType includes `conflict`, `decay`, `escalated`, `superseded`
- `info`: all other events

### 3. Filter Bar

- **Component filter:** Multi-select checkboxes or dropdown for Librarian, Attendant, Archivist, Resolutionist
- **Event type filter:** Text input or dropdown — shows known event types for selected components
- **Agent ID filter:** Text input
- **Entity type filter:** Text input
- **Level filter:** All / Info / Warning / Error
- **Date range filter:** "Last 1 hour", "Last 24 hours", "Last 7 days", "Custom range" (date pickers)
- **Full-text search:** Searches across `eventType`, `agentId`, `entityType`, `entityId`, `key`, and payload fields
- **Clear all filters** button

### 4. Expanded Row Detail

Clicking a row expands it (same pattern as Archive Explorer) to show:
- Full payload as formatted JSON (with syntax highlighting)
- All metadata fields (source, instanceId if available, correlationId if available)
- "View Entity" link — navigates to the entity detail page in Memory Explorer (if entityType/entityId are present)
- "View in Archive" link — navigates to the Archive Explorer filtered to this entity/key (if the event is an archival event)

### 5. Export

A "Export" button in the log view header:
- Exports the current filtered result set (up to 10,000 rows) as JSONL or CSV
- The user can choose format
- File named: `iranti-staff-logs-[ISO timestamp]-[component-filter].jsonl`

This is a debugging affordance: operators investigating agent behavior can export the log and share it with the Iranti team or use it for offline analysis.

### 6. Per-Component Log Pages (stretch goal)

Direct links from each Staff component name (in the badge column or from a dedicated "Staff" section in the nav) to a pre-filtered log view for that component:
- `/logs?component=Librarian`
- `/logs?component=Attendant`
- `/logs?component=Archivist`
- `/logs?component=Resolutionist`

These are deep-link URLs, not separate routes — the filter bar is pre-populated from the query string.

### 7. Error Surfacing (stretch goal)

If any `error`-level events have occurred in the last 24 hours, show a banner in the Health Dashboard: "N Staff errors in the last 24 hours — View Logs". This creates an ambient signal for operators who don't actively monitor the logs view.

**Out of scope:**

- Real-time SSE delivery in the Logs view — this is a query surface, not a live tail. Use the Activity Stream for live monitoring.
- Log retention policy configuration — retention is managed at the `staff_events` table level (infrastructure concern, not UI)
- Structured log ingestion from external sources — only `staff_events` rows are in scope
- Per-agent log subscriptions or alerts (Phase 3+ follow-on)
- Log forwarding to external logging services (Datadog, Logfire, etc.) — out of scope

## Acceptance Criteria

1. **`/logs` route** accessible from the AppShell sidebar navigation.

2. **Log table** renders with all 8 columns specified. Default sort is most recent first.

3. **Component filter** works correctly — selecting "Librarian only" shows only Librarian events; selecting multiple components shows events from all selected components.

4. **Date range filter** works correctly — "Last 24 hours" shows only events from the last 24 hours (based on `createdAt`).

5. **Level filter** works correctly — `error` level shows only events matching the error classification rules; `warning` shows warning-classified events.

6. **Full-text search** returns results matching searches against `eventType`, `agentId`, `entityType`, `entityId`, and `key` fields.

7. **Expanded row** shows full payload as formatted JSON. "View Entity" link present when entity fields are non-null and navigates correctly.

8. **Pagination** works for large result sets. Page size options: 25, 50, 100.

9. **Export** produces a valid JSONL or CSV file for the current filtered result (up to 10,000 rows). File is named as specified.

10. **Empty state:** When no events match the current filters, a clear empty state is shown. When the `staff_events` table is empty (e.g., CP-T025 not yet active), the empty state explains why: "No Staff events recorded yet. Events appear here once the Iranti Staff emitter is active."

11. **TypeScript compiles without errors.** No `any` in the new component tree.

12. **Both light and dark mode verified** for all new UI elements.

13. **CI green.**

## Dependencies

### Hard dependencies:
- CP-T039 (staff_events migration) — must be complete. Already done.
- CP-T010 (KB/archive backend) — complete. The Staff Logs view follows the same query patterns as the KB/archive endpoints.

### Soft dependencies (affect data richness, not the feature itself):
- CP-T025 (native Staff emitter injection) — without this, only Librarian and Archivist events are in `staff_events`. The logs view works with whatever data is present; it does not require full 4-component coverage to be useful.

### New backend work required:
- `GET /api/control-plane/logs` — a paginated query endpoint over `staff_events` with all filter params described above. This is analogous to `GET /api/control-plane/kb` and should follow the same query-building patterns.
- The export endpoint: `GET /api/control-plane/logs/export` (returns JSONL or CSV as a streaming response or a download link).

## Open Questions

**OQ-1:** Should the Staff Logs view and the Activity Stream be merged into a single unified surface with a "Live" tab and a "History" tab, or remain as separate routes? PM recommendation for Phase 3 initial implementation: keep them separate. The Activity Stream is a real-time tool; the Logs view is an audit/debugging tool. Their interaction patterns are fundamentally different. A unified view can be explored in Phase 3+.

**OQ-2:** What is the appropriate log retention window? As `staff_events` grows, the table may become large. The logs view should communicate the effective retention window to operators. Default: show all events (no server-side limit). A configurable retention window is a Phase 3+ infrastructure concern.

**OQ-3:** Should the component filter use the existing Staff component names exactly (`Librarian`, `Attendant`, `Archivist`, `Resolutionist`) or display names defined in the control plane? PM recommendation: use the exact component names from the `staff_events.componentName` field for consistency.

## Implementation Notes

- The backend `GET /api/control-plane/logs` endpoint should be implemented in a new file `src/server/routes/control-plane/logs.ts`, following the same structure as `kb.ts` and `events.ts`.
- The frontend component should be implemented at `src/client/src/components/logs/StaffLogs.tsx`.
- Level classification (info / warning / error) is a frontend-only concern derived from `eventType` and payload fields. No DB change needed.
- The export functionality can be implemented as a server-side streaming response for large datasets, or as a client-side download of the current page for simpler initial implementation. The AC requires up to 10,000 rows — client-side download of the full filtered result is the simpler path.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `staff_events` table grows very large without retention policy | Medium | Low (UX) | Pagination handles large tables; add retention guidance in docs |
| Attendant + Resolutionist events absent until CP-T025 ships | High | Low | Empty state explains the gap clearly; Librarian + Archivist events are still valuable |
| Export of 10,000 rows causes browser performance issues | Low | Low | Stream the download from the server; do not load all rows into frontend state |
| Level classification heuristics are incorrect for some event types | Medium | Low | Classification is frontend-derived and easy to update without a DB migration |
