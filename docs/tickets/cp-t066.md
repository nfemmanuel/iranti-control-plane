# CP-T066 — KB Search: Surface GET /kb/search in Memory Explorer

**Type:** Feature
**Priority:** P2 (Wave 9)
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 9)
**Status:** PM-ACCEPTED — 2026-03-21
**Epic:** CP-E014 (Knowledge Base Power Features)

## Summary

The Iranti API exposes `GET /kb/search` — a hybrid lexical + vector search endpoint that supports searching across all KB facts by value, across all entity types and IDs. The control plane currently only exposes ILIKE substring matching on `entityType` and `entityId` in the Memory Explorer list view. Operators cannot answer "what facts mention Project Iris?" or "which entities know something about onboarding?" without knowing the entity type or ID up front.

This ticket adds a dedicated Search surface to the Memory Explorer (or as a standalone `/search` route) that lets operators run a query against the full KB using the Iranti search endpoint, see ranked results with relevance scores, and navigate directly to the entity or fact that matched.

## Background

The `GET /kb/search` endpoint was documented in the cross-repo audit (2026-03-21):

- **URL:** `GET /kb/search`
- **Scope required:** `kb:read` (global scope — not namespace-scoped)
- **Params:** `query` (required), `limit` (1–50), `entityType`, `entityId`, `lexicalWeight`, `vectorWeight`, `minScore`
- **Returns:** A list of KB facts scored by hybrid lexical + vector relevance, each with `lexicalScore`, `vectorScore`, and combined `score`
- **Currently used by:** The diagnostics module (`GET /kb/search?query=test&limit=1`) as the `iranti_auth` and `vector_search_check` probes — confirming the endpoint is proxied correctly and reachable

The Memory Explorer's `search` param on `GET /kb` uses ILIKE on value fields (per the original API spec). This is entirely different — it only searches within a filtered view. The `GET /kb/search` endpoint operates across the entire KB and returns ranked results.

This gap was identified in the cross-repo audit as M1 ("No search diagnostic surface") and is referenced in the roadmap as a Phase 3+ candidate.

## What to Build

### Backend — `src/server/routes/control-plane/kb.ts`

Add a new proxy endpoint:

```
GET /api/control-plane/kb/search?query=<string>&limit=<n>&entityType=<optional>&minScore=<optional>
```

This proxies to `GET /kb/search` on the active Iranti instance, forwarding:
- `query` (required — pass-through)
- `limit` (optional, default 20, cap 50)
- `entityType` (optional filter)
- `minScore` (optional — do not expose `lexicalWeight`/`vectorWeight` in the UI at MVP; defaults are fine)

Return the Iranti response as-is. Handle graceful degradation: 503 if Iranti is unreachable, 400 if `query` is missing.

Add a TypeScript type `KBSearchResult` and `KBSearchResponse` to `src/client/src/api/types.ts`:

```typescript
export interface KBSearchResult {
  entityType: string
  entityId: string
  key: string
  valueSummary: string | null
  confidence: number
  source: string
  lexicalScore: number
  vectorScore: number
  score: number
}

export interface KBSearchResponse {
  results: KBSearchResult[]
  query: string
  total: number
}
```

Confirm the real Iranti response shape by checking the `/kb/search` implementation or testing against the live instance before finalizing the type.

### Frontend — Memory Explorer Search Mode

Add a "Search KB" mode to the Memory Explorer (`src/client/src/components/memory/MemoryExplorer.tsx`) or a new `/search` route.

**PM preference:** Extend Memory Explorer with a mode toggle rather than a new route. The Memory Explorer already has a search bar — this becomes a "cross-KB search" mode when the operator types in the top-level search input without first filtering to a specific entity type.

Alternatively, add a prominent search box above the existing filter bar that triggers global search. The design decision is left to the frontend developer, but the result must feel natural and not force the operator to navigate away from the Memory Explorer to search.

**Search results view:**

Each result shows:
- Entity type + entity ID (as a link to `/memory/:entityType/:entityId`)
- Fact key
- Value summary (truncated to ~80 chars)
- Combined relevance score as a percentage or bar (e.g., `score: 0.92` → "92% match")
- Confidence value
- Source

Results sorted by `score` descending (Iranti already returns them ranked).

**Empty state:** "No results for [query]." when `results.length === 0`.

**Error state:** If the search endpoint returns 503 (Iranti unreachable) or 403 (scope missing), show an explanatory message linking to the Health Dashboard.

**Scope warning:** Because `GET /kb/search` requires a globally-scoped API key (not entity-namespaced), add an informational note below the search input: "Full-text search requires a global-scope API key. Namespace-scoped keys will return a permission error." Show this note only when an error matching 403/SCOPE is returned — do not show it by default.

## Acceptance Criteria

### AC-1 — Backend proxy endpoint

`GET /api/control-plane/kb/search?query=<q>` proxies to Iranti `/kb/search`. Returns Iranti's response body. Handles 503 (Iranti unreachable), 400 (missing query), 403 (scope error). TypeScript clean, no `any`.

### AC-2 — TypeScript types

`KBSearchResult` and `KBSearchResponse` added to `src/client/src/api/types.ts`. Types match the real Iranti API response shape (verified against live instance or source). `tsc --noEmit` passes in both server and client.

### AC-3 — Search input triggers cross-KB query

Operator can type a query and receive results from the full KB — not just from within a selected entity type or a filtered view. The query goes to `GET /api/control-plane/kb/search`, not the ILIKE filter on the entity list.

### AC-4 — Results render correctly

Each result shows: entity type + ID (linked), fact key, value summary, score display, confidence, source. Clicking the entity link navigates to `/memory/:entityType/:entityId`.

### AC-5 — Score display

Combined `score` is displayed as a relevance indicator. If `vectorScore > 0`, the result is semantically matched — show both scores on hover or in an expanded row. If `vectorScore === 0` for all results, show a muted note: "Semantic search returned no vector matches — showing lexical results only."

### AC-6 — Empty and error states

Empty state: "No results for [query]." when `results.length === 0`. Error state (503): "Search unavailable — Iranti instance unreachable. Check Health Dashboard." Error state (403): "Search requires a global-scope API key. This instance's key is namespace-scoped."

### AC-7 — Does not break existing Memory Explorer

Existing entity type filter, entityId filter, key filter, and pagination all continue to work correctly. Search mode and browse mode are clearly distinguished.

### AC-8 — TypeScript clean (both)

`tsc --noEmit` passes in `src/client` and `src/server`. No `any`.

## Implementation Notes

- The `diagnostics.ts` module already makes calls to `/kb/search` — use the same URL construction pattern for the proxy.
- The cross-repo audit confirms `lexicalWeight` and `vectorWeight` are tuning params — do not expose them in the UI at MVP. Let Iranti use its defaults.
- `minScore` can be offered as an advanced filter (default 0) but is not required for MVP.
- Do not add this endpoint to the existing entity-namespaced KB routes. It is a global search and should be a separate endpoint.
