## Problem Statement

The `staff_events` table does not exist in the live `iranti_local` database. This was discovered by the qa_engineer during the CP-T030 seed test (2026-03-20) and confirmed in CP-D001's additional findings section.

The Staff Activity Stream implementation (CP-T012 backend, CP-T014 frontend) polls a `staff_events` table for events. When the table does not exist, the polling adapter either:
- returns an empty stream (if it gracefully handles `table does not exist`)
- throws a PostgreSQL error and closes the SSE connection

In either case, no Staff events appear in the Activity Stream for any live Iranti instance.

The `staff_events` table was specified in CP-T001 (Staff Event Model spike, Phase 0). The spec defines the schema in `docs/specs/staff-event-model.md`, Section 3 (Persistence Strategy, Option 1). The migration to create this table has never been applied to the `iranti_local` database.

### In scope

2. If it does not exist: apply the migration from the spec to create the table and its indexes.

## Migration SQL

The following SQL creates the `staff_events` table as specified in `docs/specs/staff-event-model.md` Section 3:

```sql
CREATE TABLE IF NOT EXISTS staff_events (
  event_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  timestamp       TIMESTAMPTZ NOT NULL DEFAULT now(),
  staff_component TEXT NOT NULL,
  action_type     TEXT NOT NULL,
  agent_id        TEXT NOT NULL,
  source          TEXT NOT NULL,
  entity_type     TEXT,
  entity_id       TEXT,
  key             TEXT,
  reason          TEXT,
  level           TEXT NOT NULL,
  metadata        JSONB
);

CREATE INDEX IF NOT EXISTS idx_staff_events_timestamp  ON staff_events (timestamp DESC);
CREATE INDEX IF NOT EXISTS idx_staff_events_component  ON staff_events (staff_component);
CREATE INDEX IF NOT EXISTS idx_staff_events_agent      ON staff_events (agent_id);
CREATE INDEX IF NOT EXISTS idx_staff_events_entity     ON staff_events (entity_type, entity_id);
CREATE INDEX IF NOT EXISTS idx_staff_events_level      ON staff_events (level);
```

**Important:** Use `IF NOT EXISTS` on both `CREATE TABLE` and `CREATE INDEX` to make the migration idempotent. Safe to run more than once.

**Important:** The column names above use snake_case. This is intentional for `staff_events` — the control plane server queries this table with raw SQL that references snake_case names (as specified in the events.ts polling adapter). Do NOT convert to camelCase — this table is not a Prisma-managed table and does not follow Prisma naming conventions.

**Note for devops_engineer:** Verify that `events.ts` SQL references match these column names before running the migration. If `events.ts` uses camelCase, that is a separate bug (similar to CP-D001) and must be fixed at the same time.

   ```bash
   psql $DATABASE_URL -c "\d staff_events"
   ```
   Confirm: all columns present, all 5 indexes created.

4. **Test the events route:**
   With the control plane server running:
   ```bash
   curl -N http://localhost:3002/api/control-plane/events/stream
   ```
   Expected: SSE stream opens, receives heartbeat `: keep-alive` comments at ~15-second intervals. No error messages. No HTTP 500.

5. **Verify graceful empty state in the browser:**
   Open `http://localhost:5173/control-plane/stream`.
   Expected: "No events yet" empty state renders. SSE connection shows as active (green indicator if present). No error message in the console or UI.

## devops_engineer Findings (2026-03-20)

### Column name alignment — CONFIRMED CORRECT

`events.ts` SQL column references were verified against `create-staff-events.sql`. All column names match:

| events.ts reference | Migration column | Match? |
|---|---|---|
| `level` | `level` | ✓ |
| `staff_component` | `staff_component` | ✓ |
| `action_type` | `action_type` | ✓ |
| `agent_id` | `agent_id` | ✓ |
| `entity_type` | `entity_type` | ✓ |
| `entity_id` | `entity_id` | ✓ |
| `timestamp` | `timestamp` | ✓ |
| `event_id` | `event_id` | ✓ |

The serializer in `events.ts` uses dual-read patterns (`row.event_id ?? row.eventId`) as a defensive fallback, but the WHERE/ORDER BY clauses all use snake_case which is correct for this non-Prisma table.

### Migration applied — 2026-03-20

```
Applied to: postgresql://postgres:***@localhost:5432/iranti_local
Result: CREATE TABLE + 5 indexes (all IF NOT EXISTS — idempotent)
Verified schema: \d staff_events — all 12 columns present, all 5 indexes present
```

### Graceful degradation — CONFIRMED

`events.ts` implements a `staffEventsTableExists` cache with `assertStaffEventsTableExists()`. Before the migration was applied:
- `GET /events` returns HTTP 503 with `{ error: "staff_events table does not exist...", code: "EVENTS_TABLE_MISSING" }`
- `GET /events/stream` returns HTTP 503 JSON (not SSE) — no crash, no unhandled error

After the migration is applied with the table empty:
- `GET /events` returns `{ items: [], total: 0, limit: 100, offset: 0, oldestEventTimestamp: null }`
- `GET /events/stream` opens SSE, emits heartbeat `: keep-alive` every 15 seconds, never closes unless client disconnects

## Context: Why the Stream Is Empty Even After This Fix

After applying this migration, the `staff_events` table will exist but will still be empty. That is expected. The Staff Activity Stream will only show events when:

1. **Short-term:** The backend_developer adds a test event insertion via the seed test path (manual SQL INSERT for smoke testing)
2. **Phase 2 (CP-T025):** The upstream Iranti server is patched to write events to `staff_events` on each Staff component action

This ticket is purely prerequisite infrastructure — creating the table so the stream can function without errors, even if no events flow yet.

## Risks

- If `events.ts` references camelCase column names for `staff_events`, applying this snake_case migration will not fix the stream. The devops_engineer must verify SQL column name alignment in `events.ts` before and after the migration.
- If other Iranti instances share the same database, the `staff_events` table will receive events from all instances interleaved. An `instance_id` column is not in the current spec (flagged as Open Question 4 in the staff-event-model spec). This is acceptable for Phase 2 since multi-instance is Phase 3 scope.
- Running `CREATE TABLE IF NOT EXISTS` is safe and idempotent, but `CREATE INDEX IF NOT EXISTS` requires PostgreSQL 9.5+. The Iranti DB is assumed to meet this requirement.
