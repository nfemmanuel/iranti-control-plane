## Problem Statement

The control plane supports multiple Iranti instances. A user selects an instance context (in the Instance Manager or shell header selector) and expects that all data views — Memory Explorer, Archive Explorer, Staff Activity Stream, Instance Manager, Health — show data for that instance.

If instance context does not persist correctly across navigation, users will read data from the wrong instance and draw incorrect conclusions without knowing they are doing so. This failure mode is particularly damaging because it is invisible: there is no error, no warning — just silently wrong data.

**Trigger scenario:**
1. User selects Instance B in the Instance Manager.
2. Navigates to Memory Explorer. Data appears to be from Instance B (correct).
3. Navigates to Staff Activity Stream. Stream defaults to "all instances" or silently reverts to Instance A.
4. User watches events and draws conclusions about Instance B's behavior. All conclusions are wrong.

Priya (Technical Founder) is at highest risk: her Phase 1 evaluation specifically requires verifying multi-instance behavior. Dev (Power User) has 3 instances and navigates between them regularly.

The PRD (FR4) requires: "The control plane must show which runtime root, instance, and database are active, and let the user switch context safely." The control plane shell design specifies a persistent instance context selector in the header.

This ticket is primarily a QA verification task. If issues are found, frontend_developer is responsible for fixing them before v0.1.0 ships.

## Scope

**In:**

### QA Verification Test Protocol

Execute the following test protocol against a control plane connected to at least 2 Iranti instances. Document results for each step.

**Prerequisites:**
- Two local Iranti instances configured (Instance A and Instance B) with distinct runtime roots and databases.
- Each instance has at least some facts written to its knowledge base (so Memory Explorer is not empty for either).
- Both instances must be reachable (Running status).

**Test Steps:**

1. Open the control plane. Observe the default instance context in the shell header. Record which instance is shown.
2. Navigate to Memory Explorer. Confirm: data shown corresponds to the default instance. Record fact count or entity names to use as a comparison baseline.
3. Select Instance B from the instance context selector in the shell header.
4. Verify: the shell header updates to show Instance B.
5. Verify: Memory Explorer updates to show Instance B data (different from Instance A data confirmed in step 2).
6. Navigate to Archive Explorer. Verify: shows Instance B archive data (or empty state if no archive for Instance B — but must be Instance B's data, not Instance A's).
7. Navigate to Staff Activity Stream. Verify: stream is scoped to Instance B, or is clearly labeled "All instances" if global. Record which behavior occurs and whether it is labeled.
8. Navigate to Health. Verify: shows Instance B health data (database target, provider key status, etc. match Instance B configuration).
9. Navigate to Instance Manager. Verify: Instance B is visibly selected or highlighted as the active context.
10. Navigate back to Memory Explorer. Verify: still showing Instance B data (context did not reset on navigation).
11. Refresh the browser. Verify: instance context either persists (preferred) or clearly resets to default with a visible indicator.

**Pass Conditions (all must be true):**
- Steps 4-9: every data-bearing view shows Instance B data after selecting Instance B.
- Step 7: Staff Activity Stream is either correctly scoped to Instance B, or is explicitly labeled "All instances" — it is not silently showing Instance A data.
- Step 10: context does not reset when navigating between views.
- Step 11: if context resets on reload, a visible indicator confirms which instance is active after reload.

**Fail Conditions (any one causes a blocker):**
- Any data-bearing view silently shows Instance A data after Instance B has been selected.
- Staff Activity Stream shows events from the wrong instance without labeling.
- The instance context selector does not visually update to reflect the selection.
- Navigating between views causes context to reset silently.

### Shell Header Instance Selector Requirements (verify alongside test)

- The current active instance must be visible in the shell header at all times, on all views.
- The selector must show a human-readable identifier: at minimum the runtime root path, or a friendly name if one is configured.
- Changing the instance from the selector must immediately update all data-bearing views.
- If global views exist (e.g., "all instances" health summary), they must be explicitly labeled — not silently mixed with single-instance views.

**Out:**

- Implementing multi-instance view features not already in Phase 1 scope (e.g., side-by-side instance comparison — Phase 3).
- Building the instance context selector itself if it does not exist (that is frontend work from the Phase 1 shell implementation; this ticket verifies it works correctly, not builds it).
