## Problem Statement

The control plane shell form factor is recommended as a local web app but has no concrete design direction yet. Neither the navigation structure, visual identity, nor technology stack has been decided. Beginning Phase 1 implementation without these decisions will produce either a generic dashboard that does not reflect Iranti's character, or require costly rework mid-implementation when the team discovers the initial direction was wrong. The visual and structural foundation must be set in Phase 0 so that Phase 1 frontend work can proceed without design debt from day one.

## User Value

A deliberately designed shell means the product looks intentional and operators can orient quickly to the right surface. It establishes the navigation structure that all 5 Phase 1 views will slot into, and defines the visual system that makes the control plane feel like a coherent product rather than a collection of tables. It also unblocks all Phase 1 frontend work by giving frontend_developer a clear, PM-approved direction to build toward.

## Scope

In:
- **Shell layout wireframe**: Navigation structure (sidebar vs top nav vs hybrid vs contextual), primary section labels, entry points for all 7 Phase 1 surfaces (Memory Explorer — KB table, Memory Explorer — Archive, Entity Detail, Staff Activity Stream, Instance & Project Manager, Health & Diagnostics, and a logical home/overview). Wireframe may be text-based (structured description) or visual (embedded image or ASCII diagram) — both are acceptable.
- **Visual direction brief**: Typography recommendation (typeface character and rationale), color palette exploration with at least 2 distinct options each in light and dark mode variants. Palettes must not resemble generic gray/blue admin dashboards — they must have a defined character. Include a brief written rationale for each palette option explaining the intended mood and operator experience.
- **Technology recommendation**: Specific framework name (e.g., React, SvelteKit, Vue), bundler, stance on component library (adopt a specific one, build primitives, or avoid entirely) — with tradeoffs documented against these criteria: local-only serving simplicity, build complexity, developer velocity for a small team, long-term maintenance burden.
- Produce all outputs in a single spec document at `docs/specs/shell-design-exploration.md`

Out:
- Implementation of any shell, component, or visual element
- Component library build (that is Phase 1 work, after PM approves this exploration)
- Any backend work
- Final color values (this is a direction brief, not a finalized design system — precise tokens are Phase 1 work under CP-E006)

## Acceptance Criteria

- [ ] Shell layout wireframe covers all 7 Phase 1 surfaces (Memory Explorer — KB, Memory Explorer — Archive, Entity Detail, Staff Activity Stream, Instance & Project Manager, Health & Diagnostics, and overview/home) in a coherent navigation structure; the wireframe makes clear how a user moves between surfaces
- [ ] Visual direction brief includes at least 2 distinct color palette options, each with a defined light mode and dark mode variant; each palette includes a written rationale explaining the intended character; neither palette option resembles a generic gray/blue admin dashboard
- [ ] Technology recommendation is concrete: specific framework name is given (not "React or Vue"), bundler is named, stance on component library is explicit (not "we could consider…") — tradeoffs against local serving simplicity, build complexity, developer velocity, and maintenance burden are documented
- [ ] The spec explicitly addresses the SPA vs MPA routing question: is client-side routing recommended, server-side rendering, or a hybrid? The recommendation is stated with brief rationale.
- [ ] Output document exists at `docs/specs/shell-design-exploration.md` with all three sections: layout wireframe, visual direction brief, technology recommendation
- [ ] PM has reviewed and approved the visual direction before any Phase 1 implementation proceeds — this approval is a hard gate

## Risks

- Visual identity decisions are sticky: a wrong direction chosen in Phase 0 creates expensive Phase 1 rework because component foundations, color tokens, and layout patterns are all built on top of this choice. The PM review gate on visual direction is not optional.
- Technology choices — especially component libraries — can balloon scope if not deliberately constrained. The recommendation must be opinionated and justified, not a menu of options.
- If the Iranti brand has no existing visual reference, the frontend_developer is operating with creative latitude that must still be anchored to the product's character as described in the PRD. Over-indexing on personal preference rather than operator utility is a risk.

## Open Questions

- Should the shell be a single-page app with client-side routing, or a multi-page app with server-rendered pages? For a local-only tool with no SEO concern and a small number of surfaces, the answer is probably SPA — but the tradeoffs should be stated explicitly.
- Is there an existing Iranti design system, brand guide, or visual reference that should inform the control plane aesthetic? If yes, frontend_developer should consult it. If no, that is a relevant finding to surface to the PM.
- Should the control plane shell have a persistent chat panel from the start (anticipating Phase 2 CP-E007), or should the layout be designed to add it later without structural rework?
