# CP-T063 — API Key Scope Audit View

**Type:** Feature
**Priority:** P2 (Wave 8)
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 8)
**Status:** PM-ACCEPTED 2026-03-21 (scope fields gracefully null — Iranti v0.2.15 does not expose scope via API)
**Epic:** CP-E009 (Provider Manager)

## Summary

Iranti v0.2.1 introduced namespace-aware API key scopes. Each configured provider key can have a scope that restricts which agent namespaces or project namespaces it services. This capability is invisible in the control plane — the Provider Manager shows key presence and reachability, but not scope assignments. Operators configuring multi-agent or multi-project setups cannot verify from the control plane which key is scoped to which namespace without reading raw config files.

This ticket adds a scope column (or expandable row) to the Provider Manager view so operators can inspect key scopes at a glance and diagnose routing misconfigurations.

## Why This Is Wave 8

Provider scopes matter most when operators are running multi-agent setups — which becomes more common as the control plane matures and operators grow past single-agent use. Wave 7 (Metrics Dashboard) surfaces per-agent write volume, making multi-agent operational patterns more visible. Wave 8 should make the provider configuration that supports those agents auditable. This is not a write surface — scopes are set in config files and not editable from the control plane — but read-only visibility closes a real operator gap.

## Acceptance Criteria

### Backend

#### AC-1 — Provider scope in the providers endpoint

The existing `GET /api/control-plane/providers` endpoint (or `GET /api/control-plane/instances/:instanceId/providers`) should include a `scope` field per provider key. Expected response addition:

```json
{
  "providers": [
    {
      "name": "anthropic",
      "keyPresent": true,
      "reachable": true,
      "scope": "namespace:agent/*",
      "scopeType": "namespace" | "global" | "unknown"
    }
  ]
}
```

If the Iranti instance does not support scopes (pre-v0.2.1), return `scope: null` and `scopeType: "unknown"`. Do not 500.

The backend should proxy the scope field from whatever Iranti's provider metadata endpoint exposes. If Iranti does not yet expose scopes in a queryable endpoint (confirm before implementing), read from the provider config file directly via the same config-reading path already used by the setup wizard (CP-T023).

#### AC-2 — TypeScript clean

`tsc --noEmit` passes. No `any`.

### Frontend

#### AC-3 — Scope column in Provider Manager

In the Provider Manager list view (`/providers`), add a "Scope" column or badge to each provider row. Display:
- For `scopeType: "global"`: a muted "global" badge
- For `scopeType: "namespace"`: the scope string (e.g., `namespace:agent/*`) in a code-styled label
- For `scopeType: "unknown"` or `scope: null`: a muted "—" (no scope info available)

If the scope string is long, truncate with a tooltip showing the full string.

#### AC-4 — Scope in provider detail panel

In the Provider Manager's right-panel detail view for a selected provider, show the full scope information in a dedicated "API Key Scope" row, with a brief explanation: "Scopes restrict which agent or project namespaces this key services. A global scope (or no scope) means the key applies to all namespaces."

#### AC-5 — TypeScript clean

`tsc --noEmit` passes. No `any`.

## Implementation Notes

- The Provider Manager components are in `src/client/src/components/providers/ProviderManager.tsx`. The backend proxy is in `src/server/routes/control-plane/providers.ts`.
- Before implementing AC-1, confirm whether Iranti's provider config endpoint (`GET /kb/providers` or similar) returns scope information, or whether the setup wizard's config-file reader is the correct source. If uncertain, check `src/server/routes/control-plane/setup.ts` for the config reading pattern.
- Do not add a write surface (scope editing) in this ticket. This is read-only visibility only.
- If Iranti does not expose scopes at all in v0.2.14 (current), implement AC-1 with `scope: null` / `scopeType: "unknown"` as a graceful placeholder and note the upstream gap in the completion report.
