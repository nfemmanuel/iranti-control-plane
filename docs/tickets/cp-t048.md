# CP-T048 — Platform Installer Packages (MSI, .dmg, .deb)

**Type:** DevOps / Distribution
**Priority:** P2
**Phase:** 3
**Epic:** CP-E011 (Platform Distribution)
**Assigned:** devops_engineer
**Status:** Open
**Created:** 2026-03-20

---

## Summary

Produce signed, distributable installer packages for the Iranti Control Plane on Windows (`.msi` or NSIS `.exe`), macOS (`.dmg`), and Linux (`.deb`, `.rpm`, `.AppImage`). The packages bundle the Node.js runtime and the control plane Express server into a standalone executable, with the Vite-built React frontend served as static assets from that binary. The UI continues to open in the user's default browser — this is not an Electron app.

---

## Background and Prior Work

This ticket was anticipated in the Phase 0 installer concept evaluation (CP-T005) and in PRD Decision 7. The CP-T005 memo explicitly evaluated a native installer (Option A in its taxonomy) and concluded:

> "Too much distribution and maintenance overhead for Phase 2. Correct long-term direction for a productized v2, but not the right Phase 2 choice."

The PRD Decision 7 records: "Phase 3 may revisit a native installer or Homebrew formula." The Phase 3 roadmap candidate list includes: "Signed macOS installer or Homebrew formula (if Phase 2 wizard adoption validates demand)."

CP-T048 is the formal Phase 3 instantiation of that roadmap item, now expanded to cover all three primary platforms at the user's request.

**Prerequisite context:** CP-T023 (CLI setup wizard, Phase 2) must be delivered and adoption data collected before CP-T048 implementation begins. The wizard tells us whether users are successfully self-installing via npm — if they are, the installer value proposition is confirmed and this ticket proceeds. If they are not, the installer becomes even more critical. Either outcome validates moving forward.

## Scope Decision

**PM Recommendation: Option D — Platform installers wrapping a Node.js Single Executable (Option C base)**

The four options considered:

**Option A — Electron wrapper:** Wraps the web app in Electron, producing a proper desktop application with its own Chrome window. Highest user-experience quality. However, this is an architectural rewrite, not a packaging decision: Electron bundles Chromium (~200MB+), requires code signing and notarization on every platform, imposes a separate update mechanism, and forces the frontend and backend into an Electron IPC model that the current Express + Vite stack does not use. This is a candidate for a separate architectural ADR if the product direction calls for a desktop-native experience. It is not appropriate as the packaging implementation path for this ticket.

**Option B — Native daemon + system tray:** The Express server runs as a Windows Service / launchd daemon / systemd service, with a tray icon app that opens the browser to the control plane. Architecturally interesting. In practice, building a tray icon app requires Electron (full circle back to Option A overhead) or a native compiled binary per platform (Go or Rust, new language dependency). The operational model (background service, system tray) also changes the UX contract for the product significantly. Too much new scope.

**Option C — Standalone executable via Node.js bundling:** Bundle Node.js + the Express server into a single executable using Node.js Single Executable Applications (SEA, available in Node 20+) or `caxa`. The Vite build output (static HTML/CSS/JS) is embedded in the binary or placed alongside it in a known directory. The user launches one binary, the control plane starts, and the browser opens automatically. No npm, no `node_modules`, no PATH management needed. The UI still opens in a browser — not in a dedicated app window.

**Option D — Platform installers wrapping Option C:** Wrap the Option C binary in proper platform-specific installer UX: NSIS or WiX for Windows `.msi`/`.exe`, `create-dmg` for macOS `.dmg`, `fpm` for `.deb`/`.rpm`, and AppImage tooling for Linux universal. This is the user-requested deliverable and the correct implementation path. Users get a genuine install wizard, start menu / Applications folder / `apt install` integration, and an uninstaller. The control plane binary itself is Option C.

**Why not Electron-builder:** Despite its name, `electron-builder` can package non-Electron Node apps (using `--prepackaged`). It is an option for the toolchain, but its Node SEA support is evolving. Evaluate alongside `caxa` + per-platform packaging tools at implementation start.

**Recommendation: Option D.** This delivers the user-requested installer UX on all three platforms without rewriting the application architecture, without bundling Chromium, and without the full operational complexity of a system daemon or tray app. The browser-based UI model is preserved, consistent with the PRD's confirmed architecture choice.

## What Is Bundled

**Inside the installer:**
- Node.js runtime (bundled via Node SEA or `caxa` — no separate Node install required)
- Iranti Control Plane Express server (`src/server/` compiled TypeScript)
- Iranti Control Plane React frontend (Vite production build, static assets)
- A launcher script that starts the server and opens the browser to `http://localhost:PORT`
- A default configuration template (`.env.iranti.example`) placed in the install directory

**Explicitly NOT bundled:**
- Iranti itself (the Iranti HTTP server at `localhost:3001`)
- PostgreSQL
- pgvector
- Any LLM provider SDK or credentials

The installer assumes Iranti is pre-installed and running on the user's machine. The control plane is a management UI for an existing Iranti instance — it is not a standalone product that runs without Iranti.

If the user has not yet installed Iranti, the Getting Started screen (CP-T035) and the CLI setup wizard (CP-T023) are the correct onboarding paths. The installer packages the control plane UI only.

## Target Platforms

### Windows
- **Format:** `.msi` (preferred for enterprise environments) and/or NSIS `.exe` (easier to produce, widely accepted)
- **Toolchain:** NSIS or WiX Toolset for `.msi`; NSIS for `.exe`
- **Code signing:** Required. Extended Validation (EV) code signing certificate (~$300/year) is mandatory for Windows Defender trust on Windows 11. SmartScreen will block unsigned `.exe` installers with a "Windows protected your PC" dialog that many users treat as a hard stop.
- **Install location:** `%ProgramFiles%\Iranti Control Plane\`
- **Start menu entry:** Yes — "Iranti Control Plane"
- **Uninstaller:** Registered in Add/Remove Programs
- **Auto-launch option:** Checkbox in installer to add to Windows startup

### macOS
- **Format:** `.dmg` (drag-to-Applications disk image)
- **Toolchain:** `create-dmg` or `appdmg`
- **Code signing and notarization:** Required. Apple Developer ID Application certificate ($99/year program membership). Notarization via `xcrun notarytool` is mandatory — unnotarized `.dmg` files are quarantined by Gatekeeper on macOS 13+.
- **App bundle:** The binary must be wrapped in a `.app` bundle for proper macOS integration (dock icon, Spotlight indexing, proper permissions)
- **Install location:** `/Applications/Iranti Control Plane.app`
- **Homebrew Cask:** Evaluate as a distribution supplement (not a replacement). Requires publishing to Homebrew's third-party tap or the official cask registry. Adds discoverability for developer users who have Homebrew.

### Linux
- **Formats (in priority order):**
  1. `.AppImage` — universal, no install required, runs on any modern Linux distro. Best for broad compatibility.
  2. `.deb` — Debian/Ubuntu package. Highest user base overlap with the target persona (developers on Ubuntu/Debian).
  3. `.rpm` — Fedora/RHEL/openSUSE. Lower priority but achievable with the same `fpm` toolchain used for `.deb`.
- **Toolchain:** `fpm` (Effing Package Management) for `.deb`/`.rpm`; AppImage tooling (`appimagetool`) for `.AppImage`
- **Code signing:** Not required for `.deb`/`.rpm`/`.AppImage` in the same mandatory sense as macOS/Windows. GPG signing for the apt repo is best practice if a hosted apt repository is ever published.
- **Install location (deb/rpm):** `/usr/local/bin/iranti-control-plane` + `/usr/share/iranti-control-plane/`
- **Desktop entry:** `.desktop` file for application menu integration in GNOME/KDE

## Acceptance Criteria

1. **Windows installer (`.exe` NSIS):** A user on Windows 10 or 11 can download and run the installer, complete the wizard without errors, find "Iranti Control Plane" in the Start menu, launch it, and have the control plane open in their default browser at `http://localhost:PORT`. The installer must pass Windows Defender SmartScreen (requires valid code signature).

2. **Windows uninstaller:** The installer registers an uninstaller in Add/Remove Programs that cleanly removes all installed files without leaving orphaned directories.

3. **macOS `.dmg`:** A user on macOS 13+ (Ventura) or 14+ (Sonoma) can open the `.dmg`, drag the app to Applications, launch it from the Applications folder or Spotlight, and have the control plane open in Safari or Chrome without a Gatekeeper security warning. The `.dmg` must be notarized.

4. **Linux `.AppImage`:** A user on Ubuntu 22.04 LTS or later can download the `.AppImage`, mark it executable (`chmod +x`), and run it without installing any dependencies. The control plane opens in the default browser.

5. **Linux `.deb`:** A user on Ubuntu 22.04 LTS or later can install via `sudo dpkg -i iranti-control-plane_VERSION_amd64.deb` and launch via `iranti-control-plane` from the terminal or via the applications menu.

6. **Browser auto-open:** On all platforms, launching the control plane binary/app opens the default browser to the correct URL automatically. If the port is already in use, a clear error message is shown and an alternate port is offered.

7. **Version display:** The control plane UI reports the version number that matches the installer version in the About or Health view.

8. **No Node.js prerequisite:** A user with no Node.js installation can run the packaged installer and use the control plane. The installer must not require npm or Node.js to be on the user's PATH.

9. **CI build pipeline:** The installer artifacts are produced by a CI pipeline (GitHub Actions) on every tagged release. Each release produces: Windows `.exe`, macOS `.dmg`, Linux `.AppImage`, Linux `.deb`. Linux `.rpm` is a stretch goal for the initial release.

10. **Artifact size:** The packaged binary (before installer wrapper) must not exceed 80MB. This rules out bundling Chromium and provides a reasonable bound for the Node runtime + server bundle. The final installer wrapper may be slightly larger.

11. **QA sign-off:** Each platform installer has been manually tested on a clean machine (or clean VM) with no pre-existing Node.js, npm, or Iranti Control Plane installation. QA documents the test environment and result for each platform.

## Dependencies

### Must be complete before CP-T048 begins:

1. **CP-T023 (CLI Setup Wizard)** — Phase 2. The wizard must ship first so that adoption data can validate demand for native installers. The installer also benefits from the setup wizard's work on platform detection and `.env.iranti` path resolution, which can be reused.

2. **Phase 2 completion (v0.2.0 accepted by PM)** — The control plane feature set must be stable before investing in packaging infrastructure. Packaging a rapidly-changing application creates rework on every build pipeline change.

3. **Code signing certificates obtained** — Windows EV certificate and Apple Developer ID must be provisioned and stored in CI secret management before the build pipeline can produce signed artifacts. Lead time: 1–4 weeks for EV certificate issuance. This should be initiated at Phase 2 exit, not at CP-T048 task start.

4. **Vite build output is self-contained** — The `src/client` Vite production build must produce a static bundle with no dynamic import paths that assume a specific filesystem layout. Verify at CP-T048 kick-off.

5. **Server can be compiled to CommonJS or ESM-compatible Node SEA input** — The Express server uses `"type": "module"` (ESM). Node SEA and `caxa` compatibility with ESM entry points must be validated in a spike before full implementation.

### Should be resolved but not blocking:

- Homebrew tap decision (is Iranti publishing a third-party Homebrew tap? — affects macOS distribution strategy but does not block the `.dmg` artifact)
- Download hosting decision (GitHub Releases vs a dedicated CDN vs Homebrew Cask — affects distribution mechanism but not the build pipeline)

## Out of Scope

- **Bundling Iranti itself (the Iranti HTTP server, PostgreSQL, pgvector)** — the installer packages the control plane UI only. Iranti must be pre-installed.
- **Electron wrapper or desktop-native UI** — if the product direction moves to a true desktop app with its own Chrome window (no browser needed), that is a separate architectural decision and a separate ticket.
- **Auto-update mechanism** — updating the installed control plane is out of scope for the first release. Users can re-download and reinstall. Sparkle (macOS), WinSparkle (Windows), and AppImageUpdate (Linux) are candidates for a follow-on ticket.
- **Linux `.rpm` as a required deliverable** — it is a stretch goal. `.AppImage` is the primary Linux deliverable.
- **Homebrew Cask as a required deliverable** — it is a distribution supplement, not a required artifact. Evaluate at implementation time.
- **Windows `.msi` (WiX)** — the NSIS `.exe` is the primary Windows deliverable. WiX `.msi` may be added in a follow-on if enterprise customers require it.
- **32-bit architecture support** — x86 only (64-bit). ARM64 (Apple Silicon native, Windows ARM) is a stretch goal for the initial release.
- **Code notarization pipeline for Linux** — not applicable.
- **Silent/enterprise deployment** — MSI transform files, Windows Group Policy deployment, or macOS MDM profiles are out of scope.
- **Uninstalling Iranti itself** — the uninstaller removes only the control plane files. Iranti's own uninstall path is out of scope.

## Open Questions

**OQ-1: ESM + Node SEA compatibility**
The Express server uses `"type": "module"` and TypeScript ESM output. Node.js Single Executable Applications (Node 20+) have known friction with ESM entry points. The devops_engineer must validate this compatibility with a proof-of-concept spike before committing to the Node SEA approach. If SEA is not viable, `caxa` or a `pkg` fork are fallbacks.

**OQ-2: Code signing certificate provisioning**
Does the project have an Apple Developer account and a Windows EV code signing certificate, or does the user need to provision these? This is a blocker for signed installers on both platforms and should be resolved before CP-T048 is scheduled. Who owns the certificate lifecycle (renewal, storage in CI, revocation if needed)?

**OQ-3: Download hosting**
Where will the installer artifacts be published? GitHub Releases is the obvious answer and requires no additional infrastructure. However, if Homebrew Cask publication is desired, a stable, versioned URL scheme must be established before submission to the Homebrew registry.

**OQ-4: Port configuration**
The control plane currently serves on a hardcoded port. The installer must handle port conflicts gracefully. Should the installer allow the user to choose a port during setup, or should the binary detect and use the next available port automatically?

**OQ-5: Should the `.dmg` include an installer wizard or use the drag-to-Applications pattern?**
The standard macOS pattern for developer tools is a drag-to-Applications DMG. A guided `.pkg` installer (like Postgres.app or Docker Desktop) adds more hand-holding but is significantly more complex to build and notarize. The recommendation is drag-to-Applications for the first release, but this is a UX decision the PM should confirm.

**OQ-6: Apple Silicon (arm64) support**
The Node.js binary bundled in the installer is architecture-specific. Should the initial release target x86_64 only, produce separate arm64 builds, or produce a universal binary (fat binary)? A universal binary is the cleanest user experience but requires a GitHub Actions runner with arm64 build capability (available with `macos-latest` on GitHub Actions).

## Implementation Notes for devops_engineer

When this ticket is picked up, the following toolchain investigation should be done before writing any pipeline code:

1. **Spike: Node SEA with ESM** — Validate that the server's ESM TypeScript output can be bundled with Node 20 SEA. Expected issues: `--input-type=module` handling, dynamic `import()` of TypeScript paths, and Prisma client compatibility. Document findings.

2. **Spike: Static frontend embedding** — Determine how the Vite build output is served from the standalone binary. Options: (a) embed assets into the binary via a virtual filesystem (Node SEA supports this via the "assets" field in the SEA config), or (b) place assets alongside the binary in a known relative path. Option (a) is cleaner for a single-file distribution but has size implications.

3. **Toolchain selection** — Choose between Node SEA + per-platform packaging tools vs `caxa` (simpler, less control) vs `electron-builder` with `--dir` mode. Document tradeoffs and confirm with PM before full implementation.

4. **CI pipeline design** — The build pipeline should use matrix strategy across platforms. Windows builds require a `windows-latest` runner. macOS notarization requires a `macos-latest` runner with Xcode command-line tools. Linux builds can run on `ubuntu-latest`. Signing secrets must be stored as GitHub Actions encrypted secrets.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Node SEA incompatibility with ESM server | Medium | High | Run spike early; `caxa` is a fallback |
| Apple notarization process change | Low | High | Follow Apple's current `notarytool` docs; avoid deprecated `altool` path |
| EV certificate provisioning delay | Medium | Medium | Start provisioning at Phase 2 exit, not at CP-T048 start |
| Installer size exceeds 80MB target | Low | Medium | Profile bundle; defer Prisma client inclusion if necessary |
| Windows Defender false positive on NSIS binary | Low | High | Use signed certificate; submit to Microsoft Security Intelligence if false positive occurs |
| Port conflict on default port | Medium | Low | Implement port-conflict detection and auto-increment before packaging |
