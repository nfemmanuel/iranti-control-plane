## Problem Statement

Navigating between views, finding a specific entity, and triggering common actions in the control plane all require multiple mouse clicks. For operators who work in the control plane frequently — reviewing entities, watching Staff activity, checking health — the click-heavy navigation pattern creates unnecessary friction. The Memory Explorer in particular, with its entity list and temporal history drill-down, rewards keyboard navigation: an operator who can type an entity ID and jump directly to its detail page is operating materially faster than one who must filter, scroll, and click.

PRD ER3 (Minimal Context Switching) requires reducing inter-view friction. The command palette is the single highest-leverage keyboard affordance the control plane can add: one trigger (Cmd+K) that enables every navigation, entity search, and action from anywhere in the app.

## User Value

An operator who knows the entity they want to inspect can press Cmd+K, type a few characters of the entityId, and be on the entity detail page in under 3 seconds — without touching the mouse. An operator who wants to trigger a health refresh, clear filters, or jump to the Activity Stream can do so from anywhere in the app without navigating to the right view first. Power operators who live in the control plane will measure their productivity by how rarely they need the mouse.

## Scope

**In:**

**Core Command Palette:**
- Global trigger: `Cmd+K` (macOS) and `Ctrl+K` (Windows/Linux). Works from any focused element in the app, including table rows, text inputs, and filter bars. Call `event.preventDefault()` to prevent browser defaults (browser address bar focus on Ctrl+K in some browsers).
- Rendered as a modal dialog at the root level (not inside any scrolling container). Uses a React portal rendered to `document.body`. Overlay: semi-transparent dark scrim.
- Opens in under 50ms from keypress to visible (measured on a 2021 M1 MacBook Pro). Must not block the main thread during open — any data fetching is deferred until after the palette is visible.
- Text input auto-focused on open. Placeholder: "Search entities, navigate, or run a command...".
- Input is not cleared when the palette is closed and reopened — it retains its last query value. (This matches Linear's behavior and is faster for operators who re-open after mistyping.)
- Escape closes the palette without any action. Focus returns to the element that had focus before the palette opened.

**Result Sections (in display order):**
1. **Recent** (max 5 entries): Recently visited entity detail pages, persisted in `localStorage` as an array of `{ entityType, entityId, visitedAt }`. Shown when the input is empty or has < 2 characters. Each entry shows: entity icon, `entityType/entityId`, and the last visit timestamp ("2 hours ago"). Clicking navigates to the entity detail page.
2. **Navigate** (max 6 entries, static): All Phase 1 navigation destinations. Shown when input is empty or input matches the view name. Entries: Overview, Memory Explorer, Archive, Activity Stream, Instances, Health. Each shows a nav icon and keyboard shortcut (e.g., "G then M"). Clicking navigates to the view.
3. **Actions** (filtered by input): Static list of operator actions:
   - "Refresh health" — triggers the health page's on-demand refresh (dispatches an event; the health view re-fetches). Keyboard hint: no shortcut.
   - "Clear all filters" — clears filters on the currently active table view. Only shown when a filterable view is active.
   - "Toggle dark mode" — toggles the `[data-theme]` attribute.
   - "Open chat panel" — opens the Phase 2 chat panel (shown but disabled in Phase 2 if CP-T020 is not yet complete; shown and functional once CP-T020 is done).
4. **Search results** (max 10 entries, API-backed): Entity search results from `GET /api/control-plane/memory?search=[query]&limit=10`. Shown when input has >= 2 characters. Results update on each keystroke (debounced at 150ms). Each result shows: entity icon, `entityType/entityId`, the matching `key` or `summary` excerpt, and confidence as a muted badge. Clicking navigates to the entity detail page. Shows a loading spinner in the section header while the API call is in flight.

**Keyboard Navigation:**
- `↑` / `↓` arrow keys: navigate between results. Selection wraps (pressing ↓ on the last result moves to the first).
- `Enter`: activate the currently selected result (navigate or execute action).
- `Tab`: moves focus to the next interactive element inside the palette (input → results → close button). Does not close the palette.
- Escape: closes the palette.
- Mouse hover changes the selected result (hover state and keyboard selection state are unified — there is only one "active" result).

**Recent Entity Tracking:**
- Whenever the user navigates to an entity detail page (Memory Explorer → entity), write `{ entityType, entityId, visitedAt: Date.now() }` to the `localStorage` key `iranti_cp_recent_entities`.
- Maximum 10 entries in the array. When the 11th is added, remove the oldest.
- No deduplication within the same day — if the user visits the same entity 3 times, the most recent visit is all that matters. Deduplicate by `entityType + entityId` (keep only the most recent entry per entity).
- The top 5 by `visitedAt` (most recent first) are shown in the Recent section.

**Empty State:**
- Input has >= 2 characters but no search results: show "No entities found for '[query]'" in the Search section.
- Input is empty and Recent section has 0 entries: show "No recently visited entities. Start by browsing the Memory Explorer."

**Out:**
- Fuzzy search across fact values (not entityId) — too expensive for Phase 2 without dedicated search indexing; Phase 3.
- Command palette customization or pinning (Phase 3).
- Two-key navigation shortcuts (G then M, G then A, etc.) — documented in best practices as Phase 2, but implementation is a separate micro-task bundled here as a stretch goal. Implement if time allows after the core palette is done.
- Keyboard shortcut reference sheet (Phase 3 — can be linked from the palette's footer as a "?" icon pointing to a docs page).

## Dependencies

- Phase 1 views complete: the navigation commands and "active view" context (for "Clear all filters" visibility) require all Phase 1 views to be implemented.
- CP-T020 (Embedded Chat Panel): the "Open chat panel" action in the palette is shown as disabled until CP-T020 is complete. The implementation must check a feature flag or the existence of the chat panel DOM element.
- `GET /api/control-plane/memory?search=` endpoint: an existing Phase 1 API endpoint (from CP-T011 or CP-T013 backend work) that supports full-text or keyword search across `entityId` and `key`. If this endpoint does not exist or does not support `search` parameter, the Search section of the palette must be built with a filter on the existing list endpoint as a fallback. Confirm at task start.
- `cmdk` npm package: the command palette implementation library. Install and import as `import { Command } from 'cmdk'`. This is a headless library — all styling uses the Iranti token system.

## Acceptance Criteria

- [ ] Cmd+K opens the palette from any location in the app, including while focused on a table row, a filter input, or the Activity Stream. The palette appears in under 50ms from keypress.
- [ ] Entity search (>= 2 characters in input) queries the API and shows results within 300ms on a local instance with a warm server. Results are fuzzy-matched against `entityId` at minimum. The search is debounced at 150ms (the API is not called on every keystroke).
- [ ] Top 5 recently visited entities are shown in the Recent section when the input is empty or has < 2 characters. Recency is determined by `visitedAt` in `localStorage`, persisted across reloads. Visiting an entity detail page updates the Recent list immediately.
- [ ] All 6 Phase 1 navigation destinations appear in the Navigate section. Clicking any of them navigates to the correct view and closes the palette.
- [ ] Escape closes the palette. Focus returns to the element that had focus before the palette opened (confirmed with keyboard-only navigation test: open palette from a table row, press Escape, confirm the table row has focus again).
- [ ] Keyboard navigation (↑/↓/Enter) works for all result types. Arrow keys do not scroll the page behind the overlay while the palette is open.
- [ ] "Refresh health" action in the Actions section: if the Health view is active, clicking triggers a re-fetch. If the Health view is not active, the action still works (navigates to Health and triggers a refresh, or triggers a background refresh — implementation decision).
- [ ] "Clear all filters" action: only shown when a filterable view (Memory Explorer, Archive Explorer) is active. Clicking clears all active filters on that view.
- [ ] "Toggle dark mode" action works from anywhere and toggles the `[data-theme]` attribute immediately.
- [ ] `localStorage` key `iranti_cp_recent_entities` is written on entity detail page visits. Maximum 10 entries, deduplicated by entity identity, most recent per entity kept.
- [ ] TypeScript compiles without errors. No `any` types in the command palette component tree.

## Risks

- **Cmd+K conflict with existing browser behavior.** In some browsers and OS configurations, Cmd+K in a browser page opens the address bar or triggers another shortcut. Call `event.preventDefault()` on the keydown event when the palette trigger fires. Test on Chrome, Safari, and Firefox on macOS; Chrome and Edge on Windows.
- **API search endpoint may not exist yet.** If the Phase 1 backend does not implement a `?search=` parameter on the knowledge_base endpoint, the Search section falls back to a client-side filter on whatever data is already loaded in the table. This is acceptable for a Phase 2 launch (make it clearly a fallback, not the intended architecture) but must be documented as a blocker for full search capability.
- **Focus management correctness.** Returning focus to the pre-palette element requires tracking it before the palette opens (`const previousFocus = document.activeElement`). This is easy to forget and hard to test manually. Write a focused unit test for this behavior.
- **Performance on slow machines.** If the entity search API is slow (> 500ms) on the local Iranti instance, the palette will feel laggy. Add a loading spinner in the Search section header and set a timeout: if no results arrive within 1000ms, show "Searching..." with a spinner rather than a blank section.

## Open Questions

- Should the palette include a "recent commands" section (the last 3 actions the user ran, e.g., "Clear filters", "Refresh health")? Recommendation: no for Phase 2 — adds complexity without high return; add in Phase 3 if requested.
- Should entity search also search `summary` field text (full-text search across fact summaries)? Recommendation: if the API supports it, yes — add `summary` to the search fields. If it requires a new API change, scope to `entityId` only for Phase 2.
- Should the palette's input clear on close, or retain its value? Final decision above is retain (matches Linear). Confirm this is consistent with team preference.
