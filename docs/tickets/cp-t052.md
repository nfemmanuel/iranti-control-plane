# CP-T052 — Health View: Decay Config + Vector Backend + Attend Status

**Type:** Feature
**Priority:** P2
**Assigned:** frontend_developer + backend_developer
**Phase:** 3 (Wave 4)
**Status:** PM-ACCEPTED 2026-03-21 — All 5 decay env vars with correct defaults, pgvector no-probe / qdrant+chroma 3s probe / missing URL = warn not crash, Attendant informational card with upstreamPRRequired, MemoryDecayCard amber-for-enabled with explanatory text, VectorBackendCard with hybrid fallback note, AttendantStatusCard in INFO tier, vectorBackend status does not affect overall health indicator.
**Epic:** CP-E005 (Health & Diagnostics extension)

## Summary

Three operator-critical signals are missing from the Health Dashboard:

1. **Memory decay configuration** — `IRANTI_DECAY_ENABLED`, `IRANTI_DECAY_STABILITY_BASE`, `IRANTI_DECAY_THRESHOLD` control the Archivist's decay behavior. Operators cannot currently tell whether decay is enabled, what the threshold is, or why facts are being archived.

2. **Vector backend status** — Iranti supports three vector backends (`pgvector`, `qdrant`, `chroma`). The Health view checks DB reachability but does not report which backend is active or whether it's reachable. This is the primary diagnostic surface for B4-style search failures.

3. **Attendant health signal** — `iranti_attend` injection classifier returns `classification_parse_failed_default_false` in known failure cases (B11 benchmark finding). Until CP-T025 is merged, attend failures are invisible. This ticket adds a best-effort Attendant health signal to the control plane Health view.

## Acceptance Criteria

### Backend (backend_developer)

#### AC-1 — Expose decay config in health endpoint
Extend `GET /api/control-plane/health` (or add a sub-endpoint) to include decay configuration read from the control plane's own `.env.iranti` (or the connected Iranti instance's env):
```json
{
  "decay": {
    "enabled": false,
    "stabilityBase": 30,
    "stabilityIncrement": 5,
    "stabilityMax": 365,
    "decayThreshold": 10
  }
}
```
Read from the environment (check `IRANTI_DECAY_ENABLED`, etc.) — these are the Iranti process's env vars, so read them from the same env file the control plane reads for DB and provider config.

#### AC-2 — Expose vector backend in health endpoint
Extend the health endpoint to include:
```json
{
  "vectorBackend": {
    "type": "pgvector" | "qdrant" | "chroma" | "unknown",
    "configured": true,
    "url": null | "http://localhost:6333"
  }
}
```
Read `IRANTI_VECTOR_BACKEND` from env. For qdrant/chroma, include `IRANTI_QDRANT_URL` or `IRANTI_CHROMA_URL`. For pgvector, `url` is null (uses the same DB).

#### AC-3 — Vector backend reachability check (pgvector only)
For `pgvector`, the existing DB reachability check serves double duty. No additional network probe needed.
For `qdrant` and `chroma`, attempt a lightweight HTTP probe to the configured URL. Report status as `ok`, `warn`, or `error` in the health response.
**Important:** Only probe if the URL env var is present and non-empty. If `IRANTI_QDRANT_URL` / `IRANTI_CHROMA_URL` is missing and the backend is set to qdrant/chroma, report status as `warn` with message "URL not configured" — do not attempt a probe against an empty URL or crash.

#### AC-4 — Attendant health signal
The Iranti health endpoint (`GET /health`) returns `{ status, version, provider }` only — no Attendant stats. This is a limitation. For this ticket: add an informational Attendant health card to the control plane health endpoint that reports:
```json
{
  "attendant": {
    "status": "informational",
    "message": "Attendant status cannot be verified without native emitter injection (CP-T025). If memory injection appears to not be working, ensure entityHints are provided to iranti_observe — entity auto-detection from raw text is unreliable.",
    "upstreamPRRequired": "CP-T025"
  }
}
```
This documents the known limitation rather than silently omitting the check.

### Frontend (frontend_developer)

#### AC-5 — Decay Configuration card in Health Dashboard
Add a "Memory Decay" card below the existing health checks. Show:
- **Enabled:** Yes / No (with status indicator — green if disabled, amber if enabled so operators know decay is active)
  - Note: green=disabled is intentional — decay being active is a notable operator state worth flagging with amber. Add a tooltip or label text (e.g. "Decay is active — facts below the stability threshold will be archived automatically") so operators are not confused by the color direction.
- **Decay threshold:** `N confidence` (below this, facts are archived)
- **Stability range:** `N–N days` (base to max)
- If decay is disabled, show: "Memory decay is disabled. Facts are archived only by expiry, low confidence (< 30), or Resolutionist resolution."
- If `IRANTI_DECAY_ENABLED` is not set, default to disabled and degrade gracefully — do not error.

#### AC-6 — Vector Backend card in Health Dashboard
Add a "Vector Backend" card:
- Backend type (pgvector / qdrant / chroma / unknown)
- Status indicator (ok / warn / error)
- For qdrant/chroma: configured URL and reachability status
- For pgvector: "Uses primary database connection"
- If backend is unknown: "IRANTI_VECTOR_BACKEND not configured — defaulting to pgvector"
- **Additional context (Iranti v0.2.13+):** Iranti v0.2.13 added a hybrid search fallback to deterministic in-process semantic scoring when pgvector is unavailable. The card should note this if applicable: "pgvector unavailable — Iranti is falling back to in-process semantic scoring (v0.2.13+). Search quality may be reduced." This is informational severity, not a warning.

#### AC-7 — Attendant Status card in Health Dashboard
Add an "Attendant" informational card:
- Status: Informational (use the same grey/informational style as other informational checks)
- Message surfacing the known limitation and the entityHints workaround
- Link to docs or CP-T025 context (can be omitted if no docs page exists yet)

#### AC-8 — Severity taxonomy consistency
All three new cards must use the existing four-tier severity taxonomy (Critical / Warning / Informational / Healthy) from CP-T028.
