# CP-T072 — Runtime Lifecycle Dashboard

**Type:** Feature
**Priority:** P2
**Assigned:** backend_developer + frontend_developer
**Phase:** 5 (Wave 11)
**Status:** OPEN
**Epic:** CP-E016 (Session Recovery & Runtime Lifecycle)
**Depends on:** CP-T068 (overview dashboard pattern)
**Upstream capability:** Iranti v0.2.16 runtime lifecycle tracking (`runtimeLifecycle.ts`, `/health` runtime field)

---

## Description

Iranti v0.2.16 added runtime lifecycle tracking for instance-backed API servers. Each running instance now writes a `runtime.json` file to its instance directory on startup, refreshes `lastHeartbeatAt` every 15 seconds, and marks `status: 'stopped'` on shutdown.

The `GET /health` endpoint now includes a `runtime` field with this data:

```json
{
  "status": "ok",
  "version": "0.2.16",
  "runtime": {
    "instanceName": "my-instance",
    "pid": 12345,
    "ppid": 1234,
    "port": 3001,
    "startedAt": "2026-03-21T10:00:00.000Z",
    "lastHeartbeatAt": "2026-03-21T10:30:00.000Z",
    "updatedAt": "2026-03-21T10:30:00.000Z",
    "status": "running",
    "healthUrl": "http://localhost:3001/health",
    "version": "0.2.16"
  }
}
```

The `runtime` field is `null` for instances not launched via `iranti run` (ad-hoc or dev-mode instances).

**The current control plane gap:** The Instance Manager shows instances by reading their `.env` files and pinging the health endpoint. It shows whether an instance is "Reachable" or "Unreachable." It does not show:
- Whether the instance's process is actually running (vs the port just happening to respond)
- The PID, uptime, or start time
- Whether the instance is stale — `runtime.status = 'running'` but `lastHeartbeatAt` is stale by >60s (process crashed without shutdown signal)
- The Iranti version the instance is running

This ticket adds **runtime lifecycle status** to the Instance Manager — transforming it from a connectivity probe into a genuine process-awareness surface.

## What to Build

### Backend — Extract Runtime from Health Response

Modify the control plane's health proxy to extract and forward the `runtime` field from the Iranti health response.

**Current shape (`GET /api/control-plane/health`):**
The health endpoint already proxies Iranti's health response and adds control plane checks. The `runtime` field from Iranti is currently passed through but not typed or surfaced.

**Changes needed:**
1. Add `runtime: InstanceRuntimeInfo | null` to the `HealthResponse` type
2. Parse the `runtime` field from the Iranti health response and include it in the control plane's health response
3. Compute `runtimeStatus` — a derived field:
   - `'running'` — `runtime.status === 'running'` AND `lastHeartbeatAt` is within 60 seconds
   - `'stale'` — `runtime.status === 'running'` but `lastHeartbeatAt` is older than 60 seconds (process may have crashed without clean shutdown)
   - `'stopped'` — `runtime.status === 'stopped'` or `'stopping'`
   - `'no-runtime'` — `runtime` is null (ad-hoc or dev-mode instance)

```typescript
interface InstanceRuntimeInfo {
  instanceName: string;
  pid: number;
  port: number;
  startedAt: string;
  lastHeartbeatAt: string;
  status: 'starting' | 'running' | 'stopping' | 'stopped';
  runtimeStatus: 'running' | 'stale' | 'stopped' | 'no-runtime';  // derived
  version: string;
  healthUrl: string | null;
}
```

The `runtimeStatus` derivation logic belongs in the backend (the frontend should not need to compute staleness from timestamps).

### Frontend — Runtime Status in Instance Manager

The Instance Manager (`/instances`) currently shows each instance as a card with name, port, reachability status, and project bindings.

**Add to each instance card:**

```
┌─────────────────────────────────────────────────────┐
│  my-instance                           [RUNNING ●]  │
│  port: 3001 · v0.2.16                              │
│  PID: 12345 · started 3h ago                       │
│  last heartbeat: 30s ago                           │
│  Project: iranti-control-plane                     │
└─────────────────────────────────────────────────────┘
```

**Runtime status badge (top-right of each card):**

| `runtimeStatus` | Badge | Color |
|----------------|-------|-------|
| `running` | `RUNNING ●` | emerald |
| `stale` | `STALE ⚠` | amber |
| `stopped` | `STOPPED ○` | muted gray |
| `no-runtime` | (no badge — show existing reachability indicator only) | — |

**Additional fields shown when runtime data is available:**
- Iranti version (e.g., `v0.2.16`)
- PID (shown as `PID: 12345` in small muted text)
- Started relative time (e.g., `started 3h ago`)
- Last heartbeat relative time (e.g., `last heartbeat 30s ago`)

**Stale warning:** When `runtimeStatus === 'stale'`, show an amber callout inside the instance card: "This instance's runtime heartbeat is stale. The process may have crashed. Try running `iranti run --instance <name>` to restart."

**No-runtime note:** When `runtime` is null (ad-hoc mode), show a muted note: "Runtime metadata unavailable — instance started without `iranti run`."

**Health check interaction:** The existing reachability check (`Reachable` / `Unreachable`) remains. When `runtimeStatus === 'stale'` but the health endpoint is still reachable, show both: "Reachable but stale runtime signal."

### Overview Dashboard Integration

The Home Overview Dashboard (CP-T068) currently shows active agents but not instance runtime state. Add a brief runtime summary to the existing `GET /api/control-plane/overview` response:

- Add `instanceRuntime: { running: number; stale: number; stopped: number; noRuntime: number }` to the overview response
- Show a "Instances" summary pill in the System Status Strip if any instance is stale (amber dot) or stopped (rose dot when expected running)

This is a small addition — the Instance Manager card in the overview is not required, just the status strip contribution.

## Acceptance Criteria

### AC-1 — Runtime field in health response

`GET /api/control-plane/health` includes a `runtime` field when the proxied Iranti instance provides runtime data. `runtimeStatus` (derived) is included. TypeScript clean, no `any`.

### AC-2 — Running instance shows correct badge

An instance launched via `iranti run` with a heartbeat within 60 seconds shows `RUNNING ●` (emerald) in the Instance Manager.

### AC-3 — Stale instance detected and shown

An instance with `runtime.status = 'running'` but `lastHeartbeatAt` older than 60 seconds shows `STALE ⚠` (amber) and the stale warning callout with the restart hint command.

### AC-4 — Stopped instance shown correctly

An instance that shut down cleanly (`runtime.status = 'stopped'`) shows `STOPPED ○` (muted gray).

### AC-5 — Ad-hoc instance handled gracefully

An instance where `runtime` is null shows no runtime badge and the "Runtime metadata unavailable" muted note.

### AC-6 — Runtime detail fields shown

PID, Iranti version, started relative time, and last heartbeat relative time are shown on instance cards that have runtime data.

### AC-7 — Overview status strip updated

If any instance is stale, the overview status strip includes an amber dot for instance health. If all instances are running, the dot is emerald (or no dot if no instances are configured).

### AC-8 — No regressions in Instance Manager

All existing Instance Manager functionality (reachability, project bindings, env file details, repair actions) continues to work. Runtime data is additive.

### AC-9 — tsc clean

`tsc --noEmit` passes in both `src/server` and `src/client`. No `any`.
