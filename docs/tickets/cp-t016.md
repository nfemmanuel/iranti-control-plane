## Problem Statement

No unified diagnostics surface exists. Users must run `iranti doctor` in the terminal, inspect `.env.iranti` manually, and check individual integration files to understand the health of their Iranti installation. The PRD specifies Health & Diagnostics as the screen new users should see after install (FR4, ER2, ER5). Building it in Phase 1 closes the operability gap for new and existing users equally.

## User Value

Operators see all health checks in one place, understand exactly what is wrong, and receive actionable remediation guidance — without running terminal commands or consulting documentation. This is the "fast time to root cause" surface (ER2).

## Scope

**In:**
- Health & Diagnostics view (`/health` route)
- Overall status badge: healthy / degraded / error — always visible at the top
- Per-check status cards for all 10 checks from CP-T002 Group 6: `db_reachability`, `db_schema_version`, `vector_backend`, `anthropic_key`, `openai_key`, `default_provider_configured`, `mcp_integration`, `claude_md_integration`, `runtime_version`, `staff_events_table`
- Status card layout: check name, status icon (ok/warn/error), short message, detail fields, remediation guidance text
- Static remediation guidance text per failing check (not dynamic LLM-generated — hardcoded strings per check type)
- Auto-refresh on 30-second interval
- Manual refresh button
- `checkedAt` timestamp displayed
- Error state for health endpoint itself failing (503)

**Out:**
- Auto-remediation actions (Phase 2 — no "Fix it for me" buttons in Phase 1)
- Provider credit / quota visibility (Phase 2 per FR8 PM decision)
- Instance-level health drill-down (Phase 1 shows health for the current active instance only)

## Acceptance Criteria

- [ ] All 10 health checks from `/api/control-plane/health` rendered as individual status cards.
- [ ] Each card shows: check name (human-readable label), status icon (green check / yellow warning / red error per Option B status tokens), message string from API response, detail fields (if detail object present in response).
- [ ] Overall status badge (healthy/degraded/error) is always visible at the top of the page and uses the correct Option B status color tokens.
- [ ] Each failing check (warn or error status) shows a static human-readable remediation guidance text. Guidance must be concrete and actionable, not generic. Examples:
  - `anthropic_key` warn → "Add ANTHROPIC_API_KEY to your .env.iranti file and restart Iranti."
  - `staff_events_table` warn → "Run the CP-T001 database migration to enable the Staff Activity Stream."
  - `db_reachability` error → "Iranti cannot connect to PostgreSQL. Ensure PostgreSQL is running and DATABASE_URL in .env.iranti is correct."
  - `vector_backend` warn → "pgvector extension is not installed. Run: CREATE EXTENSION IF NOT EXISTS vector; in your PostgreSQL database."
- [ ] Auto-refresh runs on a 30-second interval. A subtle "refreshing..." indicator shown during each refresh. The overall status badge updates if health state changes.
- [ ] Manual refresh button triggers an immediate re-fetch outside of the 30-second cycle.
- [ ] `checkedAt` timestamp shown ("Last checked: 12 seconds ago" format).
- [ ] Error state: if `/api/control-plane/health` returns 503, show a "Diagnostics unavailable" state with a retry button.
- [ ] Both light and dark mode render correctly per Option B tokens.

## Risks

- Static remediation text must be maintained as health check definitions evolve — create a separate constants file for remediation strings so they are easy to update without touching component logic.
- The `staff_events_table` check will likely warn on most Phase 1 installs (migration not yet applied) — ensure this renders as a warn, not as a critical error, and that the guidance text is clear about what to do.
- 30-second auto-refresh must not cause visible flicker or layout shift. Use React Query's background refetch with stale-while-revalidate pattern.

## Open Questions

- Should the Health view be shown as the default overview/home screen (`/` route) for new installs? Per PRD: "This should be the first screen new users see after install." Recommend: `/` (Overview) shows a health summary widget, full health detail is at `/health`. Frontend to implement this routing per CP-T017 shell structure.
- Should failing checks be sorted to the top of the cards list? Yes — sort by status: error first, warn second, ok last within each group.
