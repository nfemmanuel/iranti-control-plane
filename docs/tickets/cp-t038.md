## Problem Statement

Entity aliases — the ability to define that two different entity identifiers refer to the same logical entity — were specified in Phase 0 (CP-T006 spike), found feasible, and intentionally deferred to Phase 2 (PM Decision 2: "entity aliases: deferred to Phase 2 with full spec already written"). Phase 2 tickets were then written without including this item. This is a PRD gap flagged in the PM audit of 2026-03-20.

The PRD's Memory Explorer section explicitly lists `entity_aliases` as a required data source:
> "Core views: current facts table for `knowledge_base`, archive table for `archive`, entity detail page, temporal history timeline per `entity/key`, relationship view for `entity_relationships`, **raw JSON inspector for `valueRaw`, `properties`, and `conflictLog`**"

And in the Data Sources section:
> "- `entity_aliases`"

Users with multiple agent identities for the same logical operator (Dev's 3 instances, Priya's team members) cannot see a unified view of a logical entity across different `agentId` values without aliases.

---

## Scope

**In:**

### Backend
- Confirm whether `entity_aliases` table exists in the current Iranti schema — if not, design and add it
- `GET /api/control-plane/aliases` — list all aliases
- `GET /api/control-plane/aliases/:entityType/:entityId` — get all aliases for an entity
- `POST /api/control-plane/aliases` — create a new alias (routes through safe write pathway, not direct DB write)
- `DELETE /api/control-plane/aliases/:aliasId` — remove an alias

### Frontend
- Entity detail view (CP-T036, already shipped) — add alias management section to the entity detail page
- Show existing aliases for the entity with the ability to add/remove
- Memory Explorer — when searching by entityId, include results from aliased entities
- Navigation links from one alias to the canonical entity

**Out:**
- Automatic alias detection from write history (Phase 3 enhancement)
- Cross-alias aggregated fact views (Phase 3)
