# CP-T073 — Iranti Upgrade Coordination

**Type:** Feature
**Priority:** P2
**Assigned:** backend_developer + frontend_developer
**Phase:** 5 (Wave 12)
**Status:** OPEN
**Epic:** CP-E016 (Session Recovery & Runtime Lifecycle)
**Depends on:** CP-T072 (runtime lifecycle surface — Instance Manager must show running vs stale before upgrade actions make sense)
**Upstream capability:** Iranti v0.2.16 `iranti upgrade --restart --instance <name>`

---

## Description

Iranti v0.2.16 added upgrade coordination: `iranti upgrade --restart --instance <name>` installs a new version and coordinates a restart for a named running instance. This is new — prior versions required stopping the instance manually, running npm install, and restarting.

**The current control plane gap:** Operators who want to upgrade a running Iranti instance must:
1. Open a terminal
2. Run `iranti upgrade --restart --instance <name>`
3. Watch the terminal for success/failure output
4. Come back to the control plane to verify the new version

This is a reasonable CLI workflow but it breaks the "operate Iranti without a terminal" promise of the control plane. The Instance Manager already shows instances, their reachability, and (after CP-T072) their runtime state. An upgrade action belongs in the Instance Manager alongside the instance it applies to.

This ticket adds an **Upgrade** action to the Instance Manager's instance detail panel, allowing operators to trigger `iranti upgrade --restart --instance <name>` from the UI.

## What to Build

### Backend — Upgrade Action Endpoint

```
POST /api/control-plane/instances/:name/upgrade
```

**Request body:** `{}` (empty — the instance name comes from the path parameter)

**Implementation:**

The control plane server spawns `iranti upgrade --restart --instance <name>` as a child process and streams its output back to the client. This is a long-running operation (npm install + instance restart can take 30–60 seconds).

**Two implementation options:**

**Option A (recommended) — Async job with polling:**
1. `POST /instances/:name/upgrade` starts the upgrade as a background child process and returns `{ jobId: "...", status: "started" }` immediately (HTTP 202)
2. `GET /instances/:name/upgrade/:jobId` returns the current job status (`{ status: "running" | "complete" | "failed", output: string[], exitCode: number | null }`)
3. Frontend polls every 2 seconds until status is `complete` or `failed`

**Option B (simpler but blocking) — Streamed response:**
1. `POST /instances/:name/upgrade` returns a chunked response streaming the process stdout/stderr as text lines until the process exits
2. Frontend reads the stream and displays output incrementally

Option A is preferred — it is more robust to client disconnects and fits the existing REST pattern of the control plane.

**Security considerations:**
- The instance name from the path must be sanitized before passing to the child process — only allow alphanumeric, hyphen, and underscore characters
- The `iranti` CLI must be on the system PATH; if not found, return a clear error: "iranti CLI not found on PATH. Upgrade must be run manually."
- Log the upgrade attempt and outcome to the control plane's audit log (same pattern as integration repair actions in CP-T033)

**Response shapes:**

```typescript
interface UpgradeJobStarted {
  jobId: string;
  instanceName: string;
  status: 'started';
  startedAt: string;
}

interface UpgradeJobStatus {
  jobId: string;
  instanceName: string;
  status: 'running' | 'complete' | 'failed';
  output: string[];       // lines of stdout/stderr so far
  exitCode: number | null;
  startedAt: string;
  completedAt: string | null;
}
```

### Frontend — Upgrade Action in Instance Manager

**Where to add it:** In the Instance Manager's instance detail panel (the expanded view when an operator clicks an instance card). This panel already shows repair actions (CP-T033). The Upgrade action is a new section below the repair actions.

**Upgrade section layout:**

```
┌─────────────────────────────────────────────────────┐
│  UPGRADE                                            │
│  Running: v0.2.16  Current npm: v0.2.16 (up to date)│
│                                                     │
│  [Upgrade & Restart]                                │
│                                                     │
│  ⚠ Upgrading will restart this instance. Active    │
│  agents will experience a brief interruption.       │
└─────────────────────────────────────────────────────┘
```

**Version display:**
- Show the currently running version from the runtime lifecycle data (CP-T072: `runtime.version`)
- Optionally: check npm for the latest published version and show whether the instance is up to date. This is a stretch goal — if npm version check adds complexity, skip it and just show the running version with a note "Check npm for latest version."

**Upgrade button flow:**

1. Operator clicks **Upgrade & Restart**
2. A confirmation modal appears:
   - Title: "Upgrade and Restart `<instance-name>`?"
   - Body: "This will run `iranti upgrade --restart --instance <name>`. The instance will restart. Active agents will experience a brief interruption. This may take up to 60 seconds."
   - Buttons: "Cancel" and "Upgrade"
3. On confirm: call `POST /api/control-plane/instances/:name/upgrade`, receive `jobId`
4. Show an in-progress panel with a spinner and live output lines (polled from `GET /instances/:name/upgrade/:jobId` every 2 seconds)
5. When status is `complete`: show success message ("Instance upgraded and restarted. New version will appear in the Runtime Status section.") and trigger a health/runtime refetch
6. When status is `failed`: show error message with the last N lines of output and a "Try again" button

**Operator feedback during upgrade:**
- The instance card's runtime badge (CP-T072) should show `RESTARTING` (amber) while the upgrade job is running — update optimistically on job start, revert if job fails

## Acceptance Criteria

### AC-1 — Upgrade endpoint exists and accepts the request

`POST /api/control-plane/instances/:name/upgrade` returns HTTP 202 with `UpgradeJobStarted`. Instance name is validated (alphanumeric + hyphen + underscore only). TypeScript clean.

### AC-2 — Job status endpoint returns current upgrade state

`GET /api/control-plane/instances/:name/upgrade/:jobId` returns `UpgradeJobStatus` with output lines accumulated so far. Status transitions from `running` to `complete` or `failed` when the process exits.

### AC-3 — iranti CLI not found is handled gracefully

If `iranti` is not on PATH, `POST .../upgrade` returns HTTP 400 with a message: "iranti CLI not found on PATH. Upgrade must be run manually: `iranti upgrade --restart --instance <name>`."

### AC-4 — Instance name sanitized

Passing a path traversal string (e.g., `../etc/passwd`) as the instance name returns HTTP 400 with "Invalid instance name."

### AC-5 — Upgrade action appears in instance detail panel

The Instance Manager's expanded instance detail panel includes an "Upgrade" section with a version display and "Upgrade & Restart" button.

### AC-6 — Confirmation modal shown before upgrade starts

Clicking "Upgrade & Restart" shows a confirmation modal with the instance name and the warning about interruption. The upgrade only starts if the operator confirms.

### AC-7 — Progress output shown during upgrade

After confirming, the UI shows an in-progress panel with live output lines polled from the job status endpoint. The spinner is shown while status is `running`.

### AC-8 — Success state shown correctly

When upgrade completes successfully, the success message is shown and the instance's runtime data is refreshed (version should reflect the new version if the instance restarted successfully).

### AC-9 — Failure state shown with output

When upgrade fails (non-zero exit code), the failure message is shown with the last output lines and a "Try again" button.

### AC-10 — Audit log entry

The upgrade attempt (start and outcome) is written to the control plane's audit trail (same mechanism as CP-T033 repair actions).

### AC-11 — tsc clean

`tsc --noEmit` passes in both `src/server` and `src/client`. No `any`.
