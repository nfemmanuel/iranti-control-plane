## Problem Statement

The Iranti Control Plane assumes the user has already installed Iranti, configured a database, set up a provider key, and bound at least one project. For users who have done all of this, Phase 1's Health dashboard is a reasonable landing page. But for a new user who has just installed Iranti and opens the control plane for the first time — or for an existing user who has added a second instance or is onboarding a new team member — there is no guided path. They land on a Health dashboard that may show multiple Critical errors (database unreachable, no provider configured) with no indication of what to do or in what order.

PRD section 9 ("Installation and Onboarding") calls out this as a dedicated required surface:
- "guided install and first-run flow"
- "dependency checks and remediation"
- "instance creation"
- "database setup path selection"
- "provider setup"
- "project binding"
- "Claude/Codex integration setup"
- "verification that the system is actually working after setup"
- "recover from common setup failures without dropping to raw infrastructure commands"

PRD FR9 requires: "Reduce onboarding friction by replacing low-level debugging steps with guided views."

The CLI installer (CP-T023 — `iranti setup`) handles machine-level setup before the control plane is running. This ticket is the complementary in-product surface: what the user sees *inside* the control plane when Iranti is running but not fully configured, or when they need to add a new instance.

**Target users most impacted:**

**Marcus (Solo Dev):** Marcus installed Iranti, opened the control plane, and sees four amber/red health items. He does not know which to fix first, what each means, or that there is a specific sequence (database first, then provider, then project binding). He bounces between docs, terminal, and the control plane for 45 minutes before giving up for the day.

**Priya (Technical Founder):** Priya is onboarding a teammate onto a project already using Iranti. The teammate needs to create a second local instance bound to a different project. There is no guided flow for this — they are expected to know the CLI sequence from memory.

## Scope

**In:**

### First-Run Detection

The backend must detect whether the currently active instance is "fully configured" vs "partially configured" vs "not yet configured":

`GET /api/control-plane/instances/:instanceId/setup-status`

Response:
```typescript
interface SetupStatus {
  instanceId: string;
  steps: Array<{
    id: string;            // e.g., "database", "provider", "project_binding", "claude_integration"
    label: string;         // e.g., "Database connection"
    status: "complete" | "incomplete" | "warning" | "not_applicable";
    message: string;       // Plain-English status description
    actionRequired: string | null;  // If incomplete: specific action to take
    repairAction: string | null;    // If CP-T033 repair action available: its endpoint
  }>;
  isFullyConfigured: boolean;
  firstRunDetected: boolean;      // true if this is the first time the control plane has been opened for this instance
}
```

**Step definitions:**

| Step ID | Complete condition | Incomplete condition |
|---------|-------------------|---------------------|
| `database` | DB reachable, migrations current | DB unreachable or not configured |
| `provider` | At least one provider key present and reachable | No provider key configured |
| `project_binding` | At least one project bound to this instance | No projects bound |
| `claude_integration` | `.mcp.json` present for at least one bound project | No projects have `.mcp.json` |

First-run detection: set a flag in instance metadata (or a local file in the runtime root) when the user completes the Getting Started flow. Until that flag is set, `firstRunDetected: true`.

### Getting Started Screen

A dedicated view accessible at `/getting-started` and automatically shown on first load when `firstRunDetected: true`.

**Layout:**
- A numbered step list showing all 4 setup steps.
- Each step shows: step number, label, status badge (Complete / Incomplete / Warning), and a plain-English description of the current state and required action.
- Completed steps are visually collapsed (checkmark, greyed out) but expandable.
- The first incomplete step is expanded by default with its action prominently shown.

**Step content:**

**1. Database connection:**
- If complete: "Connected to PostgreSQL at [host:port]. [N] facts in knowledge base." (green)
- If incomplete: "Database not reachable at [host:port]. Iranti cannot store or retrieve memory without a database connection." + "Check your `.env.iranti` DATABASE_URL or run `iranti setup --repair-db` from your terminal." + a copyable CLI command.

**2. Provider configuration:**
- If complete: "Provider [name] configured and reachable." (green)
- If incomplete: "No LLM provider configured. Iranti cannot process writes without a provider key." + step-by-step: "1. Add your API key to `.env.iranti` as `ANTHROPIC_API_KEY` or `OPENAI_API_KEY`. 2. Restart Iranti. 3. Click Refresh below."
- A "Refresh" button that re-runs the provider check without full page reload.

**3. Project binding:**
- If complete: "Project [name] bound at [path]." (green). If multiple: "N projects bound."
- If incomplete: "No projects bound to this instance." + "Run `iranti bind [path/to/project]` from your terminal to bind a project directory." + a copyable CLI command with placeholder.

**4. Claude / Codex integration:**
- If complete: ".mcp.json present for [project name]." (green)
- If incomplete with projects present: ".mcp.json not found for [project name]." + repair button (from CP-T033 if available) or CLI instruction: "Run `iranti setup --mcp [path/to/project]`."
- If no projects bound: "Complete Step 3 first." (dimmed, not actionable until Step 3 is done)

**"I'm done" / "Skip" behavior:**
- A "Mark setup complete" button at the bottom sets the `firstRunDetected = false` flag.
- After marking complete, the user is redirected to Memory Explorer (the primary operational surface).
- A "Skip for now" link allows dismissal without marking complete — the Getting Started screen will reappear on next load if any step is incomplete.
- Once all 4 steps are complete, an automatic success state is shown: "Iranti is ready. [green checkmark] All setup steps complete." with a "Go to Memory Explorer" CTA.

### Persistent Setup Status Badge

When any setup step is incomplete and the Getting Started screen has not been dismissed, show a small badge or banner in the shell header:
- "Setup incomplete — [N] steps remaining" with a link to `/getting-started`.
- Dismissible per session (not permanently — reappears on next load until complete).
- Does not block navigation to other views.

### Navigation Entry Point

Add a "Getting Started" item to the navigation sidebar:
- Visible always (not just during first run) so users can return to it.
- Shows a badge count for incomplete steps.
- Positioned above the other nav items when `firstRunDetected: true`; below Health after setup is complete.

**Out:**
- Automated database setup or migration execution from the UI (requires root-level access — remains a CLI operation).
- Provider key entry from the UI (key management is Phase 3 — this ticket shows what to type in the terminal, not a key input field).
- Multi-instance setup wizard (guided creation of a second instance from scratch — complex enough for a separate ticket; this ticket addresses the common "new install" flow for a single instance).

## Dependencies

- Health endpoint must return database reachability, provider configuration state, and project binding status — confirmed available from Phase 1 health checks (CP-T017).
- Repair actions for `.mcp.json` (CP-T033) — if CP-T033 ships before this ticket, wire the repair button directly to its endpoint. If not, show CLI instruction instead. CP-T033 is a soft dependency.
- Shell navigation extension: adding "Getting Started" nav item and setup badge to the shell header (AppShell.tsx).
- First-run flag persistence: must decide on storage location — options are: (a) a record in the `knowledge_base` table keyed to the instance, (b) a local `.iranti-cp-setup-complete` file in the runtime root, or (c) localStorage in the browser. Recommendation: (b) local file in the runtime root — persists across browser sessions and is instance-specific. Confirm with backend_developer.

## Risks

- **Step sequencing correctness:** The 4 steps have dependencies (project binding before Claude integration) but users may fix them out of order. The UI must handle partial state gracefully — do not show step 4 as a problem if step 3 is not yet complete, and do not imply step 4 is complete just because `.mcp.json` was checked against 0 projects.
- **Stale health data on the Getting Started screen:** The setup status reflects the state at load time. If the user goes to the terminal, fixes a step, and comes back, the screen should refresh. Add a "Refresh all" button at the top of the Getting Started screen.
- **Over-eagerness of first-run detection:** If `firstRunDetected` is set too broadly (e.g., on any new session rather than only when the flag is absent), experienced users who open a new browser will be sent to Getting Started repeatedly. The flag must persist at the instance level (runtime root file), not the browser level.

## Open Questions

- Should "Getting Started" redirect to the Health dashboard (which shows the same information at a lower level of detail) or stay as a separate view? Product preference: separate view — Getting Started is task-oriented (actionable steps) while Health is diagnostic (all system signals). They complement each other.
- Should the screen be reachable from Health dashboard directly? E.g., a "View Setup Guide" link on Critical health items? Yes — add this as part of the Health dashboard integration.
