## Problem Statement

No read API exists for `knowledge_base`, `archive`, entity detail, or `entity_relationships`. The Memory Explorer UI (CP-T013) cannot be built without stable, spec-compliant backend endpoints. Frontend and backend must be able to develop in parallel against a shared contract — that contract is CP-T002. This ticket implements it for Groups 1–4 (KB, Archive, Entity Detail, Relationships).

## User Value

Users can query and browse memory, temporal history, and entity relationships from the control plane UI without writing SQL or opening Adminer. This is the core deliverable of Phase 1: making Iranti's memory legible without infrastructure expertise.

## Scope

**In:**
- `GET /api/control-plane/kb` — paginated, filterable knowledge_base query (CP-T002 Group 1)
- `GET /api/control-plane/archive` — paginated, filterable archive query (CP-T002 Group 2)
- `GET /api/control-plane/entities/:entityType/:entityId` — entity detail with current facts, archived facts, relationships (CP-T002 Group 3)
- `GET /api/control-plane/entities/:entityType/:entityId/history/:key` — temporal history for a specific entity+key (CP-T002 Group 3)
- `GET /api/control-plane/relationships` — entity_relationships query with filtering (CP-T002 Group 4)
- All query parameters, response schemas, and error states per CP-T002 spec
- Correct `/api/control-plane/` namespace prefix
- Error shape per CP-T002 `ErrorResponse` type

**Out:**
- Staff event endpoints (CP-T012)
- Instance/health endpoints (CP-T011)
- Auth, write endpoints, rate limiting
- Full-text search (ILIKE MVP only, not tsvector)
- entity_aliases endpoint (pending CP-T006 spike)
- EntityRecord in entity detail response (no `entities` table exists in current Iranti schema — return `entity: null` always in Phase 1; document this in code comment)

## Acceptance Criteria

- [ ] `GET /api/control-plane/kb` returns `KBListResponse` per CP-T002 §Group 1. All 9 query parameters function correctly: `entityType`, `entityId`, `key`, `source`, `createdBy`, `minConfidence`, `archivedReason`, `search` (ILIKE), `limit`, `offset`.
- [ ] `GET /api/control-plane/archive` returns `ArchiveListResponse` per CP-T002 §Group 2. All filter params work: includes `archivedReason`, `resolutionState`, `supersededBy`, `archivedAfter`, `archivedBefore` in addition to base KB params.
- [ ] `GET /api/control-plane/entities/:entityType/:entityId` returns `EntityDetailResponse` per CP-T002 §Group 3. `entity` field is always `null` (no entities table). `currentFacts` and `archivedFacts` are populated. `relationships` populated via entity_relationships query.
- [ ] `GET /api/control-plane/entities/:entityType/:entityId/history/:key` returns `KeyHistoryResponse` with all KB + archive intervals for that entity+key, ordered `validFrom DESC`. Full `valueRaw` is returned (no 4KB truncation on this endpoint).
- [ ] `GET /api/control-plane/relationships` returns `RelationshipListResponse` per CP-T002 §Group 4. `entityId` bidirectional lookup works (matches fromEntityId OR toEntityId).
- [ ] Filter params all validated: `minConfidence` must be integer 0–100; `limit` max 500; invalid ISO 8601 dates for `archivedAfter`/`archivedBefore` return 400 with `INVALID_PARAM` code.
- [ ] `valueRaw` in list responses truncated to 4KB with `valueRawTruncated: true` flag when truncated.
- [ ] Response times under 500ms for queries returning up to 100 rows against local DB.
- [ ] All endpoints return 503 `DB_UNAVAILABLE` when database connection fails.
- [ ] Endpoints mounted at `/api/control-plane/` — no conflict with existing Iranti API routes. Verify against existing route registration.
- [ ] All response field names match CP-T002 spec exactly (camelCase) with correct mapping from Prisma snake_case columns.

## Risks

- `entities` table does not exist in current Iranti schema — EntityRecord will always be null. Document clearly in both code and handoff to frontend.
- Prisma column names may differ from spec assumptions — must verify against actual Prisma schema before implementing field mapping.
- `total` count in paginated responses requires `COUNT(*)` queries — confirm this does not cause performance issues with large tables. If so, make `total` optional/null with a `limitedCount` flag.
- `search` ILIKE on `value_raw` (JSONB) may require casting — confirm Prisma/PostgreSQL behavior.

## Open Questions

- What are the exact Prisma model field names for knowledge_base and archive? (snake_case vs camelCase in Prisma schema)
- Does the `archive` table have `archivedAt`, `archivedReason`, `resolutionState`, `supersededBy` columns? Confirm against actual Prisma schema.
- Is `validFrom` / `validUntil` stored as TIMESTAMPTZ in the DB? Confirm for correct ISO 8601 serialization.
