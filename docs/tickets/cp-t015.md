## Problem Statement

No UI exists to see active instances, their config state, or project bindings. Operators must inspect `.env.iranti` files, run `iranti status`, or query the database to understand what instances exist and whether they are correctly configured. The Instance Manager gives operators one place to see all of this without manual inspection.

## User Value

Operators can see which Iranti instances exist on the host, inspect their configuration (port, database target, env file status, provider keys), view integration health indicators (Claude MCP, Codex), and switch the active instance context without running CLI commands.

## Scope

**In:**
- Instance list + detail panel two-column layout (`/instances` route) per CP-T004 wireframe (Section 1.5)
- Instance list (left column): all discovered instances from `/api/control-plane/instances`. Active instance highlighted. Instance cards show: instanceId, port, runningStatus indicator, database target summary, project count.
- Instance detail panel (right column): all `InstanceMetadata` fields from CP-T003 output schema: runtimeRoot, database (host/port/name/urlRedacted), configuredPort, runningStatus + checkedAt, irantVersion, envFile status and key completeness, integration (defaultProvider, defaultModel, providerKeys).
- Bound Projects section in detail panel: shows `projects` array. Per PM OQ-2 decision, Phase 1 shows empty state with message "Project binding data unavailable in Phase 1 — run CP-T006 spike" if projects array is empty/unavailable.
- Integration health indicators per project entry: claudeMdPresent, mcpConfigPresent, mcpConfigHasIranti, codexIntegration.configPresent — shown as compact status icons (check/warn/cross) per CP-T004 wireframe.
- Instance context switcher: clicking "Set as Active" on an instance updates the sidebar instance context switcher to show that instance's name. Context is stored in local app state.
- Deep-link route: `/instances/:instanceId` deep-links to a specific instance's detail panel.
- Empty state: no instances discovered.
- Error states for all missing/degraded fields shown as status indicators, not raw errors.

**Out:**
- Instance creation, deletion
- Env file editing from the UI
- Project binding creation/removal (Phase 2)
- "Repair" and "Setup" actions on integrations (Phase 2)
- "Run Doctor" action (Phase 2)

**Per PM OQ-2 decision:** Project bindings are stubbed in Phase 1. The projects section shows an informative empty state, not a broken UI. Once CP-T006 spike confirms a binding source, a follow-on ticket will implement full project binding display.

## Acceptance Criteria

- [ ] Instance list renders all discovered instances from `/api/control-plane/instances`. Each instance card shows: running status indicator (color-coded: green=running, gray=stopped, yellow=unreachable), port, database target summary (host:port/dbname).
- [ ] Clicking an instance in the list loads its detail panel. Deep-link `/instances/:instanceId` loads the correct instance directly.
- [ ] Detail panel renders all `InstanceMetadata` fields from CP-T003: runtimeRoot, database block (host, port, name, urlRedacted), configuredPort, runningStatus + checkedAt timestamp, irantVersion (or "unknown" if null), envFile present/missing + key completeness summary.
- [ ] Integration block shows: defaultProvider, defaultModel (or "not configured"), providerKeys (anthropic: present/absent, openai: present/absent).
- [ ] Missing/error states shown clearly and non-technically: missing env file → "No .env.iranti found — check runtime root", offline instance → "Instance offline as of [time]", missing database connection → "Database unreachable".
- [ ] Bound Projects section shows empty state with explanation per PM OQ-2 decision when projects array is empty.
- [ ] Integration health indicators (claudeMd, mcpConfig, codex) render as compact status icons when project data is available.
- [ ] "Set as Active" button updates the sidebar instance context switcher (in the shell from CP-T017) to show the selected instance name. This state persists within the session.
- [ ] Empty state: when no instances are discovered, shows a helpful message with guidance to check the Iranti runtime root.
- [ ] Both light and dark mode render correctly per Option B tokens.
- [ ] `discoveredAt` timestamp shown in the UI ("Last discovered: 2 minutes ago") to communicate data freshness to operators.

## Risks

- Instance context switcher state must be shared between the Instance Manager view and the sidebar in CP-T017 shell. Use React context or a lightweight Zustand store for this shared state.
- Running status staleness: the instance health check has a 500ms timeout. The UI must show `runningStatusCheckedAt` so operators know when the status was last verified, not imply it is live.
- If `irantVersion` is null, do not display "null" — show "Version unknown" or similar.

## Open Questions

- Should the instances list auto-refresh on a timer (e.g., every 30 seconds), or only on manual refresh? Recommendation: manual refresh button + refresh on mount. No auto-refresh in Phase 1.
- Should the detail panel load in the same view or be a separate route with full-page layout? Per CP-T004 wireframe: two-column within `/instances`. Deep-link `/instances/:instanceId` shows same two-column with specific instance selected.
