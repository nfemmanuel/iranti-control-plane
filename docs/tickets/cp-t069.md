## Summary

v0.3.0 has health information on the `/health` route and the new Home Dashboard (CP-T068). But neither surface proactively tells the operator when something goes wrong while they are using the app on another view. If the database goes down while an operator is reading the Memory Explorer, there is no signal â€” they only discover it when a data fetch fails with a raw error.

Docker Desktop shows system-level alerts in a notification area regardless of which panel is open. This ticket adds the equivalent: a lightweight toast/notification system that surfaces health degradation proactively â€” without requiring the operator to visit `/health`.

This is purely a frontend feature. It reuses the existing `GET /api/control-plane/health` endpoint that the AppShell already polls every 30 seconds via `useApiReachability`.

#### Health Degradation Poller

Extend the existing `useApiReachability` pattern in `AppShell.tsx` â€” or create a parallel hook `useHealthPoller` â€” that:

1. Polls `GET /api/control-plane/health` every **60 seconds** (not 30 â€” the overview page already polls frequently; this is a background watchdog).
2. On each poll, compares `overall` status against the previous result.
3. When `overall` transitions from `'healthy'` to `'degraded'`:
   - Fire a `warn` toast: title "Iranti degraded", message "One or more health checks are warning. [View Health â†’]"
4. When `overall` transitions from `('healthy' | 'degraded')` to `'error'`:
   - Fire an `error` toast: title "Iranti health error", message "A critical health check has failed. [View Health â†’]"
5. When `overall` transitions back to `'healthy'` from a non-healthy state:
   - Fire an `info` toast: title "Iranti healthy", message "All health checks are passing." (auto-dismiss in 4 seconds)
6. On initial load: if `overall` is already not healthy, fire the appropriate toast immediately (so an operator who opens the app in a degraded state sees the alert).

State tracking uses a React ref (`previousOverallRef`) to compare poll results without causing re-renders.

The poller must only fire toasts when there is a **state transition** or on the initial check. It must not re-fire the same alert every 60 seconds.

#### AppShell Integration

In `AppShell.tsx`:

1. Import and mount `<ToastContainer />` at the bottom of the `<div className={styles.shell}>` wrapper (after `<ChatPanel />`).
2. Pass `addToast` from `useToasts` down to the health poller.
3. Thread `toasts` and `dismissToast` into `<ToastContainer />`.

The `ToastContainer` renders fixed-positioned, so its exact DOM position in the tree doesn't affect layout.

## Acceptance Criteria

### AC-1 â€” Toast renders correctly in all three severity variants

Render an `error`, `warn`, and `info` toast. Each has the correct color semantics (rose/amber/emerald), severity icon, title, message, and dismiss button. Terminals palette tokens only â€” no hardcoded hex.

### AC-2 â€” Auto-dismiss works per severity

`info` toasts auto-dismiss after 6 seconds. `warn` toasts auto-dismiss after 10 seconds. `error` toasts are sticky (do not auto-dismiss). Dismiss button always works immediately.

### AC-3 â€” Toast deduplication

If the same `title` toast is already visible, a second call to `addToast` with the same title does not add a duplicate. The existing toast remains unchanged.

### AC-4 â€” Health degradation fires toast

With the health poller running, if the health endpoint returns `overall: 'degraded'` after previously returning `'healthy'`, a `warn` toast with title "Iranti degraded" and a "View Health â†’" action link appears.

### AC-5 â€” Health error fires error toast

If the health endpoint returns `overall: 'error'`, an `error` (sticky) toast with title "Iranti health error" appears. It does not auto-dismiss.

### AC-6 â€” Recovery fires info toast

When health transitions back to `'healthy'` from a non-healthy state, an `info` toast "Iranti healthy" appears and auto-dismisses in 4 seconds.

### AC-7 â€” Initial state check

On first load, if the health endpoint already returns non-healthy, the appropriate toast fires immediately (not waiting for a state transition).

### AC-8 â€” No toast spam

The same alert does not fire more than once per state transition. Polling every 60 seconds while in `'degraded'` state does not repeatedly add the degraded toast.

### AC-9 â€” Maximum 4 toasts

If 5+ toasts would be visible simultaneously, the oldest is dismissed to make room. The visible cap is 4.

### AC-10 â€” tsc clean

`tsc --noEmit` passes in `src/client`. No `any`.

### AC-11 â€” No regressions

All existing shell functionality (command palette, chat panel, theme toggle, setup banner, instance switcher) continues to work correctly.

## Implementation Notes

- The `useToasts` hook uses `useReducer` or `useState` with an array of toast objects. Each toast gets a `crypto.randomUUID()` or `Date.now().toString()` as its `id`.
- The health poller interval is 60 seconds. It shares the `fetch('/api/control-plane/health')` call with `useApiReachability` but should be a separate hook to keep concerns clean. Do not merge them â€” they serve different purposes (API reachability indicator vs. health degradation alerting).
- CSS transitions for toast enter/exit: use a CSS class toggle approach. On mount, add `data-visible="true"`. On dismiss, add `data-dismissing="true"`, wait for the CSS transition to complete (300ms), then remove from state. Use `setTimeout` for the cleanup.
- The `action.href` on toasts is a route path (e.g. `'/health'`). Render it as a `<Link to={href}>` from react-router-dom so navigation works correctly within the SPA.
- Keep the toast `z-index` above the command palette (`z-index: 1001` or higher).
