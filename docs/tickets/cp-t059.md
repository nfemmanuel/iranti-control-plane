# CP-T059 — Interactive Diagnostics Panel

**Type:** Feature
**Priority:** P2
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 6)
**Status:** PM-ACCEPTED 2026-03-21 (backend + frontend both accepted)
**Epic:** CP-E012 (Diagnostics)

## Summary

The current Health Dashboard is a passive read-only view: it shows what the control plane already knows about system state. It does not let operators actively trigger diagnostic checks, probe connectivity, or run end-to-end smoke tests from within the UI.

This ticket adds an Interactive Diagnostics Panel — a "Run Diagnostics" surface on the Health Dashboard where operators can actively trigger connectivity checks, Staff round-trip tests, and vector search validation, and receive actionable results with fix suggestions. This is the control plane equivalent of `iranti doctor` but with richer output, live feedback, and operator guidance.

## Why This Exists

Operators who encounter a problem today must:
1. Open a terminal
2. Run `iranti doctor` or `iranti doctor --debug`
3. Parse CLI output
4. Figure out what to do next

This is friction. The control plane should be the operator's primary diagnostic surface — not a fallback. A diagnostics panel that runs live checks, shows pass/fail/warn per check, and surfaces actionable fix commands is a high-value, differentiating feature.

From the user's direction (2026-03-21): "Run `iranti doctor` equivalent from the UI. Test connectivity to Iranti, to providers, to the database. Surface actionable error messages with fix suggestions."

## Acceptance Criteria

### Backend (backend_developer)

#### AC-1 — `POST /api/control-plane/diagnostics/run` endpoint

Trigger a full diagnostic run. Returns a structured result object. This endpoint must complete within 10 seconds — use a timeout per check, not a global timeout. Non-blocking: accept `?async=true` to return a job ID and poll separately if needed, but synchronous is preferred for simplicity at MVP.

Diagnostic checks to run:

| Check | What it tests | Pass condition |
|-------|--------------|----------------|
| `iranti_connectivity` | HTTP GET to Iranti `/health` | Returns 200 with `{ status: "ok" }` |
| `iranti_auth` | HTTP GET to Iranti `/kb/search?query=test&limit=1` with the configured API key | Returns 200 (not 401/403) |
| `db_connectivity` | Query `SELECT 1` against the control plane's local SQLite or the Iranti-proxied DB | Returns a row |
| `vector_backend` | HTTP GET probe to configured `IRANTI_VECTOR_BACKEND` URL (if qdrant/chroma) or `SELECT 1` for pgvector | Connection succeeds within 3s |
| `ingest_roundtrip` | Write a test fact (`entityType: '__diagnostics__'`, `entityId: '__probe__'`, `key: 'probe_timestamp'`) via `POST /kb/write`, then immediately read it back via `GET /kb/query`. Delete it after. | Write returns 200, read returns the written value |
| `attend_check` | Call `POST /memory/attend` with a minimal payload (`{ agent: 'control_plane_operator', currentContext: 'diagnostic probe' }`). Check that it returns 200 without `classification_parse_failed_default_false` in the response. | Returns 200 with a parseable result |
| `vector_search_check` | Call `GET /kb/search?query=diagnostic+probe&limit=1`. Check that `vectorScore > 0` for any returned result, or that the fallback in-process scoring is active. | Returns 200; if `vectorScore=0` for all results, flag as `warn` (not `fail`) |

Each check result:
```json
{
  "check": "iranti_connectivity",
  "status": "pass" | "warn" | "fail",
  "message": "Iranti v0.2.14 reachable at http://localhost:3001",
  "fixHint": null,
  "durationMs": 45
}
```

`fixHint` is a human-readable action the operator can take to resolve the issue. Examples:
- `iranti_connectivity` fail → `"Iranti may not be running. Run: iranti run --instance <name>"`
- `iranti_auth` fail → `"API key missing or insufficient scope. Check IRANTI_API_KEY in your .env.iranti. Required scope: kb:read, kb:write, memory:read, memory:write."`
- `vector_backend` warn → `"Vector backend unreachable. Vector search will use in-process fallback. Check IRANTI_QDRANT_URL or IRANTI_CHROMA_URL."`
- `attend_check` warn → `"Attendant classifier returned a parse failure. Memory injection may be non-functional. Known issue in Iranti < 0.2.13. Run: iranti upgrade"`

Full response shape:
```json
{
  "runAt": "2026-03-21T10:00:00.000Z",
  "overallStatus": "pass" | "warn" | "fail",
  "checks": [ CheckResult ],
  "totalDurationMs": 850
}
```

`overallStatus` = `fail` if any check is `fail`, `warn` if any check is `warn` and none fail, `pass` if all pass.

#### AC-2 — `GET /api/control-plane/diagnostics/last` endpoint

Returns the result of the most recent diagnostics run (in-memory cache — no persistence required at MVP). Returns 404 if no run has been performed yet in this server session. This enables the frontend to show the last result on page load without triggering a new run.

#### AC-3 — Graceful degradation

If a check itself throws (e.g., network error mid-probe), catch the exception and mark that check as `fail` with the exception message as the `message` field. Never 500 the diagnostics endpoint.

#### AC-4 — TypeScript clean

No `any`, no type assertions without comment. `tsc --noEmit` passes.

### Frontend (frontend_developer)

#### AC-5 — "Run Diagnostics" button in Health Dashboard

In `src/client/src/components/health/HealthDashboard.tsx`, add a "Run Diagnostics" button in the Health Dashboard header (alongside or near existing actions). Clicking it:
1. Shows a loading state ("Running diagnostics…" with a spinner)
2. Calls `POST /api/control-plane/diagnostics/run`
3. Renders the results inline (see AC-6)
4. On error (network failure, 503): shows "Diagnostics unavailable. Check that the control plane server is running."

On initial page load, call `GET /api/control-plane/diagnostics/last`. If it returns a previous result, show it in a collapsed "Last Run" section below the button. If 404, show nothing (no previous run).

#### AC-6 — Diagnostic results panel

Render each check result as a row in a results table:
- Check name (human-friendly label, not the internal key)
- Status badge: Pass (emerald), Warn (amber), Fail (red) — consistent with the CP-T028 severity taxonomy
- Message (one line of text)
- Duration (`45ms` right-aligned)
- Fix hint (if non-null): show below the message in a muted helper style, with any `iranti ...` commands rendered in inline monospace

Overall status is shown as a summary banner at the top of the results panel:
- Pass: "All checks passed" in emerald
- Warn: "N warning(s) — system functional but degraded" in amber
- Fail: "N failure(s) detected — action required" in red

The panel is collapsible. Default: expanded after a run, collapsed on page load (showing only summary if a last-run result exists).

#### AC-7 — Keyboard trigger

"Run Diagnostics" is reachable via the command palette (Cmd+K). Register it as "Run Diagnostics" in the palette command list.

#### AC-8 — TypeScript clean

No `any` in new component tree. `tsc --noEmit` passes.

#### AC-9 — Filter `__diagnostics__` from Memory Explorer default browse view

The `ingest_roundtrip` check writes `entityType: '__diagnostics__'` probe facts to Iranti. These are infrastructure entities — not operator memory — and must not appear in the Memory Explorer browse view by default.

In the Memory Explorer browse query or frontend filter logic, exclude `entityType: '__diagnostics__'` from default results. Operators who specifically set Entity Type to `__diagnostics__` in the filter bar should still be able to see probe entities (for debugging purposes). This is a **UI default filter only**, not a data deletion.

Note: best-effort DELETE of the probe fact is already implemented in the backend (2s timeout, silently swallowed). This AC is a safety net for cases where DELETE fails.

## Implementation Notes

- The `ingest_roundtrip` check writes a test fact with `entityType: '__diagnostics__'`. This is by design — it's a known probe entity. The filtering requirement is captured in AC-9 above.
- The `attend_check` is informational/observational — it checks whether `iranti_attend` returns a parseable result. It does not require injecting memory into a real conversation. A minimal payload is sufficient.
- The diagnostics backend should cache only the last result (a single in-memory object). No database or file persistence is required at MVP.
- `fixHint` values should be copy-edited and consistent in tone: present tense, actionable, start with a diagnosis.
