# CP-T067 — Entity Type Browser: Browse KB by Entity Type

**Type:** Feature
**Priority:** P3 (Wave 9)
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 9)
**Status:** Open — issued 2026-03-21
**Epic:** CP-E014 (Knowledge Base Power Features)

## Summary

Operators browsing an unfamiliar Iranti instance currently have no way to answer "what entity types exist in this KB?" without knowing the entity types in advance. The Memory Explorer requires the operator to type or select an entity type before browsing — there is no discovery view.

This ticket adds an Entity Type Browser: a landing panel in the Memory Explorer that shows all distinct entity types currently in the KB, their fact counts, and a "Browse" link that filters the Memory Explorer to that entity type. This gives new operators a map of the KB's structure before they start drilling down.

## Background

The KB is organized by `(entityType, entityId, key)`. The Iranti API's `GET /kb` endpoint supports filtering by `entityType` but does not return a list of distinct entity types. The control plane's own database can answer this query:

```sql
SELECT entity_type, COUNT(*) as fact_count
FROM knowledge_base
GROUP BY entity_type
ORDER BY fact_count DESC;
```

This is a read-only aggregation against the control plane's local `knowledge_base` table (synced from Iranti). It does not require a new Iranti API call.

## What to Build

### Backend — `src/server/routes/control-plane/kb.ts`

Add a new endpoint:

```
GET /api/control-plane/kb/entity-types
```

Returns:
```json
{
  "entityTypes": [
    { "entityType": "agent", "factCount": 147, "lastUpdatedAt": "2026-03-21T..." },
    { "entityType": "project", "factCount": 83, "lastUpdatedAt": "2026-03-20T..." }
  ],
  "total": 12
}
```

Query: `SELECT entity_type, COUNT(*) as fact_count, MAX(updated_at) as last_updated_at FROM knowledge_base GROUP BY entity_type ORDER BY fact_count DESC`.

The `lastUpdatedAt` field is the maximum `updated_at` (or `created_at` if `updated_at` is null) for any fact in that entity type — indicating recency.

### Frontend — Memory Explorer landing view

When no `entityType` filter is active (the default state of the Memory Explorer), show the Entity Type Browser instead of an empty table or the full unfiltered fact list.

The Entity Type Browser shows:
- A header: "Knowledge Base — [N] entity types"
- A grid or list of entity type cards, each showing:
  - Entity type name (e.g., `agent`, `project`, `user`)
  - Fact count badge (e.g., "147 facts")
  - Last updated relative time
  - "Browse →" link that activates the `entityType` filter in the Memory Explorer

When an operator clicks "Browse →" on an entity type card, the Memory Explorer switches to its normal browse mode filtered to that entity type.

This is a zero-distraction landing — it does not replace the existing browse mode, it precedes it.

## Acceptance Criteria

### AC-1 — Backend endpoint returns distinct entity types with counts

`GET /api/control-plane/kb/entity-types` returns `{ entityTypes: [{ entityType, factCount, lastUpdatedAt }], total }`. Results ordered by `factCount` descending. TypeScript clean, no `any`.

### AC-2 — Memory Explorer shows Entity Type Browser on initial load

When no `entityType` filter is active, the Memory Explorer shows the Entity Type Browser view (not an empty table or unfiltered list).

### AC-3 — Each entity type card links to filtered browse

Clicking a card or "Browse →" activates the entity type filter and switches to normal Memory Explorer browse mode for that entity type. URL should update to include the filter (e.g., `?entityType=agent`) for shareability.

### AC-4 — Fact count and recency are accurate

Fact counts match the current KB. `lastUpdatedAt` reflects the most recently modified fact in each entity type.

### AC-5 — Empty state

If the KB is empty (no entity types), show: "No entities in the knowledge base yet. Once agents start writing facts, entity types will appear here." with a link to the Staff Logs view.

### AC-6 — Does not break existing Memory Explorer

Existing entityId filter, key filter, search, and pagination continue to work when an entity type is selected. "Back to entity types" or clearing the entityType filter returns to the browser.

### AC-7 — TypeScript clean (both)

`tsc --noEmit` passes in `src/client` and `src/server`. No `any`.

## Implementation Notes

- This is a purely additive feature. The existing Memory Explorer browse mode is unchanged — the Entity Type Browser is the new initial state.
- The entity types endpoint queries the local `knowledge_base` table — no Iranti API call required.
- Consider caching: the entity type list changes infrequently. A `staleTime` of 2–5 minutes is appropriate.
- The grid of cards should work in both light and dark mode using the Terminals palette.
