## Summary

Iranti exposes a full agent registry via `GET /agents` and `GET /agents/:agentId`, returning rich per-agent stats (totalWrites, totalRejections, totalEscalations, avgConfidence, lastSeen, isActive). This data is currently invisible in the control plane — operators have no way to see which agents are registered, what they're doing, or whether any are producing high rejection or escalation rates.

This ticket adds a read-only Agent Registry view.

## Background (from cross-repo audit 2026-03-21)

Iranti's API exposes:
- `GET /agents` — list all registered agents; requires `agents:read` scope
- `GET /agents/:agentId` — single agent record; requires `agents:read` scope

Agent record shape (from Iranti source):
```json
{
  "agentId": "backend_developer",
  "name": "Backend Developer",
  "description": "...",
  "capabilities": [],
  "model": "claude-sonnet-4-6",
  "properties": {},
  "stats": {
    "totalWrites": 14,
    "totalRejections": 0,
    "totalEscalations": 2,
    "avgConfidence": 85,
    "lastSeen": "2026-03-21T09:00:00.000Z",
    "isActive": true
  }
}
```

## Acceptance Criteria

### Backend (backend_developer)

#### AC-1 — `GET /api/control-plane/agents` endpoint
Proxy or relay `GET /agents` from the Iranti runtime for the connected instance. Requires `agents:read` scope. Response shape:
```json
{
  "agents": [AgentRecord],
  "total": 5
}
```

#### AC-2 — `GET /api/control-plane/agents/:agentId` endpoint
Proxy or relay `GET /agents/:agentId`. Returns the full agent record. Returns 404 if not found.

#### AC-3 — Graceful degradation
If the upstream `GET /agents` returns 401 (no agents:read scope) or is unreachable, return `HTTP 503` with `{ error: "...", code: "AGENTS_UNAVAILABLE" }`. Do not 500.

#### AC-4 — Auth scope forwarded
The control plane must forward the `X-Iranti-Key` header with `agents:read` scope when calling the upstream agent registry. See how the existing `chat.ts` and `providers.ts` routes pass the key.

### Frontend (frontend_developer)

#### AC-5 — `/agents` route in sidebar
Add an "Agents" navigation item to the sidebar. Position: after Providers (8th item, before Getting Started). Use a suitable icon (person-group or similar from the existing Lucide icons already used in the project).

#### AC-6 — Agent list view
Show a table of registered agents with:
- Agent ID
- Display Name (if different from ID)
- Last Seen (relative time)
- Active indicator (green dot / grey dot)
- Total Writes (integer)
- Rejections (integer, red if > 0 and high relative to writes)
- Escalations (integer, amber if > 0)
- Avg Confidence (%)

Sort by lastSeen descending by default.

#### AC-7 — Agent detail drawer or page
Clicking an agent row opens a detail view showing:
- All stat fields from the list view
- Capabilities array (if non-empty)
- Model
- Properties (raw JSON expandable)
- Description

Keep consistent with the existing Terminals visual palette (emerald accent, near-black canvas).

#### AC-8 — Empty state
If no agents are registered, show: "No agents registered yet. Agents appear here after their first `iranti_handshake` call." Do not show an error.

#### AC-9 — 503 empty state
If the agents API is unavailable (no `agents:read` scope or Iranti unreachable), show the same 503 UI pattern used by other views (e.g., Staff Logs 503 state).

#### AC-10 — Sidebar badge (optional / stretch)
If any agent has `isActive: false` and `lastSeen` > 24 hours ago with `totalEscalations > 0`, show a subtle count badge on the Agents nav item. This is a stretch goal — only implement if the rest of the view is clean.

## Dependencies

- Iranti running with an API key that has `agents:read` scope
- The control plane's Iranti connection configured correctly
- Pattern reference: `src/server/routes/control-plane/providers.ts` (proxy pattern), `src/client/src/components/providers/ProviderManager.tsx` (list view pattern)
