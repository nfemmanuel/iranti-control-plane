## Problem Statement

No UI exists to browse memory. Users must use raw SQL or Adminer to inspect what Iranti believes about any entity. The Memory Explorer is the primary Phase 1 surface and the clearest proof point that the control plane adds real value: a user can answer "what does Iranti currently believe about this entity?" in under 30 seconds (ER1) without SQL.

## User Value

Operators can browse current facts, inspect temporal history, follow entity relationships, and view raw JSON values — all without database access. This directly closes the operability gap described in the PRD problem statement.

## Scope

**In:**
- KB facts table view (`/memory` route): paginated table with filter bar. All filter params from CP-T010: entityType, entityId, key, source, createdBy, minConfidence, search, archivedReason, limit/offset pagination.
- Archive table view (`/archive` route): same pattern as KB table with archive-specific filters: archivedReason, resolutionState, supersededBy, archivedAfter/Before.
- Entity detail page (`/memory/:entityType/:entityId` route): current facts + archived facts + relationships for one entity.
- Temporal history timeline view (`/memory/:entityType/:entityId/:key` route): all intervals (KB + archive) for a specific entity+key, in chronological order. Shows validFrom, validUntil, archivedReason, supersededBy.
- Relationship view: shown within entity detail. Jump links from related entity entries navigate to that entity's detail page.
- Raw JSON inspector toggle: reveals full valueRaw and properties fields.
- Per shell wireframe in CP-T004 (Section 1.3): expandable row accordion, sticky filter bar, sortable columns.
- Visual system per Option B (Terminals) approved in decision/visual_direction.
- Empty states for all views (no results, no entity found, DB unavailable).

**Out:**
- Write operations, editing, deletion
- Chat
- Staff stream (CP-T014)
- entity_aliases display (pending CP-T006 spike — no aliases table exists yet)

## Acceptance Criteria

- [ ] KB table (`/memory`) renders all KBFact fields from CP-T010: entityType, entityId, key, valueSummary, confidence, source, agentId, validFrom, validUntil, updatedAt.
- [ ] All filter params from CP-T010 are represented in the filter bar and correctly affect the query: entityType, entityId, key, source, createdBy, minConfidence, search, limit.
- [ ] Archive table (`/archive`) renders all ArchiveFact fields from CP-T010, with archive-specific columns visible: archivedAt, archivedReason, resolutionState, supersededBy.
- [ ] Temporal history view (`/memory/:entityType/:entityId/:key`) shows all intervals for a given entity+key in chronological order. Each interval shows: source (kb/archive badge), valueSummary, confidence, validFrom, validUntil, archivedAt, archivedReason. Full valueRaw visible when raw JSON toggle is active.
- [ ] Entity detail page shows current facts (from currentFacts), all archived facts (from archivedFacts), and related entities (from relationships). Entity section header shows `null`/empty gracefully since EntityRecord is always null in Phase 1.
- [ ] Relationship entries on entity detail page are jump links: clicking navigates to `/memory/:entityType/:entityId` for the related entity.
- [ ] Raw JSON toggle reveals full valueRaw and properties fields in a monospace code block. Toggle is per-row, not global.
- [ ] Empty states: all views have a meaningful empty state for 0 results (not a blank page). DB unavailable (503) shows an error state with retry prompt.
- [ ] Filter state persists within a session for each view (not required to survive page refresh). Filter state is isolated per route.
- [ ] All views render correctly in both light and dark mode per Option B tokens from CP-T017.
- [ ] Pagination controls work correctly: prev/next, page size selection (25/50/100).
- [ ] Loading states shown while data is fetching (skeleton rows or spinner per shell design).
- [ ] Column sort works for at minimum: updatedAt (default desc), confidence, entityType, key.

## Risks

- `entity` field in EntityDetailResponse is always null — must not display a broken "Entity not found" state; display entity identity from the facts themselves instead.
- Large valueRaw values (>4KB in list view): show truncated summary with "view full value" trigger that loads the history endpoint for full value.
- Filter bar at the Memory Explorer scale has many controls — ensure the layout does not overflow on common display sizes (1280px wide minimum target).
- Temporal history timeline order: ensure the UI correctly interprets validFrom=null (no start) and validUntil=null (currently valid).

## Open Questions

- Should the KB table have a column for `conflictLog` presence indicator (a badge showing "conflict history exists")? Deferred unless PM adds to scope.
- Should the archive view be a separate nav item (current wireframe: yes, `/archive` is a separate nav destination) or a tab within Memory Explorer? Per CP-T004 wireframe: separate nav item.
