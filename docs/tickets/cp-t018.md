## Problem Statement

Backend API endpoints need systematic validation against the CP-T002 spec before frontend views are built on them. Frontend tickets (CP-T013–CP-T016) depend on stable, spec-compliant API behavior. Defects found during UI development are more expensive to fix than defects found during API QA. A written test plan also creates a shared reference for what was tested and what was not.

## User Value

Frontend engineers build on a verified API contract. Phase 1 completes with documented evidence that the API behaves as specified, not just as assumed. Known deviations from the spec are surfaced to the PM before they become UI bugs.

## Scope

**In:**
- Written test plan covering all endpoint groups from CP-T002 spec:
  - Group 1: `GET /api/control-plane/kb`
  - Group 2: `GET /api/control-plane/archive`
  - Group 3a: `GET /api/control-plane/entities/:entityType/:entityId`
  - Group 3b: `GET /api/control-plane/entities/:entityType/:entityId/history/:key`
  - Group 4: `GET /api/control-plane/relationships`
  - Group 5a: `GET /api/control-plane/instances`
  - Group 5b: `GET /api/control-plane/instances/:instanceId/projects`
  - Group 6: `GET /api/control-plane/health`
  - Group 7a: `GET /api/control-plane/events`
  - Group 7b: `GET /api/control-plane/events/stream` (SSE)
- For every filter parameter: at least one valid input test and one invalid input test
- Edge case tests per CP-T003 §4 and CP-T002 error states
- SSE stream test: connect, receive events, verify Last-Event-ID resume
- Test results documented in `docs/test-plans/phase1-api-test-plan.md`

**Out:**
- UI testing (that is a separate Phase 1 QA activity, post-CP-T013–CP-T016 completion)
- Performance/load testing
- Security testing beyond confirming no credentials are returned in responses

## Acceptance Criteria

- [ ] Test plan document exists at `docs/test-plans/phase1-api-test-plan.md` before testing begins.
- [ ] Test plan covers all 10 endpoint paths (Groups 1–7 including stream).
- [ ] Every filter parameter from CP-T002 is represented in the test plan with at least one valid input test case and one invalid input test case (e.g., `minConfidence=abc` returns 400, `minConfidence=50` returns correct filtered results).
- [ ] All edge cases from CP-T003 §4 are tested:
  - Missing `.env.iranti` → instance endpoint returns `envFile.present: false` and correct null fields
  - Instance not running → `runningStatus: "stopped"` or `"unreachable"`
  - Project path missing from filesystem → `projectExists: false`
  - Malformed `.mcp.json` → `mcpConfigPresent: true`, `mcpConfigHasIranti: false`, `mcpConfigError` populated
- [ ] Response shapes validated against CP-T002 TypeScript interfaces: correct field names (camelCase), correct types, correct nullability.
- [ ] Credentials never returned: `DATABASE_URL` value not present in any response, API key values not present in any response.
- [ ] Health endpoint `staff_events_table` check returns `warn` if CP-T001 migration has not been applied.
- [ ] SSE stream test: connect to `/events/stream`, verify event format matches `StaffEvent` schema, verify `Last-Event-ID` cursor resume works (disconnect → reconnect → no missed events, no duplicates).
- [ ] All test results documented in `docs/test-plans/phase1-api-test-plan.md` with: test case ID, description, input, expected output, actual output, pass/fail status.
- [ ] Any failures or deviations from spec are reported to backend_developer with the test case ID and actual vs expected behavior. Test plan is not marked complete until all failures are resolved or explicitly accepted by PM as known limitations.

## Risks

- SSE stream testing requires tooling that can maintain a persistent HTTP connection and capture events over time (e.g., curl, Postman, a small test script). QA engineer should prepare this tooling before beginning SSE tests.
- Edge case tests for missing env file and offline instance require the QA environment to be deliberately misconfigured for those tests — document setup steps for each edge case test carefully.
- If CP-T010/CP-T011/CP-T012 are completed at different times, QA can begin with the endpoints that are ready — do not wait for all three. Start with CP-T010 (KB endpoints) as they are the most testable independently.

## Open Questions

- Should QA use a dedicated test database seeded with known data, or test against the live Iranti development database? Recommendation: dedicated test database with seeded KB entries for reproducible filter tests. QA to request seed data from backend_developer.
- Should the test plan include negative test cases for the SSE stream (e.g., invalid filter params on the stream endpoint)? Yes — include at minimum one invalid `staffComponent` value to confirm the stream returns an error event and closes.
