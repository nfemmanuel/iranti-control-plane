## Problem Statement

The control plane needs read-focused API endpoints to power Memory Explorer, Instance Manager, and Diagnostics views. None of these endpoints exist today. Building any Phase 1 UI before the API surface is specified will cause misaligned implementation — either frontend and backend build to different assumptions, or scope expands mid-sprint when the shape of the data becomes clear. This spec defines the complete read API surface that Phase 1 will be built against.

## User Value

All frontend Phase 1 work depends on a stable read API. Specifying it in Phase 0 allows frontend_developer and backend_developer to work in parallel in Phase 1 — frontend builds to the spec, backend implements to the spec — without serial blocking. This is one of the highest-leverage Phase 0 outputs: it directly multiplies Phase 1 throughput.

## Scope

In:
- API spec for 7 endpoint groups:
  1. `knowledge_base` browsing — filter by entity, key, agent_id, date range; search by content; paginated
  2. Archive browsing — filter by entity, key, archived_at; paginated
  3. Entity detail — single entity view with full temporal history (all versions, all keys)
  4. `entity_relationships` query — relationships for a given entity, optionally filtered by type or related entity
  5. Instance and project metadata — aggregated instance list with bound projects and integration status
  6. Diagnostics/health summary — DB connection, service status, Staff operational state
  7. Staff event stream endpoint — real-time or near-real-time feed of Staff events
- For each endpoint group: path, HTTP method, query parameters, response schema (JSON), error states
- Each endpoint group explicitly mapped to one or more PRD Functional Requirements (FR1–FR5, FR8)
- Identification of which endpoints can be implemented against existing DB queries vs which require new backend logic
- Staff event stream endpoint (group 7) must reference the event schema from CP-T001
- Produce spec at `docs/specs/control-plane-api.md`

Out:
- Implementation of any endpoint
- Write endpoints (POST, PATCH, DELETE)
- Auth design beyond local-only access
- Rate limiting or caching design (note as considerations, but do not spec in detail)

## Dependencies

- CP-T001 must be complete and PM-approved before this ticket can finalize the Staff event stream endpoint (group 7). All other endpoint groups can be drafted while CP-T001 is in progress.
- Access to upstream Iranti DB schema to verify actual table shapes, column names, and index coverage.

## Acceptance Criteria

- [ ] All 7 endpoint groups are specified with: HTTP method, path, all query parameters (name, type, required/optional, description), full response schema (JSON with field names and types), and at least the 2 most likely error states per endpoint
- [ ] Each endpoint group explicitly references which PRD Functional Requirements it supports (FR1–FR5 and FR8 must all be covered by at least one endpoint)
- [ ] Spec explicitly identifies, for each endpoint, whether it can be implemented against existing Iranti DB queries or requires new backend logic to be written
- [ ] Staff event stream endpoint (group 7) references the event schema from CP-T001 — no independent event field definitions are introduced in this spec
- [ ] Spec addresses the transport decision for the event stream endpoint (SSE vs WebSocket vs polling) with a recommended option and documented tradeoffs
- [ ] Spec is concrete enough that backend_developer can implement all endpoints without additional design input from system_architect
- [ ] PM has reviewed and approved the spec before Phase 1 implementation begins

## Risks

- Upstream Iranti schema may have changed since the PRD was written; architect must verify actual table shapes against the running codebase rather than assuming PRD accuracy. Discrepancies must be flagged to the PM.
- Event stream endpoint design is coupled to the CP-T001 persistence decision — if CP-T001 chooses ephemeral in-memory storage, the stream endpoint design differs significantly from a DB-backed approach. Do not finalize group 7 until CP-T001 is settled.
- Pagination and filtering requirements for the knowledge_base can expand significantly if not scoped deliberately — spec should define a minimal viable filter set and explicitly defer more advanced search to a future phase.

## Open Questions

- Should the event stream use SSE (Server-Sent Events), WebSocket, or polling? SSE is simplest for a local-only deployment with no bidirectional need; WebSocket adds complexity but enables future interactivity. What are the tradeoffs for a local-only deployment?
- Should the API require any local auth token (e.g., a token generated at startup), or is local-only port binding sufficient for v1 security?
- Should entity relationships be a separate endpoint or a parameter on the entity detail endpoint? What does the upstream schema support?
