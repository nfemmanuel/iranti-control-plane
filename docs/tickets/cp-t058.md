# CP-T058 — UX Polish: Operator Guidance Labels (M4, M5, H8)

**Type:** Enhancement
**Priority:** P3
**Assigned:** frontend_developer
**Phase:** 3 (Wave 5)
**Status:** PM-ACCEPTED 2026-03-21 — minor type gap noted (non-blocking)
**Epic:** CP-E009 (Provider Manager), CP-E004 (Instance Manager)

## Summary

Three small UX gaps were identified in the cross-repo audit (2026-03-21) where the control plane shows limited read-only data but gives no guidance on how to change it. Operators are left stranded — they can see the state but not the action. This ticket closes all three as a single frontend-only polish pass.

| Audit ref | Location | Gap | Fix |
|-----------|----------|-----|-----|
| M4 | Provider Manager | No write path, no guidance on how to change providers | Add label: "To change providers or models, run `iranti setup` in your project directory." |
| M5 | Instance Manager — unreachable instance | Shows "Unreachable" but no remediation hint | Add: "Run `iranti run --instance <name>` to start this instance." |
| H8 | Instance Manager | `IRANTI_PROJECT_MODE` value not shown | Add `IRANTI_PROJECT_MODE` to the instance metadata display |

These are pure frontend changes. No backend work is needed — the `IRANTI_PROJECT_MODE` value is already included in instance metadata from CP-T011 (or derivable from the env file data the backend already returns).

## Acceptance Criteria

### AC-1 — M4: Provider Manager guidance label

In `src/client/src/components/providers/ProviderManager.tsx`, add a static informational note in the view — position it in the view header or below the active provider/model display:

> "Provider and model configuration is read-only. To change providers or models, run `iranti setup` in your project directory."

The note must:
- Use the `Informational` severity style from the Health Dashboard severity taxonomy (CP-T028) — blue-tinted or neutral, not a warning/error
- Render `iranti setup` in an inline code/monospace style consistent with the rest of the UI
- Be visible in both light and dark mode
- Not be dismissible (this is a persistent operator hint, not a notification)

### AC-2 — M5: Instance unreachable remediation hint

In the Instance Manager view (the instance list or instance detail card), when an instance status is `Unreachable`:
- Below the "Unreachable" status badge, add a hint: "To start this instance, run `iranti run --instance <name>` in your terminal."
- Replace `<name>` with the actual instance name from the instance record
- If the instance name is not available (null/unknown), show `iranti run` without the `--instance` flag
- The hint should render as small helper text (muted, slightly smaller font) below the badge — not a full alert panel

### AC-3 — H8: IRANTI_PROJECT_MODE in Instance Manager

In the instance metadata display (the expanded card or detail view for each instance):
- Add `IRANTI_PROJECT_MODE` as a displayed field, alongside existing env-derived fields like `IRANTI_PORT`, `DATABASE_URL`, etc.
- Label: "Project Mode"
- Value: the raw env var value (e.g., `isolated`, `shared`) or `—` if not set
- If the value is `isolated`: add a tooltip or helper note: "Each project gets its own isolated memory context."
- If the value is `shared`: add: "All projects share a single memory context."
- If not set: show `—` with no tooltip (default behavior is assumed by Iranti)

Check `src/server/routes/control-plane/health.ts` and `src/server/routes/control-plane/instances.ts` (or equivalent) to confirm whether `IRANTI_PROJECT_MODE` is already included in the backend response. If it is already returned, this is purely a frontend display addition. If it is not returned, add it to the backend instance/health serializer (this would make it a small backend change too — document in your report).

### AC-4 — TypeScript clean

`tsc --noEmit` passes with zero new errors. No `any` in new code.

### AC-5 — Visual consistency

All new UI elements must use Terminals palette CSS tokens. No hardcoded colors. Light mode and dark mode both visually reviewed.
