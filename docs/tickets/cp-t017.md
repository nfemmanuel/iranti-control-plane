## Problem Statement

No frontend application exists. Before any view-level UI work can begin, the shell chrome (navigation, layout, instance switcher) and visual token system must be built. Starting any view ticket without this foundation means each view engineer independently invents layout, color, and spacing conventions — producing an incoherent product. This ticket is the unblocking foundation for all other Phase 1 frontend work.

## User Value

A coherent, intentional visual system that feels like a precision operator tool (not a generic dashboard), with a navigation structure that lets operators move between all 6 Phase 1 surfaces instantly, and an always-visible instance context switcher that ensures operators always know which Iranti instance they are looking at.

## Scope

**In:**
- **Visual token system**: all CSS custom properties for Option B (Terminals) palette — both light and dark mode. All semantic roles:
  - Backgrounds: `--bg-canvas`, `--bg-surface`, `--bg-elevated`
  - Borders: `--border-subtle`, `--border-default`
  - Text: `--text-primary`, `--text-secondary`, `--text-muted`
  - Accent: `--accent-primary`, `--accent-hover`, `--accent-subtle`
  - Status: `--status-success`, `--status-warning`, `--status-error`
  - Staff palette: `--staff-librarian`, `--staff-attendant`, `--staff-archivist`, `--staff-resolutionist`
- **Light/dark mode toggle**: data-theme attribute on `<html>` element. Default: dark. Toggle persists in localStorage.
- **Shell layout**: outer chrome, sidebar (220px fixed), main content area (flex-1), chat panel slot (hidden, 380px reserved for Phase 2).
- **Sidebar navigation**: all 6 Phase 1 nav items with icons, labels, routes, and active state. Items: Overview (`/`), Memory (`/memory`), Archive (`/archive`), Activity (`/activity`), Instances (`/instances`), Health (`/health`). Settings (`/settings`) as Phase 2 placeholder (disabled state).
- **Instance context switcher**: rendered at top of sidebar. Shows current active instance name and port. Dropdown to switch instance (loads instance list from `/api/control-plane/instances`). State managed via React context accessible to all views.
- **Topbar**: per-section title area (48px height), positioned in main content area.
- **Activity Drawer slot**: 240px slide-up panel slot at the bottom of main content area for ambient Staff event tail (drawer component to be implemented in CP-T014, slot reserved here).
- **Phase 2 chat panel slot**: hidden DOM element (width 0, `display: none`) — no implementation, just structural reservation.
- **Base typography**: Inter font stack (Option B: system-ui, -apple-system, etc. — no web font load). Monospace: Cascadia Code / Fira Code fallback. Tabular numbers enabled.
- **Density system**: 4px base unit, 36px table rows, 13px data font size, 32px sidebar nav items, 16px section padding.
- **Token documentation**: produce `docs/specs/visual-tokens.md` — complete reference of all CSS custom property names, their roles, and both light/dark hex values for Option B. This is the reference doc for all other frontend work.
- **React Router setup**: root layout route with persistent shell, child routes for each section. Hash routing vs clean paths — use clean paths (Express catch-all for SPA).
- **Vite + React 18 project bootstrap**: initial project setup at correct path within the control plane codebase. TypeScript, CSS modules, Vite config.
- **Radix UI dependency installed**: confirm package installed and usable for downstream view tickets.

**Out:**
- Individual view components (those are CP-T013–CP-T016)
- Animation system (Phase 2)
- Settings page content (Phase 2)
- Chat panel implementation (Phase 2)

- [ ] `[data-theme="dark"]` and `[data-theme="light"]` CSS scoping works — toggling `data-theme` attribute on `<html>` switches all tokens correctly. Default is dark mode.
- [ ] Light/dark mode toggle works from the UI. State persists in `localStorage` across page reloads.
- [ ] Shell layout matches CP-T004 wireframe (Section 1.2): sidebar (220px) + main content area + hidden chat panel slot.
- [ ] Sidebar renders all 6 Phase 1 nav items with correct icons, labels, and routes. Settings item rendered as disabled placeholder (Phase 2).
- [ ] Active nav item has correct active state styling per Option B tokens (accent primary background/border).
- [ ] Instance context switcher renders at top of sidebar, shows current instance name (or "No instance" if not yet loaded). Fetches from `/api/control-plane/instances` on mount. Instance switching via dropdown updates shared React context.
- [ ] Instance context accessible via React context to all child route views (CP-T013–CP-T016 must be able to read current instance from this context).
- [ ] Activity Drawer slot is present in the DOM structure at the bottom of the main content area. The drawer component itself (with Staff event content) will be wired in by CP-T014, but the slot/container must exist here.
- [ ] Topbar area (48px, section title) present in each route layout. Section title populated from route.
- [ ] React Router v6 root layout route renders shell chrome once; only main content area re-renders on navigation.
- [ ] Clean path routing: all routes use `/memory`, `/archive`, etc. (not `/#/memory`). Vite dev server and Express production serve handle SPA fallback.
- [ ] Token documentation `docs/specs/visual-tokens.md` exists with all 18 CSS custom property names, their semantic roles, and Option B hex values for light and dark mode. Format: table with columns: token name, semantic role, light hex, dark hex.
- [ ] React Query, Radix UI, React Router v6 all installed and importable. No unresolved peer dependency warnings.
- [ ] TypeScript compiles without errors. No `any` types in shell/layout/token files.

## Risks

- Visual token system is the foundation all other tickets build on — any token naming inconsistency discovered after CP-T013–CP-T016 begin will be expensive to fix. Spend extra time on the `docs/specs/visual-tokens.md` token reference doc to ensure names are finalized before downstream work begins.
- Instance context switcher depends on `/api/control-plane/instances` — which is implemented in CP-T011. If CP-T011 is not ready when CP-T017 is being built, implement with a mock/stub instance context first. Document the stub clearly and wire to real API when CP-T011 is complete.
- React Router v6 nested routes with shared layout — ensure the root layout does not unmount on navigation (use Outlet correctly). Common mistake: nesting routes incorrectly causes the shell to re-render on every navigation.
