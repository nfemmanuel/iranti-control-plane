## Problem Statement

The Phase 1 Staff Activity Stream (CP-T014) is implemented using a database polling adapter: the backend periodically queries the `knowledge_base` and `archive` tables for recent changes and surfaces them as Staff events. This approach has two fundamental limitations:

1. **Coverage gaps.** The polling adapter can only infer events from table state — it cannot observe Attendant handshakes, Attendant reconvene calls, Resolutionist decisions, or any action that does not produce a directly readable database row. These are four of the most valuable Staff events for operator debugging, and they are completely absent from the Phase 1 stream.

2. **Latency.** The polling interval is approximately 2 seconds (required to avoid excessive database load). For time-sensitive debugging — watching the Librarian process a write in real time, seeing a conflict detected the instant it occurs — 2 seconds is a significant lag.

Phase 1 made the correct tradeoff: ship a working but limited stream rather than block Phase 1 on upstream changes. Phase 2 must close the gap. The correct fix is to inject a structured event emitter interface (`IStaffEventEmitter`) directly into each of the four Staff components at their action call sites, replacing polling with a push-based model.

**This ticket requires proposing and coordinating upstream changes to the Iranti core package.** It cannot be implemented entirely within this repository. The system_architect must design the injection interface and produce an upstream PR proposal. The PM must approve the upstream scope before any implementation begins.

## User Value

Operators watching the Staff Activity Stream see events from all four Staff components — including Attendant handshakes (previously invisible), Resolutionist resolution decisions (previously invisible), Archivist archive cycles, and Librarian writes — with latency under 200ms from the time the Staff action occurs to the time it appears in the stream. The stream becomes a complete real-time window into Iranti's internal reasoning, not a partial DB-change feed with a 2-second lag. This is the differentiating capability described in the competitor analysis: no other product can show this.

## Scope

**In:**

**Part 1: Interface Design (system_architect)**
- Define the `IStaffEventEmitter` interface in a location that Iranti Staff components can depend on (likely the Iranti core package). Interface must be injectable (dependency injection, not a global singleton) so tests can mock it and the control plane can provide a real implementation.
- Minimum interface:
  ```typescript
  interface IStaffEventEmitter {
    emit(event: StaffEvent): void;
  }
  // StaffEvent type from CP-T001 spec
  ```
- The interface must be synchronous from the caller's perspective (the Staff component calls `emit()` and returns immediately — the emitter handles async delivery internally). Staff components must not be slowed by the event system.
- Define the binding pattern: how does an `IStaffEventEmitter` implementation get injected into each Staff component? Options: constructor injection, a static setter, a context object passed through the call chain. The system_architect must compare options and recommend one with rationale.
- Produce an upstream PR proposal document at `docs/specs/staff-emitter-injection.md` covering: the interface definition, the binding pattern recommendation, which files in the Iranti core package must change, the expected API impact (additive — this must be backward-compatible), and the rollout plan.

**Part 2: Staff Component Coverage (system_architect + backend_developer)**
The upstream PR proposal must specify injection points for all four Staff components with all relevant action types from the CP-T001 spec:

- **Librarian:**
  - `write_created` — new fact written to knowledge_base
  - `write_replaced` — existing fact superseded by new write
  - `write_escalated` — write triggered a conflict escalation
  - `write_rejected` — write rejected (reason included)
  - `write_deduplicated` — write identified as duplicate and skipped

- **Attendant:**
  - `handshake_completed` — agent handshake processed (agent_id, task summary included)
  - `attend_completed` — attend call processed (entity count returned)
  - `observe_completed` — observe call processed
  - `reconvene_completed` — reconvene call processed
  - `context_loaded` — memory context assembled for an agent (entity count, token estimate included)

- **Archivist:**
  - `archive_created` — fact moved to archive
  - `decay_applied` — confidence decay applied to a fact
  - `escalation_consumed` — resolved escalation consumed from queue
  - `cleanup_cycle_completed` — periodic cleanup cycle complete (count of archived facts)

- **Resolutionist:**
  - `resolution_decision_made` — Resolutionist made a resolution decision (decision type, entity, reason)
  - `escalation_created` — new escalation created (entity, conflict reason)
  - `escalation_resolved` — escalation resolved by operator (resolution type)
  - `auto_resolution_applied` — Resolutionist applied an automatic resolution (confidence-based or rule-based)

Each event emission point must include the full `StaffEvent` fields from CP-T001 spec: Staff component, action type, entity (when applicable), key (when applicable), agent_id, source, reason/note, timestamp, event level.

**Part 3: Control Plane Implementation (backend_developer)**
- Implement the production `IStaffEventEmitter` that connects to the control plane's event delivery system:
  - On `emit()`: append the event to the in-memory ring buffer (max 2,000 events).
  - On `emit()`: broadcast the event to all active SSE connections (`/api/control-plane/events`).
- Update the backend's `IStaffEventEmitter` injection: when Iranti starts with the control plane enabled, inject the production emitter into all four Staff components.
- Remove the polling adapter from CP-T014 once the native emitter is confirmed working for all four components. Keep as a fallback if the `IStaffEventEmitter` is not injected (for running without the control plane).
- Measure and confirm: event latency from Staff action to SSE delivery must be below 200ms on a local machine. Add a latency measurement to the event object: `emittedAt` (timestamp at `emit()` call) and `deliveredAt` (timestamp at SSE send). Log the delta for the first 30 days after rollout.

**Part 4: Upstream Coordination**
- The system_architect must flag this ticket to the PM as an upstream PR proposal.
- The PM must approve the upstream scope before any changes to the Iranti core package are made.
- The system_architect must determine whether the upstream Iranti maintainer accepts external PRs or whether this is an internal change. If external: produce a PR description and rationale suitable for an open-source contribution.
- If upstream changes are rejected or blocked: document a fallback strategy (enhanced polling with lower interval + log file parsing as a partial bridge) and escalate to PM.

**Out:**
- Changes to the Librarian, Attendant, Archivist, or Resolutionist business logic (the emitter is a pure side-channel — it must not affect behavior)
- Event persistence to a database table (the Phase 1 in-memory ring buffer is sufficient for Phase 2; database persistence is Phase 3 if required)
- UI changes (Phase 1 Staff Activity Stream UI is unchanged; it will automatically show richer events once the emitter is live)
- Removing the polling adapter entirely before all 4 components are confirmed working with native emission (keep as fallback)

## Risks

- **Upstream Iranti maintainer may reject the proposed changes.** If the PR is rejected, the fallback is enhanced polling (500ms interval on a dedicated events table if one can be added) or structured log file parsing. The system_architect must design the fallback in parallel with the upstream proposal, not after rejection.
- **`IStaffEventEmitter` as a synchronous interface introduces risk if the emitter implementation is slow.** The interface must be synchronous from the caller's perspective, but the implementation must be non-blocking (async delivery after a synchronous enqueue). A slow emitter that blocks Staff operations would degrade Iranti's core performance — this is a critical correctness requirement, not a nice-to-have.
- **Test coverage for event emission.** The Staff components currently may not have unit tests that verify event emission. Adding emission to untested code paths is risky. The system_architect must ensure the upstream PR includes at minimum: one unit test per Staff component confirming that the emitter is called with the expected event type for each action.
- **Backward compatibility.** The `IStaffEventEmitter` injection must be optional (defaulting to a no-op emitter if none is provided). Existing Iranti users who are not using the control plane must see zero behavior change. The no-op emitter must be the default in all constructors.

## Open Questions

- Should `IStaffEventEmitter` live in the Iranti core package's public API (importable by any consumer), or in a separate `@iranti/events` package? Recommendation: in the core package initially (simpler); extract to a separate package in Phase 3 if event bus consumers multiply.
- Should the upstream PR be submitted to the main Iranti repository, or implemented as a fork/patch for the control plane's local use? Recommendation: upstream PR is strongly preferred for maintainability. A local fork/patch creates a maintenance burden that will compound. If the upstream PR is blocked, escalate to PM before accepting a fork.
- What is the current test coverage for the four Staff components in the Iranti codebase? This determines how much test infrastructure is needed in the upstream PR. The system_architect must assess this during the spike.
