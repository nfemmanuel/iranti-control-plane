## Problem Statement

Iranti Chat exists as a CLI surface (`iranti chat`) but is not accessible from within the control plane operator workspace. Operators who want to inspect memory, write facts, test retrieval, or run slash commands must leave the control plane entirely and switch to a terminal — breaking the single-surface operator experience the PRD establishes as a core product principle (ER3: Minimal Context Switching, FR6: Embedded Chat).

This ticket implements the embedded chat panel: a collapsible right-side panel within the existing control plane shell (the slot reserved in CP-T017) that gives operators full Iranti Chat capability without navigating away from any Phase 1 view.

## User Value

An operator can be browsing the Memory Explorer, notice an unexpected fact, and immediately open the chat panel to query Iranti, write a correction, or test retrieval — all without leaving the view. The operator session is preserved, the context is intact, and the workflow is continuous. This is the difference between a monitoring tool and an operator workspace.

## Scope

**In:**
- Chat panel rendered in the Phase 2 panel slot reserved by CP-T017 (right side, 380px wide, collapsible via toggle button in the shell topbar or sidebar)
- Panel open/close state persists in `localStorage`
- Agent ID selector: a dropdown at the top of the panel allowing the operator to set the `agent_id` for the session. Defaults to the value in `.env.iranti` `IRANTI_AGENT_ID` if set; otherwise defaults to `operator`. Options are free-text editable (operator may type a custom agent ID).
- Provider/model selector: a dropdown below the agent ID selector showing available configured providers and model options. Defaults to the current default provider. Changes apply to the current session only — no global config mutation from this panel.
- Conversation history for the session: all messages in the current panel session are displayed in a scrollable message list. History is in-memory only — it does not persist across page reloads or tab closes unless the operator explicitly saves it.
- Message input: a multiline textarea at the bottom of the panel (auto-grows, max 6 lines before scrolling). `Enter` sends, `Shift+Enter` inserts newline. A send button for mouse users.
- Slash command palette: pressing `/` as the first character in the message input opens an inline autocomplete panel showing available slash commands. Commands sourced from the Iranti Chat slash command registry. Arrow keys to navigate, `Enter` to select, `Escape` to dismiss without selection.
- Retrieved memory blocks displayed inline: when a chat response includes retrieved facts from Iranti memory, display them as structured cards below the assistant message. Each card shows: entity (type/id), key, summary, confidence (as a numeric badge), and source. Cards are visually distinct from the prose message.
- "View in Memory Explorer" jump link on each retrieved memory card: clicking navigates to the entity detail page for that fact in the Memory Explorer. The chat panel stays open; only the main content area navigates.
- Session clear button: a "Clear" button (or trash icon) in the panel header clears the conversation history for the current session. Requires a single confirmation click (inline confirmation, not a modal).
- Loading state for in-flight messages: while waiting for a response, the last user message shows a spinner/pulse indicator inline. The input is disabled during this state. A "Cancel" button appears that aborts the in-flight request.

**Out:**
- Persistent cross-session conversation history (Phase 3)
- Exporting conversation transcripts (Phase 3)
- Multi-session tabs or conversation switching within the panel (Phase 3)
- Voice input (out of scope)
- Streaming response rendering (desirable Phase 2 stretch goal — wire up if the backend supports streaming; otherwise full-response rendering is acceptable)

## Dependencies

- CP-T017 (Visual System and Shell Chrome) — the Phase 2 chat panel slot must be present in the DOM. CP-T020 cannot start until CP-T017 is accepted.
- Phase 1 API endpoints fully implemented and accepted: the backend_developer must confirm that `/api/control-plane/health`, `/api/control-plane/instances`, and other Phase 1 endpoints are stable before building the chat session endpoint.
- Iranti Chat API or subprocess integration: the backend must determine the correct integration path. Two options:
  - **Option A (preferred):** Call Iranti Chat programmatically via the existing Iranti Node SDK or Chat API if one exists — no subprocess needed.
  - **Option B (fallback):** Spawn an `iranti chat` subprocess with structured stdin/stdout wrapping and translate to the chat panel's session model.
  - The backend_developer must investigate both options and choose one at task pickup. If neither is feasible without upstream changes, escalate to PM before proceeding.
- PM decision on supported providers list (CP-T022 dependency — provider/model selector in the chat panel should align with the provider list established in CP-T022 if that ticket is done first; if not, use detected env vars as the source of truth).

## Acceptance Criteria

- [ ] The chat panel opens and closes from anywhere in the control plane (any Phase 1 view) without navigating away from the current view. The main content area remains fully functional while the panel is open.
- [ ] Panel open/close state persists in `localStorage` across page reloads. If the user closed the panel, it stays closed on reload. If they left it open, it stays open.
- [ ] Agent ID is selectable per session. The selected agent ID is sent with every message to Iranti Chat. The dropdown defaults to `IRANTI_AGENT_ID` from the active instance's env if present.
- [ ] Provider/model selector shows at least the current default provider. Changing the selection changes the model used for the remainder of the session. The change is session-scoped — it does not modify the instance's env config.
- [ ] Slash commands are discoverable via `/` prefix autocomplete in the message input. At least the top 5 most common Iranti slash commands appear in the palette. Selecting a command from the palette inserts it into the input.
- [ ] A sent message receives a response displayed in the conversation thread. The full response is rendered (streaming is a stretch goal but not required for acceptance).
- [ ] Retrieved memory blocks in responses are displayed as structured cards with: entity (type/id), key, summary, confidence badge, and source. Cards are visually distinct from prose text.
- [ ] "View in Memory Explorer" link on each retrieved memory card navigates the main content area to that entity's detail page. The chat panel remains open throughout the navigation.
- [ ] Session history persists until the operator explicitly clicks Clear or closes/reloads the tab. Scrolling back through older messages in the session works correctly.
- [ ] Clear button clears the conversation history after inline confirmation. The panel remains open and ready for a new message.
- [ ] Loading/in-flight state: the input is disabled while a message is in-flight. A cancel mechanism exists that aborts the pending request.
- [ ] TypeScript compiles without errors. No `any` types in the chat panel component tree.

## Risks

- **Iranti Chat integration path is uncertain.** The backend must determine whether a programmatic API or subprocess approach is needed. If neither works without upstream changes to Iranti, this ticket stalls. The backend_developer must investigate this as the first step and escalate to PM within 2 days of task pickup.
- **Panel layout impact on narrow viewports.** At 380px panel + 220px sidebar, the main content area is reduced to `calc(100vw - 600px)` on standard 1440px displays. This is acceptable. On 1280px or narrower displays, the panel may need to overlay rather than push the main content. Define the breakpoint at task start (recommended: panel overlays at < 1280px viewport width).
- **Streaming response alignment.** If the Iranti Chat API returns streaming chunks, the chat panel must handle partial response rendering. If the API returns a single completed response, partial rendering is not needed. Do not assume streaming is available — verify during task pickup.
- **Memory block parsing.** The chat response format must include structured metadata for retrieved memory blocks, not just prose. If the API does not return structured memory blocks separately from prose, the frontend cannot display them as cards. The backend_developer must confirm the response structure before the frontend_developer begins implementing memory card rendering.

## Open Questions

- Should the slash command palette be populated from a static list (hardcoded known commands) or from a dynamic endpoint that returns the available commands for the current Iranti instance? Dynamic is better but requires a new endpoint. Decide at task pickup.
- Should the chat panel retain its conversation history when the user switches between Phase 1 views, or should it reset on navigation? Decision: retain history (ER3: Minimal Context Switching). The session is per-tab, not per-view.
- What is the maximum message length the Iranti Chat API accepts? The input must enforce this limit with a character counter and soft/hard limits.
