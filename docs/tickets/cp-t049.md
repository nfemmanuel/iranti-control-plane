# CP-T049 — Archivist Decision Transparency and Operator Review

**Type:** Feature
**Priority:** P2
**Phase:** 3
**Epic:** CP-E008 (Conflict and Escalation Review — extended to cover Archivist decisions)
**Assigned:** Unassigned
**Status:** Open
**Created:** 2026-03-20

---

## Summary

Extend the Archive Explorer to give operators visibility into *why* specific facts were archived by the Archivist, and provide a limited operator intervention capability for reviewing archival decisions that appear incorrect. This is distinct from the Resolutionist's conflict escalation workflow (CP-T021): CP-T021 handles contradictory facts; this ticket handles the Archivist's supersession, decay, and conflict_resolved archival decisions.

---

## Background and PM Assessment

The Archive Explorer (CP-T027) is already implemented. Operators can browse archived facts, filter by `archivedReason` (superseded, decay, conflict_resolved) and `resolutionState`, and expand individual rows to see `supersededBy` and `validUntil` fields.

**What is already covered:**

- Viewing archived facts with all metadata fields
- Filtering by archival reason and resolution state
- Seeing which fact superseded a given archived fact (via `supersededBy` field)
- Seeing the `validUntil` timestamp for decayed facts

**What is NOT covered and is the subject of this ticket:**

1. **Archivist decision rationale:** The Archive Explorer shows *that* a fact was archived and *what reason code* was used, but not the full Archivist reasoning — e.g., "why was this fact considered superseded?" or "what was the confidence decay formula that expired this fact?". For developers debugging Iranti's behavior, this reasoning is valuable.

2. **Operator intervention in Archivist decisions:** There is currently no way for an operator to:
   - Flag an archived fact as "incorrectly archived" for later review
   - Request re-evaluation of a decay decision
   - Restore a superseded fact to active status (with appropriate audit trail)
   - Inspect the Archivist's processing log for a specific fact lifecycle

3. **Archivist processing log:** The Activity Stream shows Archivist events in real time. But there is no persistent, queryable Archivist processing log tied to specific facts. Operators cannot ask "show me everything the Archivist did to `entity/key` over its lifetime."

**User signal:** The user asked about "archivist resolution handling" — indicating they expect to be able to inspect and potentially intervene in archival decisions, not just browse the resulting archive. This is a legitimate operator need, especially when debugging unexpected agent behavior caused by facts being archived prematurely or incorrectly.

## Scope

**In scope:**

### 1. Archivist Event Log per Fact

In the expanded row view of the Archive Explorer (the expanded panel that appears when clicking a fact row), add an "Archivist History" section that shows the Archivist processing events for that fact's lifecycle. Source: `staff_events` table filtered by `componentName = "Archivist"` and `entityType/entityId/key` matching the fact.

- Shows: timestamp, event type (decay_check, supersession_detected, archived, etc.), payload summary
- Empty state: "No Archivist events recorded for this fact. Events require CP-T025 native emitter injection."
- The section is rendered as a compact timeline (most recent first)

### 2. "Flag for Review" Action

Add a "Flag for Review" button to the expanded Archive fact row. Clicking it:
- Sets a `flagged_for_review` metadata field on the archive record (stored in a new `archive_flags` table or a JSONB field on the existing archive row — backend implementation choice, see Open Questions)
- Shows a brief text input for the operator to note why they are flagging it ("Possible premature decay", "Confidence decay parameters may be misconfigured", etc.)
- The flag is visible in the Archive Explorer as a small "⚑ Flagged" indicator on the row

Flagged facts can be filtered using a "Flagged" filter in the Archive Explorer filter bar.

### 3. Flagged Facts Review Queue

A secondary view accessible from the Archive Explorer (sub-tab or toggle): "Flagged for Review". Lists all facts that have been flagged, with:
- Entity, key, archivedReason, flag note, flagged at timestamp
- A "Clear Flag" button to dismiss after review
- A "Restore Fact" button (see below)

### 4. Fact Restoration (operator override)

From the Flagged Facts view, an operator can trigger a "Restore Fact" action on a flagged archived fact. This:
- Writes the archived fact's `valueRaw` and `valueSummary` back to the `knowledge_base` table as a new active fact (with `createdBy: "control_plane_operator"`, `source: "operator_restore"`)
- Creates a new archive record for the previously-active fact (if any) with `archivedReason: "operator_superseded"`
- Logs the restore action to `staff_events` with `componentName: "Archivist"`, `eventType: "fact_restored_by_operator"`, payload includes original archive row ID and operator note
- Requires a confirmation step (cannot be undone without another restore operation)
- Returns the restored fact's new `knowledge_base` ID

**Important constraint:** Restoration is an operator override. It does not re-run the Archivist's logic. The Archivist may re-archive the fact on its next cycle if the same conditions apply. This is documented in the confirmation dialog.

### 5. Archivist Statistics Panel (stretch goal)

A small statistics panel in the Archive Explorer header (or a dedicated section):
- Total archived facts by reason (pie chart or count badges): superseded N, decay N, conflict_resolved N
- Facts flagged for review: N
- Archivist processing rate: facts archived per hour (last 24h) — computed from `staff_events` if CP-T025 is available, else from archive row `archivedAt` timestamps

**Out of scope:**

- Bulk restoration of multiple facts at once (Phase 3+ follow-on)
- Modifying the Archivist's decay parameters from the UI (configuration management, separate ticket)
- Suppressing or disabling the Archivist for specific entities (out of scope — Iranti configuration)
- Viewing the Archivist's source code or algorithm explanation (documentation, not a UI feature)
- Reversing a conflict_resolved archive (conflict re-opening) — this requires Resolutionist involvement and is explicitly out of scope; direct CP-T021 instead

## Acceptance Criteria

1. **Archivist History section** visible in expanded Archive fact rows. Shows Archivist `staff_events` for that fact's entity/key if CP-T025 is active; shows empty state with clear message if not.

2. **Flag for Review button** present in expanded Archive fact rows. Clicking opens a brief text input and a confirm button. Flagged facts show a visible indicator in the list row.

3. **Flagged Facts filter** added to the Archive Explorer filter bar. Selecting it filters to flagged facts only.

4. **Flagged Facts Review Queue** accessible from the Archive Explorer. Shows all flagged facts with entity, key, archivedReason, flag note, flagged at, and action buttons.

5. **Restore Fact action** implemented in the Flagged Facts view. Writes a new `knowledge_base` row, archives the currently-active fact if present, logs to `staff_events`. Requires confirmation. Shows success/failure feedback inline.

6. **Confirmation dialog** for Restore Fact explicitly states: "This is an operator override. The Archivist may re-archive this fact on its next processing cycle if the same conditions apply."

7. **Audit trail:** The `staff_events` log entry for `fact_restored_by_operator` includes: original archive row ID, the entity/key restored, the operator's flag note, and a timestamp.

8. **TypeScript compiles without errors.** No `any` in new component tree.

9. **Both light and dark mode verified** for all new UI elements.

## Dependencies

- CP-T027 (Archive Explorer) — must be complete. Already accepted.
- CP-T010 (archive backend endpoints) — already complete.
- CP-T025 (native Staff emitter) — required for the Archivist History section to populate with real data. The section renders with an empty state if CP-T025 is not active, so this is not a hard blocker.
- Backend: new endpoint `POST /api/control-plane/archive/:id/restore` and a flagging mechanism (endpoint + data model) are required. These must be implemented before the frontend can wire the actions.

---

## Open Questions

**OQ-1:** Should the flag be stored as a new `archive_flags` table (clean relational model, requires migration) or as a JSONB `metadata` field on the existing `archive` table (simpler, no migration needed)? PM preference: defer to backend_developer — either is acceptable, document the choice.

**OQ-2:** Should the Restore Fact action be available without flagging first (i.e., restore from any archived fact), or only from the Flagged review queue? PM recommendation: restrict to flagged facts for the initial release — this prevents accidental restoration of facts the operator hasn't explicitly reviewed.

**OQ-3:** What happens if the fact's entity/key is currently active in `knowledge_base` when a restore is attempted? The backend must handle this case: either refuse (return 409 Conflict with a clear message), or allow the restore and supersede the current active fact. PM recommendation: supersede the current active fact with an `operator_superseded` archive record, and proceed with the restore. Show this in the confirmation dialog.

---

## Implementation Notes

- The "Flag for Review" and "Restore Fact" features have a meaningful write impact on the Iranti database. The backend must require a `?confirm=true` query parameter on the restore endpoint, consistent with the pattern established in CP-T033.
- The Archivist History timeline in the expanded row should reuse the timeline component design patterns from the Temporal History view (CP-T036) for visual consistency.
- The statistics panel (stretch goal) should use the same compact metric badge component used in the Health Dashboard.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Archivist re-archives restored facts immediately | High | Medium | Document this in the confirmation dialog; add note in the Flagged review queue |
| Restore creates inconsistent state if multiple facts exist for same entity/key | Medium | High | Backend validation: check for active KB row before writing; handle 409 per OQ-3 |
| Flag data lost if archive row is garbage collected | Low | Low | Flags should be stored relationally with cascade delete; clarify in OQ-1 |
| CP-T025 not available when this ticket ships | High | Low | Archivist History section renders gracefully with empty state |
