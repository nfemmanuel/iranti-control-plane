## Problem Statement

The Health dashboard is the first screen a new user sees after installing Iranti. Its job is to answer one question clearly: "Is Iranti working right now?"

Without a well-calibrated severity taxonomy, the Health dashboard will show amber warnings for conditions that are expected, harmless, and not actionable — and new users will interpret those warnings as problems requiring immediate attention. Specific conditions that will trigger this pattern in almost every installation:

- **Provider balance unavailable:** Anthropic does not expose remaining credits via a stable public API. The control plane cannot check balance for Anthropic-backed instances. Without correct taxonomy, this shows as "⚠ Balance check failed" — creating a false alarm for every single Anthropic user.
- **No escalations directory:** A clean install with no conflicts has no escalations directory yet. This is the expected state for most new users. Without correct taxonomy, this shows as "⚠ No escalations directory" — implying a setup problem.
- **Version behind latest:** The user's installed version is one or two minor versions behind the latest release. This is nearly always true. Without correct taxonomy, this shows as "⚠ Runtime version 0.12.1 — latest is 0.12.3" — suggesting urgency for a non-breaking update.

The user's mental model when seeing three amber warnings on the first screen: "Something is wrong. I need to fix this before I can use Iranti." They will spend 30 minutes trying to resolve non-problems. When nothing changes, they lose confidence in both the product and their own setup. This is a high-severity, high-likelihood failure mode.

The Health dashboard is also the success gate for v0.1.0 Criterion 1: "A new user can open the control plane, land on Health, and understand what each check means — without consulting documentation." That criterion cannot pass if expected conditions produce amber warnings.

## Scope

**In:**

Implement a four-tier severity taxonomy for all Health dashboard checks:

### Severity Tiers

**Critical (red)**
- Definition: Iranti cannot function. User must act before Iranti will work.
- Examples: database unreachable, no provider configured at all (no API key of any kind), runtime process not running.
- Visual: red icon or badge, prominent placement in the summary header.
- Required: at least one actionable next step (specific CLI command, link to docs) for every Critical item.

**Warning (amber)**
- Definition: Iranti is functional, but a specific capability may be degraded or a non-urgent issue exists.
- Examples: provider balance API unavailable (capability-specific, not a global failure — but only if the provider actually supports balance checking and the check returned an error, not if the provider simply doesn't expose it), a project binding has a missing `.mcp.json` that was previously present.
- Visual: amber icon, shown below the summary header.
- Required: one-line plain-English explanation of what is degraded and why.

**Informational (blue/grey)**
- Definition: Expected state for a standard installation. No action required. Not a problem.
- Examples:
  - Provider balance unavailable because the provider (Anthropic) does not expose balance via API — label explicitly as "Balance visibility not supported by this provider. This is expected."
  - No escalations directory — label as "No conflicts have occurred yet. This directory is created on first escalation."
  - Version behind latest (minor version) — label as "Version 0.12.1 installed. 0.12.3 available. Non-breaking update."
- Visual: blue/grey icon, shown in a collapsible or visually subordinate section (below Critical and Warning).
- Required: one-line explanation that normalizes the state ("this is expected").

**Healthy (green)**
- Definition: All checks passed. No action required.
- Visual: green icon per check.
- Summary header: if no Critical or Warning items, show "Operational" as the top-level status label — not "3 informational items."

### Summary Header Rule

The top-level status in the dashboard header must reflect the most severe active state:
- Any Critical item → show "Action Required" in red
- Warning items only (no Critical) → show "Operational with warnings" in amber
- Informational items only (no Warning or Critical) → show "Operational" in green
- No items of concern → show "Healthy" in green

This prevents the user from interpreting "3 informational items" as a degraded state.

### Per-Item Requirements

Every health check item — regardless of severity — must display:
1. A one-line plain-English description of what was checked and what was found
2. A severity icon consistent with the taxonomy above
3. For Critical items: at least one specific actionable next step
4. For Warning items: one-line explanation of what is degraded
5. For Informational items: one-line normalization statement ("this is expected because...")

### Specific Check Classifications (confirmed)

| Check | Correct Tier | Copy guidance |
|-------|-------------|---------------|
| Database unreachable | Critical | "Cannot connect to PostgreSQL at [host]. Iranti requires a database connection to function." |
| No provider API key configured | Critical | "No LLM provider is configured. Iranti cannot process writes without a provider key." |
| Provider balance unavailable (Anthropic or any provider that does not expose balance) | Informational | "Balance visibility not supported by [provider]. This is expected — [provider] does not expose credits via API." |
| Provider balance low (if provider DOES expose it and balance is low) | Warning | "Remaining credits low: [amount]. Consider topping up to avoid write failures." |
| No escalations directory | Informational | "No conflicts have occurred yet. The escalations directory is created automatically on first conflict." |
| Runtime version behind latest (minor) | Informational | "[version] installed. [latest] available. Update is non-breaking." |
| Runtime version behind latest (major) | Warning | "[version] installed. Major update [latest] available. Review release notes before updating." |
| Vector backend unreachable | Critical | "Cannot reach vector backend at [host]. Semantic retrieval will fail." |
| Claude integration (.mcp.json) missing | Warning | "Claude integration file not found for project [name]. Slash command memory access may not work." |
| Claude integration (.mcp.json) present | Healthy | — |

**Out:**

- Adding new health checks not already in Phase 1 scope (separate ticket if needed).
- Implementing auto-remediation actions from health checks (Phase 2).
- Per-user configuration of which informational items to suppress.

## Acceptance Criteria

- [ ] The four-tier taxonomy (Critical / Warning / Informational / Healthy) is implemented and visually distinct for all four tiers.
- [ ] The summary header shows "Operational" (green) when no Critical or Warning items are present, even if Informational items exist.
- [ ] Provider balance unavailable for Anthropic (or any provider that does not support balance API) shows as Informational with normalization copy — not as Warning or "Balance check failed."
- [ ] "No escalations directory" shows as Informational with normalization copy — not as Warning.
- [ ] "Version behind latest" (minor) shows as Informational — not as Warning.
- [ ] Every Critical item includes at least one specific actionable next step.
- [ ] Every Informational item includes a normalization statement.
- [ ] The v0.1.0 success Criterion 1 user test can pass: a new user correctly identifies Iranti as operational and does not treat Informational items as blocking problems.
- [ ] All severity tiers are visually distinguishable in both light and dark mode, including for users with color vision deficiency (do not rely on red/green color alone — use icons or text labels as additional differentiators).
- [ ] TypeScript compiles without errors.

## Dependencies

- Backend health check endpoint must return severity metadata (or the frontend must classify severity based on check type and result). Confirm with backend_developer which approach is in scope.

## Risks

- **Taxonomy requires backend cooperation:** If the backend health endpoint returns raw check results without severity labels, the frontend must implement classification logic. This is acceptable but must be explicit — classification rules must be documented and testable, not implicit.
- **New checks added post-taxonomy:** Any new health check added after this ticket ships must be classified using the taxonomy. Document this as a contribution rule for the health check system.
