## Problem Statement

The control plane is approaching design partner handoff for v0.1.0 (pending CP-D001 hold lift + QA regression). Two documentation artifacts are missing that design partners will need immediately:

1. **A v0.1.0 Release Notes document** — what shipped, what changed, what version it is, and what phase it represents. Design partners need a stable reference point for what they are testing.

2. **A Known Issues document** — an honest, up-to-date list of confirmed bugs and limitations that design partners should expect to encounter. Without this, partners will spend session time filing known bugs instead of providing new product feedback.

These were flagged by the technical_writer during the Phase 2 docs pass (2026-03-20) and confirmed as required by the PM before handoff.

---

## Scope

### Deliverable 1: `docs/reference/v010-release-notes.md`

Structure:
- **What is v0.1.0** — one paragraph, plain English, what this release represents (Phase 1 complete + CP-T036, first installable release)
- **What's included** — table of all Phase 1 views and features with their ticket IDs and brief descriptions
- **What's not included** — explicit list of Phase 2 features (reference the Phase 2 table in getting-started.md)
- **How to install** — brief pointer to getting-started.md
- **Known issues** — pointer to the Known Issues doc (Deliverable 2)
- **Version** — v0.1.0, date 2026-03-20

### Deliverable 2: `docs/reference/known-issues.md`

This document must be accurate, complete, and honest. No glossing over known bugs. Each issue must have:
- Issue ID (KI-001 through KI-N)
- Title
- Affected view(s)
- Severity (P0 / P1 / P2)
- Description — plain English, what the user will see
- Workaround (if any)
- Status (fixed / in-fix / known, no workaround)
- Ticket reference (if tracked)
