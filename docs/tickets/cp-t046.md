## Background

CP-T034 delivered the Provider Status section inside the Health Dashboard: per-provider cards with reachability, masked key, and model list. Three AC items from CP-T034 were explicitly deferred due to the absence of live balance APIs at the time of implementation:

1. A dedicated `/providers` route (Provider Manager as a standalone view with provider list and detail panel)
2. Configurable warning thresholds (per provider, persists across reloads)
3. Health dashboard warning item for triggered warning thresholds

Additionally, two provider integrations from the CP-T034 scope were deferred:
- Together AI `/v1/billing/credit` endpoint (marked "coming in a future release" in providers.ts)
- Groq live rate limit header extraction from most recent API call

This ticket closes those deferred items as a Phase 2 follow-on once design partner feedback confirms whether operators want a dedicated provider surface beyond the Health Dashboard section.

---

## Problem Statement

The CP-T034 Health Dashboard section gives operators a quick status check on their providers, but it is embedded inside the health view. Operators who primarily care about providers — their model choices, reachability, and (when live balance is available) credit health — have no dedicated surface to work from. The PRD (FR8) explicitly calls for a provider management surface, not just a health sub-section.

Additionally, for operators using OpenAI with an API key that has `org:read` scope, or Together AI operators, the ability to see remaining balance and set a warning threshold before writes start failing is a genuine operational need. The CP-T034 backend already has the quota endpoint scaffolding in place — it returns `balance: null` with an informative message for OpenAI. If those operators obtain an `org:read`-scoped key or if Together AI's endpoint is tested and stable, the control plane should be ready to surface that data with a warning threshold.

## Scope

**In:**

### 1. Standalone `/providers` Route (Provider Manager View)

A dedicated view at route `/providers`, accessible from the navigation sidebar (replacing or alongside the existing sidebar entry if present).

**Provider list panel:**
- Same cards as in HealthDashboard ProviderStatus section, but with more space and full detail.
- Each provider card: name, key presence, masked key, reachability badge, last checked, Default badge.
- "Refresh all" button at the top of the list.
- Clicking a provider card opens the Provider Detail Panel (see below).

**Provider Detail Panel (right-side drawer or full panel):**
- Full quota response data: balance (if available), rate limits (if available), warning state.
- For providers where quota is not supported: shows the `reason` message from the backend with normalization copy ("Anthropic does not expose credits via API. Check your Anthropic Console for usage.") — no amber warning.
- Model list (same lazy-load as CP-T034 Health section, but without the 8-model truncation — show all models in the detail panel).
- "Refresh now" button that bypasses the 5-min quota cache.
- Editable warning threshold field (see below).

### 2. Configurable Warning Threshold

An editable field in the Provider Detail Panel allowing the operator to set a balance warning threshold per provider.

- Default: $5.00 USD (or 10% of the last observed balance — implementation choice, consistent with CP-T034 ticket spec).
- Threshold persists across page reloads. Storage mechanism: `localStorage` key `iranti_cp_provider_thresholds` — a JSON object keyed by `providerId`. This avoids introducing a new backend storage mechanism for a single UI preference.
- When the threshold is triggered (balance.remaining < threshold): the provider card shows an amber "Low balance" badge. The backend `warningThreshold.triggered` field already exists in the quota response shape — this is a pure frontend check against the stored threshold value and returned balance.
- Threshold is only shown in the UI for providers where `supported: true` on the quota endpoint. For Anthropic (permanently `supported: false`), the threshold field is hidden.

### 3. Health Dashboard Warning Banner for Triggered Thresholds

When any provider's balance is below its configured threshold (checked against `localStorage` thresholds and the cached quota response), the Health Dashboard adds a Warning-level item to the health check list:

- Text: "Provider credit low: [provider name]. Balance: [amount] [currency]. Consider topping up before your next session."
- This is a frontend-only check: read `iranti_cp_provider_thresholds` from localStorage, compare against the last `/providers` API response (already cached via React Query), compute whether any provider is below threshold.
- No new backend endpoint required — the `GET /providers` endpoint already returns reachability and can be extended to include cached quota data if warranted. For Phase 2, use the frontend comparison.

### 4. Together AI Integration (conditional on API stability)

Attempt the Together AI `/v1/billing/credit` endpoint with the `TOGETHER_API_KEY` environment variable:
- If the key is present and the endpoint returns a recognizable balance shape, return `supported: true` with `balance.remaining`.
- If the endpoint shape is unexpected or returns an error, fall back to `supported: false` with `reason: "Together AI balance API response format unexpected — check Together AI Console."`.
- This must be implemented defensively: any parse failure returns the fallback, not a 500.
- If `TOGETHER_API_KEY` is not set, Together AI is not shown in the provider list.

### 5. Groq Rate Limit Header Extraction (conditional on active key)

On a lightweight authenticated GET to `https://api.groq.com/openai/v1/models`, capture the rate limit headers (`x-ratelimit-limit-requests`, `x-ratelimit-remaining-requests`, `x-ratelimit-reset-requests`) from the response.

- Return these as `rateLimits` in the quota response: `{ requestsPerMinute: null, requestsRemaining: N, requestsResetAt: ISO }`.
- Label in the UI as "Rate limit snapshot (from last check)" with a tooltip: "This reflects the rate limit state at the time of the last API check, not a persistent balance."
- If the key is not set or the headers are absent, return `supported: false` with reason as in CP-T034.

**Out:**
- Direct provider key entry or rotation from the UI (key management is a Phase 3 security surface — this ticket is visibility only).
- Billing history or usage charts (Phase 3).
- Automated notifications when balance drops below threshold (Phase 3 notification system).
- Creating or editing providers from the UI — always out of scope for Phase 2.

## Risks

- **Together AI endpoint instability:** Together AI's billing API is not widely documented. The defensive parsing requirement is critical — parse errors must never return 500s. Wrap the entire provider response in try/catch and return the fallback shape.
- **Groq headers may be absent on model list endpoint:** Test with a real Groq key before implementing. If headers are not present on `/v1/models`, try a minimal authenticated POST to `/openai/v1/chat/completions` with an empty or minimal body — this is the most likely endpoint to return rate limit headers. If that approach is too side-effectful, return `supported: false` for Groq and document the limitation.
- **Navigation sidebar change:** Adding a `/providers` nav item changes the sidebar layout. Confirm with the PM whether this replaces the "Settings" placeholder or adds a new item. Do not add a 9th nav item without PM approval.

## Open Questions

- Should the `/providers` view replace the Health Dashboard ProviderStatus section, or should both surfaces coexist? PM preference: both coexist — Health Dashboard shows a compact summary, `/providers` shows full detail. The ProviderStatus section in Health should remain as a quick overview.
- Should the warning threshold default be `$5.00 USD` (absolute) or `10%` (relative to last observed max)? PM preference: `$5.00 USD` as the default for simplicity; operators can override per-provider.
