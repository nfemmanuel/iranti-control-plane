## Problem Statement

Iranti operators using LLM providers that expose credit or quota information have no way to see their remaining balance from the control plane. They must log in to provider dashboards separately, often only discovering a depleted balance when a write unexpectedly fails. The PRD (FR8) explicitly requires: "inspect provider usage, remaining credits, and quota health where upstream APIs expose that data."

Phase 1 correctly classifies Anthropic's balance as Informational (CP-T028: "Balance visibility not supported by this provider. This is expected.") because Anthropic does not expose credits via a stable public API. But other providers — OpenAI (usage API), Together AI (credits endpoint), Replicate (billing), Groq (usage) — do expose this data. For operators using those providers, zero visibility is a product gap, not an expected state.

**Failure mode for Dev (Power User):** Dev runs 4+ concurrent agent sessions against an OpenAI-backed instance. His writes fail mid-session when the account hits its rate limit or quota. He has no advance warning from the control plane. He diagnoses via OpenAI's usage dashboard after the fact. A control plane that could have shown "quota at 78% — approaching limit" would have prevented this entirely.

**Failure mode for Priya (Technical Founder):** Priya is evaluating Iranti for a team deployment. She needs to show that the control plane can surface operational signals for budget-conscious team environments. A "balance visible: not supported" message for every provider is not a credible answer for Phase 2 design partners.

This ticket implements credit/quota visibility for providers that expose it, within the Provider Manager view, with clear labeling of which providers support this and which do not.

## Scope

**In:**

### Backend: Provider Credit/Quota Endpoint

`GET /api/control-plane/instances/:instanceId/providers/:providerId/quota`

The endpoint must:
1. Determine the provider type for the given instance (OpenAI, Anthropic, Together AI, Groq, Replicate, or other).
2. If the provider exposes a quota or credit API: make an authenticated call using the stored provider key for this instance, return structured quota data.
3. If the provider does not expose quota data: return `{ supported: false, reason: "Provider does not expose credit or quota via API" }` — not an error.
4. Cache results for 5 minutes to avoid hammering provider APIs on repeated page loads.
5. Never return the raw provider API key in any response — only use it server-side for the outbound request.

**Provider support matrix (Phase 2 initial set):**

| Provider | Quota API | Notes |
|----------|-----------|-------|
| OpenAI | Yes | `/v1/usage` (requires `org:read` scope) |
| Anthropic | No | Does not expose credits via stable public API |
| Together AI | Yes | `/v1/billing/credit` |
| Groq | Partial | Rate limit headers only — no persistent balance |
| Replicate | No | Billing only via dashboard |
| Other/custom | No | Show "not supported" |

For providers where support is "Partial" (Groq rate limit headers): return the rate limit data from the most recent API call header if available, with a label indicating it reflects the last request, not a persistent balance.

**Response shape (supported provider):**
```typescript
interface ProviderQuotaResponse {
  supported: true;
  providerId: string;
  providerName: string;
  balance?: {
    remaining: number;
    currency: string;    // "USD" or "credits"
    asOf: string;        // ISO timestamp of the API call
  };
  rateLimits?: {
    requestsPerMinute: number | null;
    tokensPerMinute: number | null;
    requestsPerDay: number | null;
  };
  warningThreshold?: {
    triggered: boolean;
    message: string | null;   // e.g., "Balance below $5.00"
  };
  cached: boolean;
  cachedAt: string | null;
}
```

**Warning threshold rule:** If `balance.remaining` is below a configurable threshold (default: $5.00 USD, or 10% of the last observed maximum), set `warningThreshold.triggered = true`. The threshold must be configurable per instance via the Provider Manager UI (see below).

### Frontend: Provider Manager View

A new top-level view accessible from the navigation sidebar, replacing the Phase 1 Settings placeholder:
- Route: `/providers`
- Navigation label: "Providers"

**Provider list panel:**
- Lists all configured providers for the currently active instance.
- Each provider row shows: provider name, API key presence (masked — "sk-...XYZ"), last health check status, quota/credit status (badge: "Balance visible", "Not supported", "Warning: low balance"), and a "Refresh" button.

**Provider detail panel (drawer or side panel):**
- Opens on row click.
- Shows: full quota response data (balance, rate limits, warning state), last refreshed timestamp, "Refresh now" button.
- For providers where quota is not supported: shows the "not supported" message with normalization copy ("Anthropic does not expose credits via API. Check your Anthropic Console for usage.") — no amber warning.
- Editable warning threshold field (default $5.00 / 10%).

**Integration with Health dashboard:**
- If any provider has `warningThreshold.triggered = true`, the Health dashboard must surface a Warning-level item: "Provider credit low: [provider]. Balance: [amount]. Consider topping up."
- This requires the health endpoint to query the quota cache (not make a live provider call on every health check).

**Out:**
- Direct provider key entry or rotation from the UI (key management is a Phase 3 security surface — this ticket is visibility only).
- Billing history or usage charts (Phase 3).
- Automated alerts or notifications when balance drops (Phase 3 notification system).
- Support for providers not in the Phase 2 matrix above (new providers can be added incrementally without a new ticket if the API pattern is similar).

## Risks

- **Provider API instability:** Provider quota/credit APIs are not always stable or well-documented. OpenAI's usage API has changed schema before; Together AI's credit endpoint is not widely documented. Implement with defensive parsing: if the response shape is unexpected, return `{ supported: false, reason: "Provider API response format changed — update required" }` rather than failing with a 500.
- **Authentication scope requirements:** OpenAI's usage API requires `org:read` scope, which is separate from the standard `api` scope. Some users' API keys will not have this scope. Handle the 403 case explicitly: return `{ supported: false, reason: "API key does not have org:read scope to access usage data." }`.
- **Warning threshold calibration:** A default threshold of $5.00 may be too sensitive for high-volume operators or too late for low-volume users. Consider making the default a percentage of the last observed balance rather than an absolute dollar amount.
