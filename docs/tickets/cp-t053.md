# CP-T053 — Memory Explorer: ConflictLog Timeline + Field Label Fixes

**Type:** Feature
**Priority:** P2
**Assigned:** frontend_developer
**Phase:** 3 (Wave 4)
**Status:** PM-ACCEPTED 2026-03-21 — ConflictTimeline in MemoryExplorer and ArchiveExplorer with all 4 badge types and conditional fields. Written by label in expanded rows of both explorers, Source tooltip present. stability/lastAccessedAt in MemoryExplorer only, not ArchiveExplorer. conflictLog raw expand removed, valueRaw and properties expands retained. Minor: table column header still reads 'Created by' (not 'Written by') — recommend follow-on alignment, not a blocker.
**Epic:** CP-E002 (Memory Explorer)

## Summary

Two UX improvements to the Memory Explorer fact detail view, both flowing from the cross-repo audit:

1. **ConflictLog rendered as a timeline** (H4) — The `conflictLog` field on each KB fact is an append-only array of conflict events. Currently it's shown as raw JSON. Each event has `type`, `at`, `reason`, `usedLLM`, `existingScore`, `incomingScore` — enough to render a readable conflict history. This replaces the raw JSON expand with a structured event list.

2. **`createdBy` vs. `source` label distinction** (M7) — Iranti distinguishes `createdBy` (the authenticated agentId that made the write API call) from `source` (a caller-supplied provenance string). These are two different concepts but the Memory Explorer currently shows only one or labels them ambiguously. Both should be shown with distinct labels.

## Background (from cross-repo audit 2026-03-21)

From `prisma/schema.prisma`:
- `createdBy` = the agentId used for the authenticated write call (e.g. `"backend_developer"`)
- `source` = caller-supplied provenance string (e.g. `"mcp"`, `"git"`, `"manual"`)
- `conflictLog` = Json, default `[]`, append-only array of conflict events

ConflictLog entry structure (from Iranti source):
```ts
{
  type: "CONFLICT_ESCALATED" | "CONFLICT_REJECTED" | "CONFLICT_RESOLVED" | "IDEMPOTENT_SKIP",
  at: ISO timestamp,
  reason: string,
  usedLLM: boolean,
  existingScore?: number,
  incomingScore?: number,
  incomingSource?: string,
  incomingValue?: string
}
```

## Acceptance Criteria

### AC-1 — ConflictLog timeline in expanded fact row

In the Memory Explorer expanded fact row (already shows Value, Confidence, Source, Agent, etc.):

- **If `conflictLog` is empty or null:** Do not render a conflict section.
- **If `conflictLog` has one or more entries:** Render a "Conflict History" section below the existing fields.
  - Each entry shown as a row: timestamp (relative + absolute on hover), event type badge (color-coded: `CONFLICT_ESCALATED` → amber, `CONFLICT_REJECTED` → red, `CONFLICT_RESOLVED` → green, `IDEMPOTENT_SKIP` → grey), reason text, "Used LLM: Yes/No" indicator.
  - If `existingScore` and `incomingScore` are present: show them as "Existing: N vs. Incoming: N".
  - If `incomingSource` is present: show it.
  - Keep the design consistent with the Terminals palette.

### AC-2 — ConflictLog timeline in Archive Explorer expanded row

Same rendering as AC-1 in the Archive Explorer. Archive facts also carry `conflictLog`.

### AC-3 — `createdBy` and `source` shown with distinct labels

In the fact row (both collapsed summary and expanded detail):
- Rename the current "Agent" label to **"Written by"** (maps to `createdBy`)
- Keep "Source" for the `source` field but ensure the tooltip/description clarifies: "Caller-supplied provenance label (e.g. 'mcp', 'git', 'manual')"
- If both are identical strings, it's fine to show both — do not collapse them

### AC-4 — `stability` and `lastAccessedAt` in expanded fact row (H1)

From cross-repo audit H1:
- Add `stability` (Float, days) to the expanded fact detail: "Stability: N days"
- Add `lastAccessedAt` to the expanded fact detail: "Last Accessed: [relative time]"
- These are relevant for understanding decay behavior
- If null, omit rather than show "null"
- **Scope:** MemoryExplorer (live KB facts) only. The `Archive` Prisma model does not have `stability` or `lastAccessedAt` fields — do not attempt to render these in the ArchiveExplorer expanded row.

### AC-5 — No raw JSON fallback needed for conflictLog

Once the conflict timeline is rendered, the raw JSON expand for `conflictLog` specifically can be removed from the expanded row. Other JSON fields (`properties`, `metadata`) retain their raw expand behavior.

## Implementation Notes

- The Memory Explorer expanded row is in `src/client/src/components/memory/MemoryExplorer.tsx`
- The Archive Explorer expanded row is in `src/client/src/components/memory/ArchiveExplorer.tsx`
- The API already returns `conflictLog` in the fact payload — this is a pure frontend change
- The control plane server serializes `conflictLog` as `Record<string, unknown> | null` in `types.ts` — the frontend must parse it as `ConflictEntry[]`
