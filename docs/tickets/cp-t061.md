# CP-T061 — Entity Alias Management UI

**Type:** Feature
**Priority:** P2 (Wave 8)
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 8)
**Status:** Open — issued 2026-03-21
**Epic:** CP-E014 (Knowledge Base Power Features)

## Summary

Entity aliases have been deferred since Phase 1 (CP-T006 spike). The Iranti API already exposes:
- `POST /kb/alias` — create an alias from one entity to another
- `GET /kb/entity/:entityType/:entityId/aliases` — list all aliases for an entity

The control plane has no surface for this capability. Operators who need to merge entity representations or surface canonical forms must use the raw API or CLI. This ticket adds alias management to the Entity Detail view.

## Why This Is Wave 8

Aliases are a power-user feature. They matter most to operators who have ingested data from multiple sources and now have fragmented entity representations (`user/alice`, `user/alice@company.com`, `project/alice-project` all referring to the same person). As the control plane matures and operators accumulate data, alias management becomes a routine housekeeping task. Wave 7 (Metrics Dashboard) closes the historical visibility gap. Wave 8 closes the write-surface gap for knowledge base maintenance.

## Acceptance Criteria

### Backend

#### AC-1 — `GET /kb/entity/:entityType/:entityId/aliases` proxy

Add a proxy endpoint to the control plane server that proxies `GET /kb/entity/:entityType/:entityId/aliases` from the active Iranti instance. Response shape:

```json
{
  "aliases": [
    { "aliasEntityType": "user", "aliasEntityId": "alice@company.com", "createdAt": "2026-03-21T..." }
  ]
}
```

Return empty array (not 404) if the entity has no aliases. Graceful degradation if the Iranti instance is unreachable.

#### AC-2 — `POST /kb/alias` proxy

Add a proxy endpoint: `POST /api/control-plane/kb/alias` that proxies to `POST /kb/alias` on the active Iranti instance. Request body:

```json
{
  "fromEntityType": "user",
  "fromEntityId": "alice",
  "toEntityType": "user",
  "toEntityId": "alice@company.com"
}
```

Return 201 on success. Return 400 with clear error message if either entity is not found. Return 503 if Iranti is unreachable.

#### AC-3 — TypeScript clean

`tsc --noEmit` passes. No `any`.

### Frontend

#### AC-4 — Aliases panel in Entity Detail

Add an "Aliases" tab (or collapsible section) to the Entity Detail view. Show:
- A list of existing aliases for the entity: `aliasEntityType/aliasEntityId` pairs, each with a "Created" timestamp
- An empty state: "No aliases — this entity has no aliases yet."
- A "Create alias" form (inline, collapsed by default) with two fields: "Alias entity type" and "Alias entity ID"
- A "Create" button that calls `POST /kb/alias` and refreshes the list on success
- Inline error message if the alias creation fails

The Aliases section should sit below the Relationships section in the Entity Detail layout, above the footer.

#### AC-5 — Alias navigation

Each alias in the list should link to the aliased entity's detail page (`/memory/:aliasEntityType/:aliasEntityId`), so operators can navigate between aliased entities without going back to the Memory Explorer.

#### AC-6 — TypeScript clean

`tsc --noEmit` passes. No `any`.

## Implementation Notes

- The Entity Detail component is `src/client/src/components/memory/EntityDetail.tsx`. The Aliases panel should follow the same section pattern as the Relationships panel.
- The backend proxy endpoints should live in `src/server/routes/control-plane/kb.ts` alongside the existing KB proxies.
- Do not add an Aliases section to the Memory Explorer list view — only the Entity Detail page warrants the full management surface.
- No new migration is needed. Aliases are stored by Iranti's KB layer, not the control plane's DB.
