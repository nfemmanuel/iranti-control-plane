## Problem Statement

Iranti installation currently requires too much manual infrastructure work: finding and installing PostgreSQL, enabling pgvector, constructing DATABASE_URL by hand, running migrations, configuring env files, registering MCP integrations, and binding projects. Users fail at every one of these steps. CP-T005 (Installer Concept Evaluation) was commissioned to identify the top failure points and recommend the best installation path.

This ticket implements the recommended path from CP-T005: a first-class `iranti setup` CLI wizard using interactive prompts that guides a new user through the full Iranti installation and configuration flow end-to-end. It is the minimum viable installer that eliminates the top three failure points identified in CP-T005 and lands the user in a confirmed-working state within 3 minutes on a macOS machine.

**Prerequisite:** CP-T005 must be accepted by the PM before this ticket is picked up. The top failure points, recommended install path, and Phase 2 scope boundaries identified in the CP-T005 concept memo are the product requirements this ticket is implementing. If CP-T005 is not yet accepted, do not begin implementation.

## User Value

A new user can type `iranti setup` (or `npx iranti setup`) into their terminal and be walked through every step of getting Iranti running — Node version check, PostgreSQL check, pgvector check, DATABASE_URL construction, migration run, provider key setup, MCP registration, project binding, and final health verification — with clear explanations at each step and guided remediation when something is missing or wrong. They reach a confirmed-working state without reading documentation or understanding database infrastructure. This directly fulfills PRD FR9 and ER5.

## Scope

**In:**

**Wizard Framework:**
- Implemented using `clack` (npm: `@clack/prompts`) — a modern Node.js interactive CLI prompt library. Clack provides: spinners, text inputs, selects, multiselects, confirm dialogs, progress groups, and cancel handling. Its output is cleaner than Inquirer.js and it handles Ctrl+C gracefully.
- The wizard runs in a single-process Node.js script. No external process spawning except where explicitly specified below.
- All steps are grouped into named sections displayed as `clack` group headers: "System Checks", "Database Setup", "Provider Setup", "Integrations", "Verification".
- Each step shows its status inline: spinner while running, green check on success, amber warning on skip/warn, red X on failure.
- If a step fails with a recoverable error: show the error message, show the expected fix, and offer to retry or skip. The user is never forced to abort the whole wizard for a single step failure.
- Full verbose log saved to `~/.iranti/setup-log-[timestamp].txt` after every run (success or failure). The user is shown the log path at the end.

**Section 1: System Checks**
- **Node.js version check:** Verify `node --version` meets the minimum Iranti requirement (minimum version from Iranti docs or `engines` field in `package.json`). If below minimum: display current and required version, show `nvm install [version]` and `nvm use [version]` as the fix. Mark as error (cannot proceed if Node is too old).
- **macOS version check (macOS only):** Soft check — Iranti is supported on macOS 12+. Show a warning if below but do not block.
- **PostgreSQL check:** Run `pg_isready -h localhost` or equivalent. If not running: detect whether PostgreSQL is installed (check `which psql` or `pg_config`). If not installed: show Homebrew install command `brew install postgresql@16` with instructions to add to PATH. If installed but not running: show `brew services start postgresql@16`. Mark as error (cannot proceed without PostgreSQL).
- **pgvector check:** Connect to the target database (after DATABASE_URL is constructed) and check `SELECT * FROM pg_extension WHERE extname = 'vector'`. If not installed: show `psql -d [dbname] -c "CREATE EXTENSION IF NOT EXISTS vector"` as the fix, with an option to auto-run it if the user confirms.

**Section 2: Database Setup**
- **DATABASE_URL construction wizard:** Rather than asking the user to type a DATABASE_URL from scratch, guide them through its components with individual prompts:
  - Host: default `localhost`
  - Port: default `5432`
  - Database name: default `iranti` — offer to create it if it does not exist (`createdb iranti`)
  - Username: default is the result of `whoami`
  - Password: optional, hide input, default empty for local development
  - Construct and display the resulting `DATABASE_URL` for confirmation before writing it
- **Database creation:** If the named database does not exist, offer to create it with `createdb [dbname]`. Show the command being run and the result.
- **Migration runner:** Run Iranti's migration command (determined from the Iranti package — typically `npx iranti migrate` or equivalent). Show migration output in a scrollable log within the wizard. If migrations fail: show the error, show the likely cause, and offer a "Check prerequisites" link.
- **Write to `.env.iranti`:** Write the constructed `DATABASE_URL` and other setup values to `~/.iranti/instances/local/.env.iranti` (or the appropriate instance env path). Confirm before writing. Show a diff of what will be written (new file or additions to existing).

**Section 3: Provider Setup**
- **Provider selection:** A multiselect showing the supported providers (Anthropic, OpenAI, Ollama, Groq, Mistral). The user selects which they want to configure now. "Skip for now" is an option — they can always configure later.
- **Per-provider key entry:** For each selected provider: prompt for the API key (hidden input). Show the expected env variable name. Show a link to the provider's API key page (displayed, not opened automatically). After entry: write the key to the instance env file.
- **Default provider selection:** After configuring at least one provider, prompt "Which provider should be the Iranti default?" with a select.
- **Ollama detection:** If Ollama is selected, check `http://localhost:11434/api/tags`. If Ollama is not running, show `brew install ollama && ollama serve` and instructions to pull a model.
- **`instances.json` registry write:** As part of setup completion, write (or update) `~/.iranti/instances.json` to register this instance. This addresses the `decision/instances_registry` Phase 2 action item. Format: `{ "instances": [{ "name": "local", "port": 3001, "root": "~/.iranti/instances/local", "env": "~/.iranti/instances/local/.env.iranti" }] }`.

**Section 4: Integrations**
- **MCP registration check:** Check whether `.mcp.json` exists in the current working directory and whether it references Iranti. If not: show the MCP configuration block that needs to be added and offer to auto-write it (with the user's confirmation).
- **Project binding:** Ask "Would you like to bind this project to Iranti now?" If yes: run `iranti bind` (or equivalent) in the current directory. Show the result.
- **Claude Code integration:** Check for `.claude/settings.local.json` in the current project or home directory. If Iranti MCP is not listed: show the addition needed.

**Section 5: Verification**
- **Health check:** Run `iranti doctor` (or the equivalent health check endpoint) after all setup steps. Display each check result inline using the same traffic-light model as the control plane Health view (green check, amber warning, red X).
- **Success state:** If all critical checks pass: show a clear success message with the Iranti version, the active instance name, and the next steps:
  - "Start the control plane: `iranti open`"
  - "Try Iranti Chat: `iranti chat`"
- **Partial success:** If some non-critical checks show warnings: acknowledge them clearly. "Setup complete with 2 warnings. Run `iranti setup` again to address them."
- **Failure state:** If critical checks fail: show which check failed, what was tried, and what to do next. The log file path is shown prominently.

**Platform support:**
- **macOS:** Full support. All steps above are implemented for macOS. Homebrew-based remediation steps. Target: fresh macOS install completes in under 3 minutes.
- **Windows:** Reduced depth. Implement: Node version check, DATABASE_URL construction, migration runner, provider key setup, `instances.json` registry write, and final health check. Skip or stub: Homebrew steps (replace with WSL/Chocolatey notes), pgvector auto-install (show manual instructions instead), MCP file auto-write. Document the Windows scope explicitly at the start of the file.
- **Linux:** Not in scope for Phase 2 MVP. Show a clear message: "Linux setup guidance is not yet available in the wizard. See the documentation at [URL]." Do not silently fail.

**Out:**
- Signed macOS `.pkg` installer or Homebrew formula (Phase 3 — too complex for Phase 2 MVP)
- Docker-based setup (out of scope for local-first installer)
- Multi-instance setup via the wizard (Phase 3)
- Automated PostgreSQL installation via the wizard (show the command, do not auto-install without confirmation)
- Windows full-depth support at parity with macOS (Windows reduced depth is explicitly accepted for Phase 2)

## Acceptance Criteria

- [ ] `iranti setup` (or `npx iranti setup`) launches the wizard on macOS. The entry point exists and is runnable.
- [ ] Fresh macOS install: a user starting with Node.js already installed, PostgreSQL installed but no Iranti database, and an Anthropic API key completes the full wizard in under 3 minutes (timed from running `iranti setup` to the "Setup complete" message).
- [ ] The top 3 failure points from CP-T005 are each addressed by a specific wizard step. For each: (a) the step is present, (b) the guided remediation is shown when the failure is detected, and (c) the acceptance criterion for this step is confirmed by the devops_engineer with a test against a fresh-failure scenario.
- [ ] Setup ends with a health verification step that shows the actual status of all critical system components (PostgreSQL, pgvector, migration state, provider key, MCP registration). "Setup complete" is shown only when all critical checks pass.
- [ ] `~/.iranti/instances.json` is created or updated by the wizard on successful completion, registering the configured local instance. The file format matches the structure defined in `decision/instances_registry`.
- [ ] The user can re-run `iranti setup` on an already-configured instance without data loss. Re-running updates the configuration but does not drop the database, re-run migrations (unless explicitly requested), or overwrite existing provider keys (prompts to confirm overwrite).
- [ ] Windows: `iranti setup` on Windows runs without crashing. At minimum, the Node version check, DATABASE_URL construction, migration runner, and provider key setup steps execute. Steps not supported on Windows display a clear "Not available on Windows — [manual steps]" message.
- [ ] All user inputs that accept secrets (API keys) use hidden input mode (no echo to terminal). The verbose log file does not contain any API key values.
- [ ] The verbose log file is written to `~/.iranti/setup-log-[timestamp].txt` after every run. The path is shown to the user at the end.
- [ ] Ctrl+C during the wizard exits gracefully with a message: "Setup cancelled. Run `iranti setup` to continue." No partial writes to env files if the wizard is cancelled mid-step.

## Risks

- **Integration with the main Iranti CLI package.** If registering `iranti setup` requires changes to the upstream Iranti package, the devops_engineer must flag this as an upstream PR proposal and coordinate with the PM. The wizard may need to be a standalone package (`iranti-setup`) with `npx iranti-setup` as the fallback entry point for Phase 2.
- **`pg_isready` and `psql` not on PATH.** On macOS with Homebrew PostgreSQL, the binaries may not be on the user's PATH by default. The wizard must detect this and provide specific `export PATH` instructions rather than failing with an opaque "command not found" error.
- **Migration idempotency.** Running migrations on an already-migrated database must be safe. The devops_engineer must confirm that the Iranti migration runner is idempotent (running it twice does not corrupt data or cause errors).
- **API key writing to env files.** Writing secrets to disk must be handled carefully. Use `fs.chmod` to set `0600` permissions on the env file after writing. Document this in the implementation.
- **Scope creep from Windows edge cases.** Do not attempt to achieve full Windows parity in Phase 2. Explicitly document the Windows scope boundary at the top of the implementation and in the wizard output when a Windows-limited step is encountered.

## Open Questions

- Should the wizard detect and handle existing Iranti installations (upgrade path) or only support fresh installs? Recommendation: support both, but prioritize fresh install polish. On re-run, detect existing config and offer to update vs. reset.
- Should `iranti setup` be the entry point for the control plane launch as well? ("Setup complete. Open the control plane now? [Y/n]") Recommendation: yes — end the wizard with an offer to launch the control plane, reducing the step count for new users.
- What is the correct mechanism for running `iranti migrate` — is it a CLI subcommand, a direct Node script call, or a package script? The devops_engineer must confirm from the Iranti codebase.
