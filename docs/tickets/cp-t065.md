# CP-T065 — Entity Alias Panel: Rewrite for Real Iranti Alias Shape

**Type:** Bug Fix / Feature Correction

**Assigned:** frontend_developer
**Phase:** 3 (Wave 9)
**Status:** Open — issued 2026-03-21
**Epic:** CP-E014 (Knowledge Base Power Features)

## Summary

CP-T061 implemented an alias panel for the Entity Detail view, but the frontend was built against an incorrect assumed API shape from the CP-T006 spike. The real Iranti v0.2.15 alias API returns flat string alias tokens, not entity cross-references.

The CP-T061 backend was accepted (uses the real shape). This ticket rewrites the CP-T061 frontend to match the actual API contract.

## Background — What the Real API Returns

**GET /api/control-plane/kb/entity/:entityType/:entityId/aliases** returns:

```json
{
  "canonicalEntity": "user/alice-doe",
  "aliases": [
    { "alias": "alice", "aliasNorm": "alice", "source": "query", "confidence": 50, "createdAt": "2026-03-21T..." },
    { "alias": "alice-doe", "aliasNorm": "alice-doe", "source": "manual", "confidence": 90, "createdAt": "2026-03-19T..." }
  ],
  "total": 2
}
```

Aliases are human-readable lookup shorthands for a canonical entity. They are not references to other entities. There is no `aliasEntityType` or `aliasEntityId`.

**POST /api/control-plane/kb/alias** accepts:

```json
{
  "canonicalEntity": "user/alice-doe",
  "alias": "alice",
  "source": "manual",
  "confidence": 90,
  "force": false
}
```

## What Must Change in the Frontend

### 1. TypeScript types in `src/client/src/api/types.ts`

Replace the current `EntityAlias` and `EntityAliasesResponse` interfaces:

```typescript
// WRONG (current):
export interface EntityAlias {
  aliasEntityType: string
  aliasEntityId: string
  createdAt: string
}
export interface EntityAliasesResponse {
  aliases: EntityAlias[]
}

// CORRECT (CP-T065 target):
export interface EntityAlias {
  alias: string
  aliasNorm: string
  source: string
  confidence: number
  createdAt: string
}
export interface EntityAliasesResponse {
  canonicalEntity: string
  aliases: EntityAlias[]
  total: number
}
```

### 2. `AliasRow` component in `EntityDetail.tsx`

Replace the entity-link rendering with token display:

- Show `alias.alias` (the human-readable token) as the primary label, in monospace/code style
- Show `alias.source` and `alias.confidence` as secondary metadata
- Show `alias.createdAt` as relative time (existing `formatRelativeTime` helper)
- No Link to another entity detail page — flat aliases do not map to entity routes
- Use `alias.aliasNorm` as the React `key` (normalized form is stable)

### 3. `CreateAliasForm` component in `EntityDetail.tsx`

Replace the two-field (entity type + entity ID) form with a single-field form:

- One text input: "Alias token" — the string to register (e.g., `alice`)
- `canonicalEntity` is derived from the current `entityType`/`entityId` props: `${entityType}/${entityId}`
- POST body: `{ canonicalEntity: \`${entityType}/${entityId}\`, alias: token.trim() }`
- Optional: add `source` and `confidence` fields (default: source `"manual"`, confidence 80) as secondary collapsed inputs — but keep the primary path as single-field
- Clear inline errors on mutation success; surface Iranti errors (e.g., alias already exists) to the user

### 4. `AliasesPanel` query key

The query key `['entity-aliases', entityType, entityId]` and the fetch URL are correct and unchanged.

- Empty state still shows "No aliases — this entity has no aliases yet."

- Confidence shown with the same `ConfidenceBar` component used elsewhere in EntityDetail

- No layout regressions on the Entity Detail page

`tsc --noEmit` passes in both `src/server` and `src/client`. No `any` introduced.
