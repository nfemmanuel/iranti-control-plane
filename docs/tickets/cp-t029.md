## Problem Statement

The Instance Manager performs health probes to determine whether each Iranti instance is running. There are two distinct failure modes in the current design that will cause users to distrust the health display:

**Failure Mode A — Stale status:**
The control plane probes health on page load and caches the result. If the user leaves the page open for an extended period (e.g., 30 minutes, 2 hours — both common for a management tool), the cached status becomes stale. The instance may have stopped. The display still shows "Running." The user reads a false positive and makes decisions based on incorrect state.

**Failure Mode B — Ambiguous timeout result:**
When a health probe times out (because the instance runtime is stopped or unreachable), the display may show "status: checking..." indefinitely, or may show "unknown" — a label that is technically accurate but operationally useless. "Unknown" does not tell the user whether the instance stopped, the probe failed, the network is misconfigured, or the control plane has a bug.

Priya (Technical Founder) is the most at-risk persona: she is specifically evaluating multi-instance management as a Phase 1 capability. If she cannot trust the health status of her two test instances, the entire Instance Manager fails its core purpose. The v0.1.0 Criterion 4 (Instance Manager accuracy) requires: "Status shows 'running' for a stopped instance (or vice versa) without a staleness indicator" as an explicit fail condition.

## Scope

**In:**

### Last-Checked Timestamp

Every instance health status display must include a "last checked" timestamp:
- Format: relative time ("checked 2 minutes ago", "checked 34 minutes ago")
- The timestamp must update whenever a new probe completes.
- If the status data is older than 5 minutes, add a visual staleness indicator: amber or grey styling with a label such as "Status may be stale" and a "Refresh" affordance.
- If the status data is older than 10 minutes, never display the cached state as authoritative without the staleness indicator prominently visible.

### Manual Refresh

Add a per-instance "Refresh" button (or a global "Refresh all" button for the instance list) that triggers an on-demand health probe. The probe must complete and update the display before the button becomes interactive again (show a loading state during the probe).

### Status Label Precision

Replace ambiguous status labels with a precise four-state vocabulary:

| State | Label | Meaning | Condition |
|-------|-------|---------|-----------|
| Running | "Running" (green) | Health probe succeeded — Iranti runtime responded | Probe returned HTTP 200 (or equivalent health check success) |
| Unreachable | "Unreachable" (amber) | Probe tried and got no response or connection refused | Probe timed out or connection refused |
| Stopped | "Stopped" (grey) | Explicit shutdown signal received from the runtime, or known to be stopped by instance metadata | Future: runtime sends shutdown event; for now, treat as alias for Unreachable when probe times out cleanly |
| Not Configured | "Not configured" (grey) | No connection information available for this instance — cannot probe | Instance metadata has no database host or port |

**Do not use "unknown" as a displayed status.** Unknown is an internal state — when a probe has not yet completed, show "Checking..." with a spinner. When a probe fails, show "Unreachable."

### Staleness Warning Rule

- Health data older than 5 minutes: show amber "Status may be stale" text next to the status label
- Health data older than 10 minutes: show amber warning + "Last checked [time]" with a visible "Refresh" CTA
- "Running" status older than 10 minutes must never be displayed without a staleness warning — a 10-minute-old "Running" signal is not a reliable running signal

**Out:**

- Automatic periodic background polling (nice to have for Phase 2 — adds real-time reliability but increases API load).
- Implementing a shutdown event from the Iranti runtime (would enable precise "Stopped" state — Phase 2).
- Alerting or notifications when an instance goes down.

## Acceptance Criteria

- [ ] Every instance health status display includes a "last checked" relative timestamp ("checked 2 min ago").
- [ ] Timestamp updates whenever a new probe completes.
- [ ] A manual "Refresh" button (per-instance or global) triggers an on-demand health probe and updates the status and timestamp.
- [ ] Health data older than 5 minutes shows a staleness indicator next to the status label.
- [ ] Health data older than 10 minutes shows a staleness indicator plus a visible "Refresh" CTA.
- [ ] "Running" is never displayed for health data older than 10 minutes without a staleness warning.
- [ ] A probe timeout or connection refusal results in "Unreachable" — not "Unknown" and not an indefinite "Checking..." state.
- [ ] "Checking..." with a spinner is shown only while an active probe is in flight (not as a persistent state after a failed probe).
- [ ] "Not configured" is shown when connection information is missing — not "Unreachable."
- [ ] After removing a required env key and refreshing, the status reflects the updated configuration state within one probe cycle.
- [ ] All status states are visually distinct in both light and dark mode, including without relying on color alone.
- [ ] TypeScript compiles without errors.

## Risks

- **Probe latency on stopped instances:** Probing a stopped instance may take the full timeout duration (e.g., 5-10 seconds) before returning a connection refused error. This creates a noticeable delay when the user opens the Instance Manager with a stopped instance. Mitigation: set probe timeout to a maximum of 3 seconds for Phase 1; show "Checking..." spinner immediately on load so the user knows a probe is in progress.
- **Race condition on manual refresh:** If the user clicks "Refresh" while a probe is already in flight, the system must not start a second parallel probe. Debounce the Refresh button: disable it while a probe is active.
