## Problem Statement

Any user who opens the control plane against a freshly installed or lightly-used Iranti instance will encounter empty tables and blank views. Without explicit empty-state UI design, the user cannot distinguish between three fundamentally different conditions:

1. **Connected, no data yet** — Iranti is reachable, the database is connected, but the instance has no memory written yet (or very little). This is expected and healthy.
2. **Not connected** — the control plane cannot reach the Iranti runtime or database. The empty appearance is caused by a connection failure.
3. **Filtered results returned nothing** — the user applied a search or filter that matched zero records. The data exists; the query returned nothing.

These three conditions require completely different user responses:
- Condition 1: no action needed — write a fact to see data appear.
- Condition 2: investigate the connection — check Health dashboard.
- Condition 3: adjust the filter or search query.

Showing the same blank table for all three is a trust-destroying failure. Marcus (Solo Dev) is the most at-risk persona: he may open the control plane specifically because something feels wrong. An empty table with no explanation confirms his anxiety without helping him understand whether the system is working.

Empty states are one of the most common reasons first-time users abandon a tool after the first session. This is a pre-v0.1.0 blocker.

## Scope

**In:**

Implement distinct, context-appropriate empty states for the following views:

### Memory Explorer (Knowledge Base table)

**Condition A — Connected, no data:**
- Heading: "No facts in memory yet"
- Body: "Your Iranti instance is connected. Write your first fact using `iranti write` or open Iranti Chat."
- Optional: link to docs or CLI quick-start

**Condition B — Not connected / API error:**
- Heading: "Unable to load memory"
- Body: "The control plane could not reach your Iranti instance. Check the Health dashboard for connection details."
- CTA button: "Open Health Dashboard" (navigates to Health view)

**Condition C — Filtered / searched, no results:**
- Heading: "No facts match your filter"
- Body: "Try adjusting your search or clearing your filters."
- CTA: "Clear filters" button that resets all active filters

### Archive Explorer

**Condition A — Connected, no archived facts:**
- Heading: "No archived facts"
- Body: "Facts appear here when they are superseded, contradicted, expired, or decayed by the Archivist."

**Condition B — Not connected:**
- Same pattern as Memory Explorer Condition B.

**Condition C — Filtered, no results:**
- Same pattern as Memory Explorer Condition C.

### Staff Activity Stream

**Condition A — Connected, stream active, no events yet:**
- Heading: "No events yet"
- Body: "The stream is connected. Events will appear here when the Librarian processes a write. Try running `iranti write` in a terminal to generate an event."
- This must NOT say "No events" with no further explanation — that looks like a broken stream.

**Condition B — Stream not connected:**
- Heading: "Stream not connected"
- Body: "The control plane could not connect to the Iranti event stream. Check the Health dashboard."
- CTA: "Open Health Dashboard"

**Condition C — Filtered, no results:**
- Heading: "No events match your filter"
- CTA: "Clear filter"

### Shell-Level Connection Status

- Add a connection status indicator in the application shell header: a small dot or badge that shows green (Iranti API reachable) or red (Iranti API unreachable) with a brief label.
- When the API is unreachable, all data views show Condition B empty states.
- This indicator prevents the user from needing to navigate to Health to understand why all views are empty.

**Out:**

- Animated illustrations or heavy onboarding wizard flows (Phase 2).
- Guided setup from within empty states (links to CLI docs are acceptable; full in-app setup is Phase 2).
- Skeleton loaders (those are loading states, not empty states — separate concern).

## Acceptance Criteria

- [ ] Memory Explorer shows three distinct empty state variants: connected+no data, not connected, filtered+no results. Each has appropriate heading, body text, and CTA.
- [ ] Archive Explorer shows the same three variants with archive-appropriate copy.
- [ ] Staff Activity Stream shows: stream connected+no events yet, stream not connected, filter+no results — each with distinct messaging.
- [ ] The "stream connected, no events yet" state explicitly tells the user the stream is healthy and suggests running `iranti write` to generate an event — it does not look like a broken stream.
- [ ] Shell-level connection indicator is present in the header: green when Iranti API is reachable, red when not.
- [ ] When the Iranti API is unreachable, all three data views display the "not connected" variant.
- [ ] All empty state copy is visible and legible in both light and dark mode.
- [ ] TypeScript compiles without errors.
