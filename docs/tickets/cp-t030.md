## Problem Statement

The temporal history timeline in Memory Explorer must display the full history of a fact: the current value plus all superseded, contradicted, expired, and decayed intervals from the `archive` table. This requires a JOIN across `knowledge_base` (current facts) and `archive` (historical intervals), with proper handling of `archivedReason`, `supersededBy`, and related fields.

There are two independent failure modes that will cause the temporal history view to show an incomplete or misleading timeline:

**Failure Mode A — Backend omits archive JOIN:**
If the temporal history API endpoint queries only `knowledge_base` — or if the `archive` JOIN is implemented but filters incorrectly — the endpoint returns only the current value. The timeline renders with one entry. A user who knows this fact has been superseded multiple times will see a single-entry timeline, conclude the feature is broken or incomplete, and verify in Adminer. This is not a recoverable UX issue — it is a backend correctness issue.

**Failure Mode B — `archivedReason` returned as raw enum codes:**
The `archive` table stores `archivedReason` as codes (`superseded`, `contradicted`, `expired`, `decayed`). If these are returned to the frontend without human-readable labels, the timeline shows raw codes. Users who are not deeply familiar with Iranti's internal model cannot interpret them. Dev (Power User) can interpret them — but he will correctly note it as a product quality failure. Marcus (Solo Dev) cannot interpret them at all.

Dev (Power User) is the highest-impact persona for this ticket. His primary Phase 1 job is temporal history inspection. He currently maintains saved SQL queries to answer "why did this fact change?" — the control plane's temporal history view is meant to replace those queries. If the view returns incomplete data, Dev will cross-check in Adminer, confirm archived intervals exist, and the control plane loses its core credibility for him.

## Scope

**In:**

### Backend: Temporal History Endpoint Correctness

The `GET /api/control-plane/memory/:entityType/:entityId/:key/history` endpoint (or equivalent) must:

1. Return the current fact from `knowledge_base` if one exists (with `status: current`)
2. Return all archived intervals from `archive` for the same `entity`+`key` pair, sorted by `validFrom` descending (most recent first)
3. For each archived interval, include: `valueRaw` (or summary), `validFrom`, `validUntil`, `archivedReason` (human-readable label — see below), `supersededBy` (resolved as entity reference, not raw UUID, where possible), and `conflictLog` (raw, available for expansion)
4. Return an empty `history: []` array (not an error) when no archived intervals exist — with a flag `hasHistory: false` the frontend can use to show "No history — this fact has not been superseded or archived"

### `archivedReason` Human-Readable Labels

The following mapping must be applied before returning `archivedReason` to the frontend:

| Raw code | Human-readable label |
|----------|---------------------|
| `superseded` | Superseded by newer write |
| `contradicted` | Contradicted by conflicting source |
| `expired` | Expired (validUntil passed) |
| `decayed` | Decayed by Archivist |

If additional reason codes exist in the schema, they must be mapped before ship. Unknown codes should fall back to the raw code with a "(unknown reason)" suffix — not silently dropped.

### QA: Seed Test for v0.1.0 Sign-Off

Before v0.1.0 ships to design partners, QA must verify the temporal history view against seeded data with deliberate history. The seed scenario:

1. Create an entity `test/temporal_history_check` with key `test_value` and value `"version 1"` using `iranti write`.
2. Write a second value `"version 2"` to the same entity+key with a different source, triggering a supersession. Confirm the first value is archived with `archivedReason: superseded`.
3. If Archivist decay is testable: trigger a decay event to archive a third value with `archivedReason: decayed`.
4. Open the temporal history view in the control plane for `test/temporal_history_check / test_value`.
5. Verify: all intervals appear (current + archived), in reverse-chronological order; archive reasons show human-readable labels; supersededBy reference is present for the superseded interval.

This test must pass before v0.1.0 is handed to design partners. QA should document the result in Iranti as `ticket/cp_t030` key `qa_seed_result`.

**Out:**

- Conflict log rendering / formatting (the raw `conflictLog` JSON is accessible for expansion but rendering it with structured UI is a Phase 2 improvement).
- Diff view between adjacent intervals (Phase 2 enhancement — out of Phase 1 scope).
- Event replay from the timeline (explicitly Phase 2 per PRD).
- Fixing the frontend timeline component rendering (this ticket is backend correctness + QA; frontend timeline rendering issues are separate bugs if found during QA).

## Acceptance Criteria

- [ ] `GET /api/control-plane/memory/:entityType/:entityId/:key/history` returns the current fact plus all archived intervals for the same entity+key pair.
- [ ] Archived intervals include: `validFrom`, `validUntil`, `archivedReason` (human-readable label), and `supersededBy` where available.
- [ ] `archivedReason` raw codes are mapped to human-readable labels before the response leaves the backend. No raw enum codes reach the frontend.
- [ ] When no archived intervals exist, the endpoint returns `{ current: {...}, history: [], hasHistory: false }` — not an error, not an empty 200 with no history field.
- [ ] QA seed test passes: entity with 2+ archived intervals shows all intervals in the temporal history view, in reverse-chronological order, with human-readable archive reason labels.
- [ ] The seed test result is documented in Iranti: `ticket/cp_t030` key `qa_seed_result`.
- [ ] Backend endpoint has a test covering the archive JOIN: given an entity with archived intervals, the response includes all intervals. Given an entity with no archived intervals, `hasHistory: false` is returned.

## Dependencies

- `archive` table schema — confirm `archivedReason` column name, type, and valid values with the backend team before implementing the mapping.
- `supersededBy` field — confirm whether this field stores a UUID reference and whether a lookup to resolve the human-readable entity reference is feasible within Phase 1. If UUID-only, return the UUID; do not fabricate a label.
- QA environment with a seeded Iranti instance — qa_engineer must have access to an Iranti instance where `iranti write` commands can be run to create supersession events.

---

## Risks

- **Archive JOIN performance:** For entities with long histories (Dev has facts written by many agent sessions over 4 months), the JOIN could return many rows. Implement pagination or a default limit (e.g., last 50 intervals) with a "Load more" affordance in the frontend. Confirm performance requirements with the PM.
- **Unknown `archivedReason` codes:** The mapping table above covers the documented codes. If undocumented codes exist in production data (from earlier Iranti versions or direct DB writes), the fallback to "raw code (unknown reason)" must not cause a UI error.
