# CP-T064 — Documentation: Update Memory Explorer Guide for CP-T056 and CP-T057

**Type:** Documentation
**Priority:** P3 (Wave 8)
**Assigned:** technical_writer
**Phase:** 3 (Wave 8)
**Status:** PM-ACCEPTED 2026-03-21 (with noted spec gaps — see PM acceptance note below)

## Summary

CP-T056 (Temporal History asOf picker) and CP-T057 (WhoKnows Contributor Panel) were both accepted in Wave 5 (2026-03-21). Both features add meaningful new capability to the Memory Explorer and Entity Detail views. The `docs/guides/memory-explorer.md` guide (or equivalent) has not been updated to reflect either of these features.

This ticket dispatches the technical_writer to update the guide with accurate documentation for:
1. The asOf date/time picker in the Temporal History view (CP-T056)
2. The Contributors panel in the Entity Detail view (CP-T057)

Additionally, the `docs/guides/getting-started.md` guide should be updated to mention the Metrics Dashboard (CP-T060, PM-ACCEPTED 2026-03-21).

## Why This Is Wave 8

The guides are operator-facing documentation. A feature that exists but is undocumented is effectively invisible to operators who rely on the guide rather than discovering features by clicking through the UI. CP-T056 and CP-T057 were accepted in Wave 5 but the guide update was deferred to avoid blocking Wave 7. Now that Wave 7 is accepted and Wave 8 is beginning, the documentation debt must be paid.

## Acceptance Criteria

#### AC-1 — Temporal History asOf picker documented

In `docs/guides/memory-explorer.md` (or the correct guide file for the Temporal History view), add a section or expand the existing Temporal History section to describe:
- What the asOf picker does: "lets you query the knowledge base as it existed at a specific point in time — useful for reconstructing what was known before a recent write or conflict resolution"
- How to use it: the date/time input, the "Query" button or equivalent trigger, and what changes in the UI when an asOf query is active (e.g., facts shown reflect the state at that timestamp, not the live state)
- What the "Current" badge means when an asOf query is active vs. not
- Any known limitations (e.g., asOf is bounded by when `staff_events` data begins)

#### AC-2 — Contributors panel documented

In the appropriate guide file for the Entity Detail view, add a section or expand the existing Entity Detail section to describe:
- What the Contributors panel shows: "the agents and sources that have written facts to this entity, with write count, rejection count, and first/last seen timestamps — derived from `staff_events` via the `GET /kb/whoknows` endpoint"
- How to open it (tab, collapsible section, or whatever the implementation uses)
- Empty state: what is shown when no contributor data is available (e.g., before `staff_events` is populated)
- The relationship to the `GET /kb/whoknows` endpoint — contributors are computed by Iranti, not stored separately

#### AC-3 — Metrics Dashboard entry in getting-started.md

In `docs/guides/getting-started.md`, add a row to the "What's Available Now" table for the Metrics Dashboard:
> **Metrics** | Historical KB growth, per-agent write volume, and activity summary statistics at `/metrics` — line and bar charts over 7d/30d periods, plus summary stat cards for total facts, facts last 24h, active agents last 7d, and rejection rate. (Wave 7 — CP-T060, PM-accepted 2026-03-21)

#### AC-4 — No new files created unnecessarily

Update existing guide files in place. Do not create a separate "Metrics guide" unless the PM explicitly asks for one. The Metrics Dashboard is simple enough to describe in the getting-started table with a short entry.

## PM Acceptance Note — 2026-03-21

**Decision: PM-ACCEPTED with two noted spec gaps.**

AC-1 (asOf picker): PASS. The guide correctly describes reactive firing (no separate Query button), which matches the actual implementation. The spec said "Query button or equivalent" — "equivalent" covers the reactive approach. The technical_writer made the right call to document actual behavior rather than the spec phrase.

AC-2 (Contributors panel): PASS with noted gap. The guide documents `writeCount` and `lastContributedAt` accurately. The spec's mention of `rejectionCount` and `firstSeen` was aspirational — those fields do not exist in the live `GET /kb/whoknows` response shape. The guide is accurate to the implementation. The field gap is an upstream Iranti limitation, not a documentation error.

AC-3 (Metrics Dashboard in getting-started.md): PASS. Row present and accurate.

AC-4 (no unnecessary new files): PASS. Existing guides updated in place.

Both spec-to-implementation gaps are recorded here. No follow-on ticket needed — documenting non-existent fields would be misleading.
