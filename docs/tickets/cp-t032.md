## Problem Statement

The Memory Explorer in Phase 1 exposes entity facts and temporal history in a table/list format. The `entity_relationships` table — which records how Iranti knows entities relate to each other (agent-sourced relationships, project bindings, cross-entity provenance) — has a read API endpoint (`GET /api/control-plane/entity/:entityType/:entityId/relationships`) but no dedicated UI surface. Users who want to understand how entities connect must read raw JSON or manually cross-reference fact tables.

The PRD (FR1) explicitly calls out "relationship view for entity_relationships" as part of the Memory Explorer's required capabilities. The Phase 1 entity detail page shows relationships as a flat JSON list. This fails the product principle "readability before cleverness": relationships are inherently a graph-shaped concept and a flat list is not how humans reason about them.

Specific failure modes for Phase 2 target users:

**Dev (Power User):** Dev has 4 months of agent-sourced entity relationships across 3 instances. He wants to understand which entities are co-referenced by his agents, and whether a relationship he believes was established still exists or has been superseded. A flat list of relationship rows makes this cross-referencing manual.

**Priya (Technical Founder):** Priya is evaluating Iranti as infrastructure for shared agent memory across team projects. She wants to show stakeholders how Iranti tracks entity context — a graph visualization makes that case materially more clearly than raw rows.

The Phase 1 implementation left the relationships endpoint working but the surface investment at zero. This ticket closes that gap.

## Scope

**In:**

### Backend: Relationship Subgraph Endpoint

A new endpoint (or extension of the existing relationships endpoint) to support graph traversal:

`GET /api/control-plane/entity/:entityType/:entityId/relationships/graph`

Query parameters:
- `depth` (default: 1, max: 2) — how many hops from the root entity to traverse
- `relationshipTypes` (optional) — comma-separated filter on `relationshipType` values
- `limit` (default: 50 max per depth level) — prevent graph explosion

Response shape:
```typescript
interface RelationshipGraph {
  rootEntity: { entityType: string; entityId: string };
  nodes: Array<{
    entityType: string;
    entityId: string;
    factCount: number;       // current facts in knowledge_base for this entity
    isRoot: boolean;
  }>;
  edges: Array<{
    fromEntityType: string;
    fromEntityId: string;
    toEntityType: string;
    toEntityId: string;
    relationshipType: string;
    confidence: number | null;
    source: string | null;
    createdBy: string | null;
  }>;
  truncated: boolean;        // true if depth or limit caused results to be capped
}
```

The endpoint must handle cycles gracefully (do not traverse a node more than once).

### Frontend: Relationship Graph Panel

Within the entity detail page (Memory Explorer), add a "Relationships" tab alongside the existing "Facts" and "History" tabs.

The Relationships tab must include:

**Graph canvas:**
- Force-directed or hierarchical graph layout using a lightweight library (D3 force layout or react-flow — implementation choice; confirm with system_architect which is already in the dependency tree).
- Root entity rendered as a visually distinct node (filled, larger).
- Related entities as secondary nodes with entity type and ID as labels.
- Edges labeled with `relationshipType`.
- Clicking a node navigates to that entity's detail page.
- Depth control: a toggle between depth=1 and depth=2. Depth=2 must show a warning if the graph is truncated.

**Fallback (empty state):**
- If the entity has no recorded relationships, show: "No relationships recorded for this entity. Relationships are created when Iranti agents write cross-entity facts or establish provenance links."

**List toggle:**
- A toggle to switch from graph view to a flat list view (table of edges) for users who prefer tabular reading or have accessibility needs that make graph canvas interaction difficult.

**Out:**
- Relationship creation or editing from the UI (read-only — consistent with Phase 1 principle; write operations belong to Phase 3 if at all).
- Full knowledge graph across all entities (start from a root entity, not a global graph view — global graph is a Phase 3 surface with significant performance implications).
- Custom relationship type management.
- Graph export (PNG/SVG) — nice to have for Phase 3.

---

## Risks

- **Graph library size:** Adding a graph visualization library may increase the frontend bundle. Prefer D3 submodules (force simulation only) over a full graph framework. Confirm bundle impact before shipping.
- **Graph readability at depth=2:** At depth=2 with a well-connected entity (e.g., a project entity referenced by many agents), the graph may be visually overwhelming. The `limit` parameter and truncation warning mitigate this, but the UX of a dense graph still needs design attention.
- **`entity_relationships` sparsity in Phase 1 installs:** Most Phase 1 design partners will have few or no relationships in their data. The empty state and the explanation copy are therefore as important as the graph itself for first impressions.

---

## Open Questions

- Which graph library is already in the dependency tree? Confirm with system_architect before implementing to avoid adding a new heavy dependency.
- Should relationship directionality be bidirectional by default (show edges regardless of from/to direction relative to root), or should directional asymmetry be preserved in the layout? Product preference: bidirectional for Phase 2, directional arrows as optional Phase 3 toggle.
