## Problem Statement

Iranti installation currently requires too much infrastructure knowledge: setting up PostgreSQL, running Docker or a local server, constructing env files, configuring MCP settings, and connecting Claude or Codex integrations. PRD FR9 and ER5 require a dramatically simpler installation path, but the right form — a dedicated OS-level installer, a guided setup flow embedded in the control plane, or a pre-install bootstrap tool — has not been decided. Cutting Phase 2 installer tickets without first evaluating the options risks either under-scoping (missing the real friction points) or over-scoping (building a full installer when a simpler guided flow would suffice). This concept memo resolves that ambiguity before any implementation work is committed.

## User Value

New users can get Iranti installed and working without already knowing how to set up a local PostgreSQL instance, assemble scattered config files, or understand how MCP integration files are structured. Reducing the install barrier directly expands who can use Iranti. Every step removed from the current setup path is a user who doesn't drop off.

## Scope

In:
- Document the current Iranti install path end-to-end: every manual step, every dependency (PostgreSQL, Node, Docker, etc.), every config file that must be created, every environment variable that must be set. This is the baseline.
- Identify where users most commonly fail or drop off in the current setup path — top 3 failure points minimum, with evidence or reasoning.
- Evaluate at least 2 install path options in concrete terms:
  - **Option A: Dedicated installer** — e.g., a signed macOS `.pkg`, a Homebrew formula, an `npm` global install, or a shell script. Covers how dependencies are bundled or verified, distribution mechanism, update path.
  - **Option B: Guided setup flow embedded in the control plane** — a first-run wizard or setup checklist inside the control plane UI that walks the user through dependencies, config, and first connection. Covers what "pre-requires" must already be satisfied for the control plane itself to launch.
  - Additional options may be evaluated if the devops_engineer identifies a materially different approach.
- Produce a recommendation with rationale explicitly tied to PRD FR9 and ER5.
- Estimate the Phase 2 implementation complexity for the recommended path: rough effort, key unknowns, likely subtasks.
- Produce all output in a concept memo at `docs/specs/installer-concept.md`

Out:
- Implementation of any installer or onboarding flow
- Changes to upstream Iranti install scripts (may be noted as proposals, flagged clearly as out of scope for this repo)
- Scoping of Phase 2 installer tickets — that work happens after the PM reviews and approves this memo

## Acceptance Criteria

- [ ] Current install path is accurately documented: every manual step in sequence, every named dependency with version requirements where relevant, every config file that must be created with its purpose described. The baseline must be specific enough to count the steps — not a summary.
- [ ] Top 3 most common failure points in the current setup path are identified and described. Each failure point includes: what breaks, why it breaks (root cause), and what the user experiences when it breaks.
- [ ] At least 2 install path options are evaluated with concrete pros and cons for each — not abstract bullet points. Pros/cons must be tied to real operator experiences or specific implementation realities.
- [ ] The recommended option is stated clearly with rationale explicitly referencing PRD FR9 ("dramatically simpler installation path") and ER5 ("installation and onboarding are first-class product surfaces"). The recommendation must be a single choice, not a "it depends."
- [ ] Phase 2 implementation complexity estimate is included for the recommended path: rough effort framing (e.g., 2-week spike vs 6-week feature), top 3 unknowns, and a sketch of the likely Phase 2 subtasks this memo would generate.
- [ ] Memo identifies any installer scope that should be explicitly excluded from MVP (Phase 2) to keep scope manageable.
- [ ] PM has reviewed and approved the memo before Phase 2 installer tickets are cut.

## Risks

- Installer scope can grow unboundedly if every edge case and OS variant is addressed. The memo must be deliberately opinionated about the MVP scope for Phase 2 — "we will address macOS first and treat Linux as a stretch" is an acceptable and useful stance.
- Distribution mechanisms (signed `.pkg` installers, Homebrew formulas, `npm` packages) each carry their own complexity: code signing, notarization, registry publishing. These must be surfaced in the memo so the PM understands the total cost, not just the UX surface.
- If the recommended path requires the control plane to be running before setup can begin (e.g., a guided UI flow), the memo must address the chicken-and-egg problem: how does the user get the control plane running in the first place?

## Open Questions

- Should the installer be OS-specific (macOS first, given the likely primary user base) or cross-platform from the start? Cross-platform is more inclusive but significantly more complex; macOS-first allows faster delivery of a polished experience.
- Should the control plane itself be the install surface (a first-run wizard inside the running app), or is a pre-install bootstrap tool necessary before the control plane can run at all?
- What is the minimum set of dependencies that must be pre-installed by the user vs can be bundled or auto-provisioned by the installer? PostgreSQL is the most significant question here — bundling it is non-trivial.
- Are there existing Iranti community-maintained install scripts or guides that reveal where users actually struggle? If yes, these are high-value inputs to the failure point analysis.
