## Description

CP-T048 (Platform Installer Packages) has been PM-accepted in all respects except one: AC-11 (clean-machine installer validation). This is the sole remaining gate for formal GitHub Release of both v0.3.0 and v0.4.0.

**Current state:**
- Node SEA single-executable binaries are built for Windows (`.exe`), macOS (`.dmg`), and Linux (`.AppImage`)
- CI validates the build artifacts exist and pass basic smoke checks
- The binaries are available in `dist/`
- **No human has tested the installer on a machine with no Node.js pre-installed**

The clean-machine test is the product promise of CP-T048: *"A user with no Node.js installation can install and launch the Iranti Control Plane using a platform-native installer on Windows, macOS, or Linux."* Without this test passing, that claim is unverified.

This ticket is not an implementation task. It is a formal QA execution task. The QA engineer runs the test plan, documents results, and either accepts the release or documents exactly what fails.

## What to Do

### Step 1: Read the test plan

Read `docs/qa/cp-t048-clean-machine-test-plan.md` in full before beginning.

If the test plan file does not exist at that path, check `docs/qa/cp-t048-ac11-checklist.md` (referenced in v0.3.0 release notes). Read whichever file exists.

### Step 2: Prepare clean-machine environments

The test requires machines (or VMs) with:
- No Node.js, npm, or nvm installed
- No existing Iranti installation
- Fresh OS state (VM snapshot recommended so the test can be repeated)

**Minimum required coverage (to accept AC-11):**

| Platform | Installer | Required |
|----------|-----------|---------|
| Windows 10/11 | `.exe` | Yes |
| macOS 13+ | `.dmg` | Yes |
| Linux (Ubuntu 22.04 LTS) | `.AppImage` | Yes |

At least one of the three must pass for a partial acceptance. All three must pass for full acceptance.

**If a VM is not available for all platforms:** Document which platforms were tested and which were not. A partial pass (e.g., Windows passes, macOS not yet tested) is acceptable as a partial acceptance. PM will decide whether to ship with the partial pass or wait for full coverage.

### Step 3: Run the test procedure

For each platform, follow the test plan exactly. Record results in the pass table format below. For each test step, record:
- Pass / Fail
- Notes (error messages, screenshots if available, environment details)

**Test steps (minimum):**
1. Download the installer from `dist/` (or the CI artifact location)
2. Run the installer — no command line, just the installer UI or double-click
3. Launch the control plane — should open a browser window or tray app
4. Connect to a running Iranti instance (connection URL prompt or config file)
5. Verify Memory Explorer loads and shows at least the Getting Started page
6. Verify Health Dashboard shows green for DB and Iranti connectivity
7. Verify no "Node.js not found" or runtime errors appear anywhere

### Step 4: Document results

Fill in the pass table:

```
## Clean-Machine Test Results — 2026-03-21

| Platform | Installer | Step 1 | Step 2 | Step 3 | Step 4 | Step 5 | Step 6 | Step 7 | PASS/FAIL |
|----------|-----------|--------|--------|--------|--------|--------|--------|--------|-----------|
| Windows  | .exe      |        |        |        |        |        |        |        |           |
| macOS    | .dmg      |        |        |        |        |        |        |        |           |
| Linux    | .AppImage |        |        |        |        |        |        |        |           |
```

For any FAIL: describe exactly what failed, the error message shown, and whether it is a known issue or a new defect.

### Step 5: Report to PM

Report the results to the PM with one of the following conclusions:

**A. Full pass:** "All three platforms pass AC-11. Recommend tagging v0.3.0 and v0.4.0 and creating GitHub Releases."

**B. Partial pass:** "Windows and Linux pass. macOS .dmg not yet tested (VM unavailable). Recommend PM decides whether to ship with partial coverage or wait."

**C. Fail with known defect:** "Windows .exe fails at Step 2 — [describe defect]. This is either a known issue or a new regression. Requires a fix ticket before release."

**D. Fail with new defect:** "New defect found: [describe]. Filing as [defect ticket ID]. Release blocked pending fix."

## Acceptance Criteria

### AC-1 — Test plan read and followed

The qa_engineer has read `docs/qa/cp-t048-clean-machine-test-plan.md` (or the equivalent checklist file) before beginning.

### AC-2 — At least one platform tested

At minimum, one clean-machine test has been run on a machine with no Node.js pre-installed. The result is documented.

### AC-3 — Pass table completed

A pass table with per-step results is documented in this ticket (or in a linked test report file at `docs/qa/cp-t048-ac11-results-2026-03-21.md`).

### AC-4 — PM receives a clear recommendation

PM receives one of the four conclusion types (A, B, C, or D) with enough evidence to make a release decision.

### AC-5 — If full pass: GitHub Releases created

If conclusion A, PM tags `v0.3.0` and `v0.4.0` and creates GitHub Releases with the installer artifacts. The release notes files (`docs/releases/v0.3.0-release-notes.md`, `docs/releases/v0.4.0-release-notes.md`) are updated to change status from "Release Candidate" to "Released — [date]."

### AC-6 — If fail: defect documented

If conclusion C or D, the specific defect is documented as a new bug flag or ticket. The release remains at Release Candidate status until the defect is resolved.
