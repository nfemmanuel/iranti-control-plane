## Problem Statement

The command palette (CP-T024) provides fast navigation to every view in the control plane, but operators new to Iranti — or operators who haven't used the palette recently — cannot discover what each view does without navigating to it first. The current palette shows a view name and an icon; it does not explain what Memory Explorer, Archive, Activity Stream, or Conflicts actually contain or why an operator would navigate there.

Additionally, the palette is a natural home for surfacing keyboard shortcuts, but the current implementation has no mechanism to show them. The only keyboard hints visible are the global navigation keys (↑↓↵ Esc) in the palette footer. View-specific shortcuts (e.g., the fact that ↵ on a KB row opens entity detail, or that the Activity Stream supports filter-by-component) are entirely invisible.

The user explicitly requested a "commands manual" visible within the palette — descriptions and documentation for each command, accessible while browsing.

This ticket extends the command palette to become a discoverable reference surface, not just a navigation launcher.

---

## User Value

An operator opening the palette for the first time sees a one-line description of every view — they can read "Memory Explorer — Browse and search all facts Iranti currently holds, filtered by entity type, key, or source" and immediately understand where to go without trial and error. An experienced operator presses Cmd+K and reads the keyboard shortcut hints for their current view without breaking flow. The palette becomes both a fast-action surface and an embedded reference manual.

## Scope

**In:**

### 1. One-Line Descriptions on Navigation Commands

Each navigation command in the palette's Navigate section gains a second line of text below the label: a concise description of what that view contains and the primary operator use case.

Descriptions (PM-authored, not implementation-chosen):

| View | Description |
|------|-------------|
| Memory Explorer | Browse and search all facts Iranti currently holds, filtered by entity type, key, or source. |
| Archive | Inspect retired and superseded facts with their archival reason and full history. |
| Staff Activity | Watch Librarian, Archivist, Attendant, and Resolutionist events in real time as they happen. |
| Instances | View all connected Iranti instances, their project bindings, and instance health status. |
| Health | Diagnose system connectivity, provider config, and Iranti component health at a glance. |
| Conflicts | Review pending escalations — conflicting facts the Resolutionist couldn't auto-resolve. |
| Getting Started | Step-by-step setup status for a new Iranti installation. |

Layout: the description renders as a second line below the label in the result row, using a smaller, muted font. It must not overflow the row or push action tags out of view. On single-line-height screens (unlikely but possible), the description truncates with an ellipsis.

### 2. Keyboard Shortcut Hints Section

A "Shortcuts" section appended to the palette results list — visible when the query is empty or when the user types "?" or "help" or "shortcuts".

The Shortcuts section shows keyboard hints for the **current active view**. If no view-specific shortcuts apply (e.g., overview page), the section shows global shortcuts only.

**Global shortcuts (always shown in Shortcuts section):**

| Shortcut | Action |
|----------|--------|
| Cmd+K / Ctrl+K | Open command palette |
| Esc | Close command palette / dismiss panel |
| ↑ / ↓ | Navigate results |
| ↵ | Select / activate |

**View-specific shortcuts (shown when that view is active):**

| View | Shortcut | Action |
|------|----------|--------|
| Memory Explorer | ↵ on a row | Open entity detail |
| Memory Explorer | Escape | Clear search / close detail |
| Archive | ↵ on a row | Open archive detail |
| Activity Stream | Space | Pause / resume live stream |
| Activity Stream | F | Focus component filter |
| Health | R | Trigger health refresh |
| Instances | D | Open Doctor drawer for selected instance |

Note: the view-specific shortcuts listed above must be **actually implemented** in the respective views before this ticket marks them as documented. The frontend_developer must confirm each shortcut exists in the target component before listing it. If a shortcut is not yet implemented, it is excluded from this ticket (not invented here — documented elsewhere as a gap).

### 3. "?" Footer Trigger

A small "?" keyboard hint button in the palette footer (right side). Clicking it or typing "?" in the input filters the results to the Shortcuts section. This gives new operators a discoverable path to the shortcuts reference without knowing to type "help".

**Out:**
- A dedicated `/help` route (Phase 3 if user research confirms demand)
- An expandable command detail panel or flyout (too complex for P2 — the second-line description is the right scope)
- Shortcuts for views not yet built (CP-T020 embedded chat, CP-T022 provider manager — add in follow-on tickets when those views are complete)
- Full keyboard shortcut configuration or remapping (Phase 3)
- Tooltips on result rows (conflicts with keyboard-only navigation model)

## Risks

- **Description text drift**: If view names or purposes change in future phases, descriptions in the palette become stale. Mitigate: PM owns the description copy. Any view rename ticket must include a palette description update step.
- **Row height regression**: Adding a second line of text to every nav row will increase the palette height. Verify the palette does not overflow the viewport on screens smaller than 900px tall. If overflow occurs, add `max-height` + internal scroll to the results list (which is already present in the Phase 1 implementation).
- **View-specific shortcut verification scope creep**: The frontend_developer should not implement new keyboard shortcuts as part of this ticket — only document existing ones. If verification reveals that most shortcuts don't exist yet, the Shortcuts section ships with only the global shortcuts for Phase 2, and a follow-on task documents the shortcut implementation gap to the PM.

---

## Open Questions

- Should the description text be visible at all times, or only when the result row is selected/hovered? Recommendation: always visible — this is a discovery feature, not a detail-on-demand feature. Hiding it defeats the purpose for new operators browsing the palette.
- Should the Shortcuts section be shown by default (empty query shows shortcuts at the bottom), or only on explicit "?" trigger? Recommendation: show by default when query is empty — keeps the palette useful as a reference surface without requiring operators to know to type "?".
