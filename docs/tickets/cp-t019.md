## Problem Statement

A root `package.json` exists at the repo root with a full set of workspace convenience scripts (`npm run dev`, `npm run build`, `npm run setup`, `npm run migrate`), powered by `concurrently` for running server and client in parallel. This file was created after the Phase 1 DevOps artifacts (CI pipeline, setup scripts, README) were authored, so those artifacts do not reference it. As a result:

1. `npm run dev` at the project root will fail — `concurrently` is listed as a devDependency but has never been installed because `npm install` at root was never run.
2. The README quick-start does not mention `npm run dev` as a single-command start path — a significant developer experience gap.
3. The CI pipeline installs `src/client` and `src/server` dependencies but never installs root dependencies, meaning CI cannot use root workspace scripts.
4. The setup scripts (`dev-setup.sh` and `dev-setup.ps1`) do not mention `npm run dev` as the preferred single-command start after setup.
5. The release archive omits the root `package.json`, meaning operators deploying the archive cannot use workspace scripts from root.

## User Value

A developer cloning this repo and following the README should be able to run `npm install && npm run dev` from the root and get both server and client running. That path is broken today because `concurrently` is never installed. Fixing this makes the standard developer onboarding flow work as documented.

## Scope

**In:**

- Run `npm install` at the repo root to install `concurrently` and produce a root `node_modules/` (or verify it works correctly when a new contributor runs it).
- Update `README.md`:
  - Add `npm install` (or `npm run setup`) at root as step 3 in the Quick Start.
  - Replace or supplement the two-terminal start steps with `npm run dev` as the primary single-command start.
  - Keep the `--prefix` variants as the "run individually" alternative — do not remove them.
  - Add a "Root workspace scripts" reference table documenting all root scripts: `dev`, `build`, `start`, `migrate`, `setup`.
- Update `scripts/dev-setup.sh`:
  - After installing server and client deps, add a root `npm install` step to install `concurrently`.
  - Update the post-setup echo block to show `npm run dev` as the primary start command.
- Update `scripts/dev-setup.ps1`:
  - Same changes as the bash script — add root `npm install`, update post-setup echo.
- Update `.github/workflows/ci.yml`:
  - Add a root `npm install` step (or `npm ci` if a lockfile is added) before the client and server installs in each job, so CI can use root workspace scripts if needed in future steps.
  - Alternatively, keep the current per-prefix install pattern and add a comment explaining the root package.json and when it is relevant. PM preference: add the root install step — do not just add a comment.
- Update `.github/workflows/release.yml`:
  - Include the root `package.json` in the release archive so deployers can use workspace scripts from root after extracting the archive.

**Out:**

- Changing the structure of the `package.json` itself (scripts are correct as-is).
- Adding new scripts to the root `package.json` beyond what already exists.
- Migrating to npm workspaces (`workspaces` field in package.json) — this is a separate architectural decision.
- Adding a root lockfile if none exists — include it if `npm install` naturally produces one, but do not force a lockfile strategy change.

## Risks

- Root `npm install` will produce a `node_modules/` at the root. The existing `.gitignore` already has `node_modules/` which covers root — confirm the pattern applies to root before committing. It does: the `.gitignore` entry `node_modules/` is unanchored and matches at any depth.
- If a root `package-lock.json` is generated, it should be committed so CI can use `npm ci`. If it is not committed, CI installs may behave differently across runs. PM preference: commit the lockfile.
- `concurrently` labeled output format may differ from what contributors expect. Default format is fine — no custom prefix config needed.

## Open Questions

- Should the root `npm install` step in CI use `npm install` or `npm ci`? Answer: use `npm ci` if a lockfile is present (preferred), `npm install` if not. Devops engineer to decide based on whether the lockfile is committed.
- Should the release archive include a root `node_modules/` or just the `package.json`? Answer: just the `package.json` — deployers run `npm install` themselves.
