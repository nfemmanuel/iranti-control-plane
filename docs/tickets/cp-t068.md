# CP-T068 — Home Overview Dashboard

**Type:** Feature
**Priority:** P1
**Assigned:** backend_developer + frontend_developer
**Phase:** 4 (Wave 10)
**Status:** PM-ACCEPTED — 2026-03-21
**Epic:** CP-E015 (Iranti Desktop)

## Summary

v0.3.0 ships a capable but navigation-first product. Every view is a discrete section you navigate to. There is no at-a-glance landing page that tells you what is happening right now — the app opens to `/health` by default, which is a full diagnostic table rather than a status summary.

The user's feedback after v0.3.0-rc: *"you know how the Docker Desktop control panel looks? Why don't we have something like that?"*

Docker Desktop's value is that one screen tells you: what's running (system state), what happened (recent activity), how much (resource consumption), and what to do next (quick actions). This ticket builds that for Iranti.

The result is a new `/overview` route that becomes the default landing page. The moment an operator opens the control plane, they see the full system picture without clicking anything.

## What to Build

### Backend — `src/server/routes/control-plane/overview.ts`

Add a single aggregated endpoint:

```
GET /api/control-plane/overview
```

This endpoint assembles a lightweight overview payload from existing data sources. It must complete in under 500ms — use `Promise.allSettled` so partial failures degrade gracefully rather than returning 500.

**Response shape:**

```typescript
interface OverviewResponse {
  health: {
    overall: 'healthy' | 'degraded' | 'error'
    checks: Array<{
      name: string
      status: 'ok' | 'warn' | 'error'
    }>
    fetchedAt: string
  }
  kb: {
    totalFacts: number
    factsLast24h: number
    activeAgentsLast7d: number
    truncated: boolean   // true when staff_events table absent
    fetchedAt: string
  }
  recentEvents: Array<{
    id: string
    staffComponent: string
    actionType: string
    agentId: string | null
    entityType: string | null
    entityId: string | null
    key: string | null
    reason: string | null
    timestamp: string
  }>
  activeAgents: Array<{
    agentId: string
    isActive: boolean
    lastSeen: string | null
    totalWrites: number
  }>
  fetchedAt: string
}
```

**Data sources — no new DB queries beyond what already exists:**

1. **health**: Call the same logic used by `GET /health` — import the check functions or re-call the health endpoint internally. Only include `name` and `status` per check (no `detail` or `message` to keep payload small).
2. **kb**: Call `GET /metrics/summary` internally or inline the same aggregate SQL query (prefer the inline approach to avoid an internal HTTP round-trip). Fields: `totalFacts`, `factsLast24h`, `activeAgentsLast7d`.
3. **recentEvents**: Query `staff_events` for the 8 most recent rows ordered by `timestamp DESC`. If the table doesn't exist, return `[]` with no error. Fields: `id`, `agent_id`, `action_type`, `staff_component`, `entity_type`, `entity_id`, `key`, `reason`, `timestamp`.
4. **activeAgents**: Call the Iranti `/agents` proxy (same as `GET /agents`) but limit to agents where `isActive = true` or `lastSeen` is within the last hour. Cap at 6 agents. If Iranti is unreachable, return `[]`.

**Error handling:** If any single source fails, that field gets its zero value (`[]` for arrays, `0` for numbers, `'error'` for overall health). Never return 500. Always return 200 with partial data and a `fetchedAt` timestamp.

**Mount in `src/server/routes/control-plane/index.ts`:**
```typescript
controlPlaneRouter.use('/overview', overviewRouter)
```

### Frontend — Home Overview Dashboard

Create a new component at `src/client/src/components/overview/OverviewDashboard.tsx` (and a corresponding `.module.css`).

The visual language must match the Terminals palette exactly — emerald accent, near-black canvas, monospace typography where appropriate. This is not a data table. It is a dashboard composed of named cards arranged in a 2-column grid on wide viewports, stacking to single-column on narrow.

**Layout:**

```
┌────────────────────────────────────────────────────────────┐
│  ALERT BANNER (shown only when health overall ≠ 'healthy') │
├─────────────────────────────┬──────────────────────────────┤
│  SYSTEM STATUS STRIP        │  KB SUMMARY CARD              │
│  (health check mini-icons)  │  (totalFacts, 24h, agents)   │
├─────────────────────────────┼──────────────────────────────┤
│  RECENT ACTIVITY FEED       │  ACTIVE AGENTS CARD           │
│  (last 8 staff events)      │  (active agents list)         │
├─────────────────────────────┴──────────────────────────────┤
│  QUICK ACTIONS ROW                                          │
│  Search KB | Run Diagnostics | Browse Memory | View Logs   │
└────────────────────────────────────────────────────────────┘
```

**Component breakdown:**

#### 1. Alert Banner

Shown when `health.overall` is `'degraded'` or `'error'`. A full-width dismissible callout above the card grid.

- `'error'`: red/rose variant — "Iranti is reporting critical health issues. [View Health Dashboard →]"
- `'degraded'`: amber/yellow variant — "Iranti is degraded — N checks are warning. [View Health Dashboard →]"
- Dismiss button stores dismissed state in React component state (resets on remount/reload — same pattern as SetupBanner in AppShell)
- Clicking "View Health Dashboard" navigates to `/health`

#### 2. System Status Strip

A compact horizontal strip inside a card. Shows each health check as a mini pill: icon + name + status color.

- `ok` → emerald dot
- `warn` → amber dot
- `error` → rose dot

Show all checks from `health.checks`. On click anywhere in the strip or on the "Details →" button, navigate to `/health`.

Loading state: 5 skeleton pills.

#### 3. KB Summary Card

Three stat cells in a row:

| Stat | Value | Label |
|------|-------|-------|
| Total facts | `kb.totalFacts` (formatted with commas) | "total facts" |
| Written 24h | `kb.factsLast24h` | "last 24 hours" |
| Active agents | `kb.activeAgentsLast7d` | "active this week" |

If `kb.truncated` is true, show a muted note: "Activity data requires `npm run migrate`."

Footer link: "View Metrics →" navigates to `/metrics`.

#### 4. Recent Activity Feed

A mini list of the last 8 staff events. Each row:
- Staff component badge (monospace chip: `Librarian`, `Archivist`, `Attendant`, `Resolutionist`) — color-coded by component using emerald/teal/indigo/violet
- Action type (formatted: `write_created` → "Write Created")
- Agent ID (muted, truncated to 18 chars)
- Relative timestamp (e.g. "2 min ago")

Empty state (no events): "No recent Staff activity. Run `npm run migrate` to enable the event stream."
Loading state: 4 skeleton rows.

Footer link: "View All Logs →" navigates to `/logs`.

The feed does not need to be live/SSE for this ticket. A fetch-on-mount with a 30-second polling interval is sufficient. Mark with a comment for future SSE wiring.

#### 5. Active Agents Card

A compact list of agents active in the last hour. Each row:
- Agent ID (monospace)
- Status dot (emerald = active, gray = inactive)
- Total writes count (muted)
- Last seen relative timestamp

If `activeAgents` is empty: "No agents seen recently."
Cap display at 6 rows; if more exist show "View all agents →" footer linking to `/agents`.

#### 6. Quick Actions Row

Four prominent action buttons arranged in a horizontal row:

| Label | Icon | Action |
|-------|------|--------|
| Search KB | `⊕` | Navigate to `/memory` (Memory Explorer — KB search is there) |
| Run Diagnostics | `⊡` | Navigate to `/health` and trigger the Run Diagnostics panel |
| Browse Memory | `▦` | Navigate to `/memory` |
| View Logs | `≡` | Navigate to `/logs` |

Style: full-width-within-column button cards with icon + label + description text. Use the `--color-accent-emerald` border on hover. These are `<Link>` or `<button onClick={navigate(...)}` elements styled as cards, not small buttons.

**Quick action for "Run Diagnostics":** Navigate to `/health`. The Health Dashboard's Run Diagnostics button is already there — the quick action just routes the operator there. (A future ticket may add a direct trigger via URL query param.)

## Implementation Notes

- Do not create a new health endpoint. The overview backend should import and call the health check functions from `health.ts` directly, or re-use the `healthRouter` handler internally. The cleanest approach is to export the check functions from `health.ts` as named exports and call them from `overview.ts` with `Promise.allSettled`.
- For the KB summary, inline the aggregate SQL rather than making an internal HTTP request to `/metrics/summary`. The SQL is already written in `metrics.ts` — extract it into a shared helper or repeat the key fields.
- The `recentEvents` query is: `SELECT id, staff_component, action_type, agent_id, entity_type, entity_id, key, reason, timestamp FROM staff_events ORDER BY timestamp DESC LIMIT 8`. Handle the `42P01` table-not-found code gracefully (return `[]`).
- For `activeAgents`: call the existing Iranti agents proxy with a short timeout (3s). Filter to `isActive = true`. If the call fails for any reason, return `[]`.
- The overview backend route file should be under 200 lines. Keep it focused.
- The polling interval for the frontend is 30 seconds. Use TanStack Query's `refetchInterval: 30_000`.
