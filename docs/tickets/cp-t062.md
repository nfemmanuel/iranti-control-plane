# CP-T062 — Relationship Graph: B9 Blocker Note + GET /kb/related Awareness

**Type:** UX / Product Clarity
**Priority:** P3 (Wave 8)
**Assigned:** frontend_developer
**Phase:** 3 (Wave 8)
**Status:** PM-ACCEPTED 2026-03-21
**Epic:** CP-E002 (Knowledge Base Explorer)

## Summary

The Entity Relationship Graph (CP-T032, PM-ACCEPTED 2026-03-20) renders relationships using a BFS traversal from the `GET /kb/entity/:entityType/:entityId/graph` endpoint. The control plane's graph endpoint uses the `entity_relationships` table.

However, there is a parallel capability in Iranti: `GET /kb/related` — which returns entities related to a query entity via semantic vector search, not explicit relationship edges. This capability is blocked in the MCP layer by B9: "no MCP read tool for vector-based related entity search." The control plane does not expose `GET /kb/related` at all, which is correct for now. But the gap is invisible — an operator looking at the Relationship Graph has no way to know that semantic relationships are a planned capability or that it is currently inaccessible.

This ticket adds a brief informational note to the Relationship Graph view so operators understand what they are seeing (explicit graph edges only) and what they are not seeing (semantic similarity via vector search).

## Why This Is Wave 8

This is a one-file, low-complexity frontend change. It has high operator value because without it, operators may incorrectly conclude that the Relationship Graph represents all connections between entities. Given the CP-T051 Agent Registry and CP-T052 health extensions surface vector backend status, it is now timely to close this communication gap: operators know what the vector backend is and can see whether it is healthy — so the Relationship Graph note about `GET /kb/related` will be meaningful rather than confusing.

## Acceptance Criteria

#### AC-1 — Informational note in the Relationship Graph section

In the Entity Detail view's "Relationships" tab (or wherever the relationship graph is rendered), add an informational note below the graph. The note should say something like:

> "This graph shows explicit relationships between entities. Semantic relationships via vector similarity (`GET /kb/related`) are not yet available from the control plane — this requires MCP read tool support (tracked as B9)."

The note should:
- Use the same muted/tertiary text style as other informational labels in the product
- Not be dismissible (it is persistent context, not a one-time notice)
- Link to the Health Dashboard's vector backend status card if the operator might want to confirm their vector backend is running

#### AC-2 — Only shown when graph renders

The note should only appear when the Relationship Graph section is visible (i.e., when an entity has relationships). If the entity has no relationships and the empty state is shown, the note is not needed.

#### AC-3 — TypeScript clean

`tsc --noEmit` passes. No `any` introduced.

## Implementation Notes

- The Relationship Graph is rendered in `src/client/src/components/memory/EntityDetail.tsx`. The note is a small addition after the `<RelationshipGraph>` component render.
- This is a pure frontend change. No backend work is needed.
- Use the existing `--color-text-tertiary` token and the established information label styles from the rest of the product.
- "B9" should be written as a note reference, not a hyperlink to an external issue tracker. The control plane does not have an issue tracker surface yet.
