## Problem Statement

There is no first-class structured Staff event bus in Iranti. The control plane's Staff Activity Stream (FR3) requires a real-time or near-real-time stream of structured events from the Librarian, Attendant, Archivist, and Resolutionist. Today, these actions are either not logged in a machine-readable way or scattered across unstructured output. Without a defined event model, CP-T002 (API surface) cannot specify the event stream endpoint, and CP-E003 (Staff Activity Stream) cannot be built in Phase 1.

## User Value

Operators can watch what the Staff is doing in real time, diagnose unexpected behavior, and correlate writes/reads/archives to specific agents and timestamps — without reading raw logs or querying the database manually. This is one of the most differentiated capabilities of the control plane: making Iranti's internal reasoning legible to the operator.

## Scope

In:
- Define event schema covering all 4 Staff components: Librarian, Attendant, Archivist, Resolutionist
- Define at least 3 distinct action types per Staff component
- Each event type must include: Staff component, action type, entity (when applicable), key (when applicable), agent_id, source, reason/note, timestamp, event level
- Define persistence strategy with explicit tradeoffs: at least 2 options compared (e.g., append-only DB table vs structured log vs in-memory ring buffer)
- Define event levels with a clear, implementable separation between audit-grade events (always emit) and debug-grade events (configurable)
- Produce a written spec document at `docs/specs/staff-event-model.md`

Out:
- Implementation of the event bus
- Changes to upstream Iranti core (spec may propose changes, but must flag them as proposed — not in-scope for this repo to implement)
- UI for displaying events (that is CP-E003 in Phase 1)

## Dependencies

- Access to upstream Iranti codebase (or current docs) to understand what logging or eventing infrastructure exists today
- No other tickets block this one — CP-T001 can start immediately

## Acceptance Criteria

- [ ] Event schema covers all 4 Staff components (Librarian, Attendant, Archivist, Resolutionist) with at least 3 distinct action types per component defined and documented
- [ ] Each event type definition includes all required fields: Staff component, action type, entity (when applicable), key (when applicable), agent_id, source, reason/note, timestamp, event level
- [ ] Persistence strategy decision is documented with explicit tradeoffs between at least 2 options (e.g., append-only DB table vs in-memory ring buffer vs structured log file); the recommended option is identified with rationale
- [ ] Event levels are defined with a clear, implementable separation between audit-grade events (always emitted) and debug-grade events (configurable at runtime); the boundary between them is explicit, not ambiguous
- [ ] Spec is concrete enough that backend_developer can implement the event bus without requiring further architectural input from system_architect
- [ ] Spec explicitly identifies any proposed changes to upstream Iranti core, flagged clearly as out-of-scope for this repo but necessary for full implementation
- [ ] PM has reviewed and approved the spec before CP-T002 proceeds to the event stream endpoint section

## Risks

- Upstream Iranti may have incomplete or inconsistent logging today; the spec may need to propose additions to the Iranti core that are outside this repo's direct control. These should be documented as explicit dependencies, not ignored.
- Event volume could be very high if every internal step is emitted at audit level — the event level system must address this from the start, or the stream will be unusable in practice.
- If the upstream Iranti codebase is not accessible, the architect must flag this as a blocker to the PM rather than speculating about what exists.

## Open Questions

- Should the event stream persist events to a DB table, or remain ephemeral (in-memory ring)? What is the right retention window for audit-grade events?
- Should the spec propose changes to upstream Iranti core, or design a wrapper/adapter layer that reads existing surfaces without requiring upstream changes?
- Are there any existing structured log formats in the Iranti codebase that could be adopted or extended rather than replaced?
