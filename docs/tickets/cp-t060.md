# CP-T060 — Metrics Dashboard: Agent Write Volume and KB Growth Over Time

**Type:** Feature
**Priority:** P2 (Wave 7 candidate)
**Assigned:** backend_developer + frontend_developer
**Phase:** 3 (Wave 7)
**Status:** Dispatched 2026-03-21 — backend_developer + frontend_developer
**Epic:** CP-E013 (Metrics)

## Summary

The Iranti Control Plane has no historical data visualization. Operators can see current state (Memory Explorer, Health Dashboard, Agent Registry) but cannot answer: "How fast is the KB growing?" or "Which agents have been most active this week?" or "When did my rejection rate spike?"

This ticket adds a Metrics Dashboard view (`/metrics`) that visualizes key operational signals over time: KB growth rate, agent write volume by day/week, rejection and escalation trends, and archive rate. All data is derived from the existing `staff_events` table (CP-T039), requiring no new data collection infrastructure.

## Why This Is Wave 7

Wave 5/6 completes the operator insight surface (temporal query, contributors, diagnostics). With those features shipped, the control plane becomes a complete real-time operator surface. The natural next step is the time-dimension view: trends and historical shape of activity, not just current state. This is the highest-value feature not yet in the backlog because:

1. It uses only data that already exists (`staff_events`)
2. It gives operators a fundamentally new capability — spotting trends and anomalies across sessions
3. It completes the "passive→interactive→historical" progression of the Health/Diagnostics surface

## Implementation Notes

- All data comes from `staff_events` — the same table powering the Staff Logs view (CP-T050). The backend team already knows this table's schema: `event_id`, `timestamp`, `staff_component`, `action_type`, `agent_id`, `source`, `entity_type`, `entity_id`, `key`, `reason`, `level`, `metadata`.
- Chart library decision: if a chart library is not already installed, use native SVG. The control plane has precedent for pure SVG visualization from the Entity Relationship Graph (CP-T032). Do not add a heavy charting dependency (no Chart.js, no Recharts) without PM approval.
- The metrics backend should use SQL aggregate queries (`DATE_TRUNC`, `GROUP BY`) rather than loading all events into memory. The `staff_events` table can grow large.
- Metrics are read-only and non-critical — they can degrade gracefully if the `staff_events` table is empty or very small.

## PM Decisions (resolved at dispatch 2026-03-21)

1. **`totalFacts` source:** Derive `totalFacts` in the summary endpoint from `staff_events` accumulation (count `WRITE_ACCEPTED` minus `ARCHIVED` to date). Do not issue a full `GET /kb/query` with no filters — that would be an expensive unbounded scan. `staff_events` is sufficient for trend data; any small delta from facts written before the table existed is acceptable at MVP. If the `staff_events` table was recently created and has insufficient history, show `truncated: true` and surface whatever is available.

2. **Period options at MVP:** Offer **7d and 30d only**. 90d is deferred — `staff_events` history may not yet be long enough to make a 90d view useful, and it would create the impression of long-term data that doesn't exist yet. Add 90d in a follow-on when there is confirmed sufficient history.

3. **Chart library:** **SVG-native only.** No Recharts, Chart.js, or any other charting dependency. The control plane already has SVG precedent from the Entity Relationship Graph (CP-T032). Keep the dependency footprint minimal.

4. **DB index recommendation:** backend_developer should add a DB index on `(timestamp, agent_id, action_type)` on the `staff_events` table if not already present. Aggregate GROUP BY queries on this table will benefit significantly from it, especially as event volume grows.
