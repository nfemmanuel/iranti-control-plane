## Problem Statement

Provider and model configuration in Iranti currently requires manually editing `.env.iranti` files, understanding the key naming conventions for each provider, and restarting the Iranti server to pick up changes. There is no UI that shows which providers are configured, whether they are reachable, or how to fix a broken provider setup. PRD FR7 (Provider and Model Configuration) and FR8 (Provider Credit Visibility) require that operators can inspect and manage provider configuration from the control plane without editing raw env files.

The Phase 1 Health view (CP-T015) shows basic provider health as a traffic-light status card, but provides no management capability — it is read-only and limited to the default provider check. This ticket builds the full Provider and Model Manager view.

## User Value

An operator can open the Provider and Model Manager, see at a glance which LLM providers are configured and whether each is currently reachable, change the default provider or set a per-task model override, verify that API keys are present (without exposing their values), and navigate to provider-specific setup guidance if something is misconfigured — all without touching a terminal or env file.

## Scope

**In:**

**Provider List:**
- A list of all configured providers detected in the active instance's `.env.iranti`. Each provider shown as a card or row with:
  - Provider name and icon/logo (Anthropic, OpenAI, Ollama, Groq, Mistral — at minimum; full list per PM supported providers decision)
  - Reachability status: `Reachable` / `Unreachable` / `Key Missing` / `Checking...` — traffic-light badge
  - Default provider indicator: one provider has a "Default" badge. Only one can be default at a time.
  - API key presence: "Key configured" (presence only, never expose the value) or "No key found" with the expected env variable name shown
  - Last reachability check timestamp: "Checked 12s ago" or "Checked at 14:23:07"

**Default Provider Selection:**
- A "Set as Default" button on each non-default provider card. Clicking:
  1. Calls the backend to update the default provider via the supported configuration pathway (see Technical Note below).
  2. The previous default provider's "Default" badge is removed. The new default provider's badge appears.
  3. A success toast: "Default provider updated to [provider name]".
- The update must not write directly to the env file. See Technical Note on Configuration Pathway.

**Task-Type Model Overrides:**
- A secondary section (or expandable panel within the Provider view) showing task-type routing overrides. Task types are the Iranti-standard set: `embedding`, `completion`, `summary`, `classification`, `extraction`, and any others defined in the current Iranti spec.
- For each task type: currently assigned provider + model (or "Inherits default").
- Edit mode: clicking a task type override opens an inline selector to choose provider and model. Saving calls the backend configuration pathway.
- Reset button: "Reset to default" for each override. Returns the task type to "Inherits default".

**Provider Reachability Check:**
- A "Check Reachability" button per provider card. Triggers a backend call that attempts to ping the provider's health or models endpoint with the configured API key.
- "Check All" button in the view header refreshes all providers.
- Reachability check results show in the badge within 5 seconds (add a timeout: if the provider does not respond in 5 seconds, mark as `Unreachable` with the reason "No response within 5s").
- The reachability check is read-only — it does not write anything, burn quota, or modify state.

**Per-Provider Setup Guidance:**
- Each provider card has a "Setup Guide" link. Clicking opens an inline panel (or navigates to a dedicated panel) with:
  - The expected env variable name(s) for that provider (e.g., `ANTHROPIC_API_KEY`).
  - Where to get an API key (provider docs URL, opened in a new tab).
  - Common misconfiguration errors for that provider and how to fix them.
  - For Ollama (local): whether Ollama is detected running locally, and the URL to configure.
- Setup guidance is hardcoded per provider (not fetched from an API). It is static content maintained in the frontend.

**Key Presence Display:**
- API keys are never shown — not even partially (no `sk-****` truncation that reveals prefix pattern). Show only: "Key present" (green badge) or "No key found" (red badge) + the expected variable name.
- The variable name shown is the one the backend actually checked (aligns with the `decision/default_provider_key` decision: try multiple possible key names, show which one was found).

**Out:**
- Directly writing API key values into env files from the UI (never — this is a security boundary)
- Provider billing integration or credit balance display (Phase 3 — FR8; the provider telemetry capability matrix must be built first)
- Adding entirely new providers not already supported by Iranti (out of scope — this is a configuration surface, not an extension surface)
- Model fine-tuning or custom model registration (out of scope)

**Technical Note on Configuration Pathway:**
Changing the default provider or a task-type override cannot write directly to the `.env.iranti` file. The correct pathway must be one of:
1. An Iranti CLI command that updates the configuration (e.g., `iranti config set default_provider anthropic`) — call via subprocess from the backend.
2. An Iranti configuration API endpoint if one exists.
3. A structured config file separate from `.env.iranti` (e.g., `~/.iranti/config.json`) that the control plane can safely update.

The backend_developer must identify the correct pathway at task start. If none of these exists, escalate to the system_architect for a design decision before building the frontend configuration management UI. In this case: implement the provider list and reachability check in Phase 2, and defer the write-capable configuration management to a follow-on ticket.

## Dependencies

- Phase 1 Health endpoint (CP-T015) — extends it. The health view's provider status check is a simplified version of what this ticket fully implements. Ensure backward compatibility with what CP-T015 built.
- PM decision on supported providers list: at task start, the PM must confirm the list of providers the control plane will support in Phase 2. Minimum: Anthropic, OpenAI, Ollama. The decision determines which provider cards, icons, and setup guidance entries must be built.
- `decision/default_provider_key` (already written to Iranti) — the backend must check the keys listed: `IRANTI_DEFAULT_PROVIDER`, `DEFAULT_PROVIDER`, `DEFAULT_LLM_PROVIDER`. If none found: `warn`, not `error`.

## Acceptance Criteria

- [ ] All configured providers are shown with reachability status. A provider is "configured" if at least one of its expected API key variables is present in the active instance's env. Providers not configured are shown in a greyed "Not configured" state with setup guidance link visible.
- [ ] Reachability check works on-demand per provider and for all providers simultaneously. Results update the badge within 5 seconds. Timeout after 5s with "No response" state.
- [ ] Default provider selection: clicking "Set as Default" changes the default provider via the supported configuration pathway. The change is reflected immediately in the UI (optimistic update). If the backend call fails, the optimistic update is rolled back with an inline error message.
- [ ] Task-type model overrides are displayed for all Iranti standard task types. Each override can be changed inline. Saving calls the backend configuration pathway. "Reset to default" works per override.
- [ ] API key presence is shown as "Key present" or "No key found" — no key values or partial key values are ever displayed, including in tooltips, aria labels, or the DOM.
- [ ] Per-provider setup guidance is accessible from each provider card. Guidance shows the correct env variable name(s) for that provider and a link to provider documentation (opens in new tab).
- [ ] "Check All" button in the view header triggers reachability checks for all configured providers simultaneously.
- [ ] Provider view is accessible from the Phase 2 navigation (Settings or a dedicated Providers nav item — per PM navigation decision).
- [ ] TypeScript compiles without errors. No `any` types in the provider manager component tree.

## Risks

- **Configuration write pathway may not exist.** If Iranti has no programmatic mechanism to change the default provider (no CLI config command, no config API, no separate config file), the write-capable parts of this ticket must be scoped down. Implement the read-only provider list and reachability check as the Phase 2 deliverable, and flag the write pathway as a Phase 3 or upstream Iranti request. Do not implement direct env file writes as a workaround.
- **Provider reachability checks may have rate limit implications.** Calling a provider's API to check reachability consumes API quota or triggers rate-limit counters for some providers. The reachability check must use the lightest-weight endpoint available (e.g., `GET /v1/models` for OpenAI rather than a completion call). For Anthropic, use the models list endpoint. Document which endpoint is called for each provider.
- **Ollama local detection.** Ollama does not have a cloud API key — it runs locally. Detecting whether Ollama is running requires checking `http://localhost:11434/api/tags` (or the configured `OLLAMA_HOST`). The backend must handle the Ollama case specially — "key present" does not apply; instead show "Ollama detected at [URL]" or "Ollama not detected."

## Open Questions

- What is the full list of providers Iranti supports in the current version? The backend_developer must enumerate these from the Iranti codebase or docs at task pickup. The PM decision at task start will confirm the supported set for the control plane UI.
- Should task-type overrides be per-instance (stored per `.env.iranti`) or global (stored in `~/.iranti/config`)? This depends on the configuration pathway. The backend_developer must confirm the current Iranti behavior.
- Should the Provider Manager be a dedicated nav item ("Providers") or a sub-section of a Settings page? Recommendation: dedicated nav item for Phase 2, since provider health and configuration is a frequent operator concern. PM to confirm.
